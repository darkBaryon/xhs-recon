# 平台接入指南

新平台按“能力”接入，不要求实现完整平台。比如第一版抖音只需要关键词搜索，就只注册
`SearchCollector`；创作者主页和内容详情能力保持未注册，调用时会明确报“不支持”。

## 抖音只接搜索的最短路径

1. 新建 `src/recon/platforms/douyin/search.py`，实现：
   - `platform_id = "douyin"`
   - `collect_search(request)`
   - `collect_search_batch(requests)`
2. 在 collector 内把抖音 payload 翻译成统一的 `Creator`、`Content` 和
   `SearchCollectionResult`。所有 ID 必须使用 `PlatformId("douyin")`，避免与 XHS 同 ID
   冲突。
3. 在 composition root 创建 collector，并只调用：

   ```python
   registry.register_search_collector(douyin_search)
   ```

4. 把 registry 返回的 search collector 注入现有 `SearchContents`。关键词扩展、时间窗、排名、
   关键词归属、repository 和输出都直接复用，不应在抖音包里复制。
5. 增加 fixture 契约测试，再接真实采集。首版不创建空的 creator-feed/content-detail 类。

可执行示例见 `tests/recon/test_search_only_platform_contract.py`。它用一个只实现搜索的抖音
collector 跑完整 search 用例，并验证其他两种能力会 fail fast。

## 依赖边界

- 平台包可以依赖 `application.ports` 和统一 `domain`。
- 平台包不得依赖 repository、Web、CLI、其他平台或旧 `src.core/src.models/src.pipelines`。
- 平台原始字段只在平台翻译层出现，不能泄漏到 domain/application。
- 平台差异通过 registry 能力表达，不在核心代码里写 `if platform == ...`。

## 何时增加其他能力

当前采集 Port 只按平台能力划分：`SearchCollector`、`CreatorFeedCollector` 和
`ContentDetailCollector`。account 与 watchlist 共享后两者，不能各建一套平台采集器；两者的
全量/增量差异留在 application。只有真实需求出现时再注册缺少的能力，不应让平台提供假
返回值。
