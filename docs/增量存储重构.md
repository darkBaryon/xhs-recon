# 增量存储重构（本地 SQL + 标记已拉 + 只拉未拉）

## 动机

旧管线每次运行都**全量拉取**：重搜全部关键词、对全部 watchlist 账号重拉最新 N 篇、对
全部典型笔记重采评论。评论是最危险的一步（每篇一次详情请求），全量重采容易触发平台
控制机制。目标：把采集侧改成**增量**——见过的不再拉，只拉新增的，贴合「低频/小范围/
单线程」运行边界。

## 架构决策：重构，不重写

现有六边形架构（`pipelines → core → models` + `adapters` + `ports`）正好能吸收持久化：
新增一个 **Store 端口**（core 定义抽象）+ **MySQL 实现**（adapters），抓取前查「已见」。
core 仍平台无关、不 import 具体实现。3000 行已测代码（关键词扩展/账号打分/watchlist
合成/专业度打分/评论四字段红线/MediaCrawler 子进程驱动）全部保留、零改动。

## 真相载体的转变

| | 旧 | 新 |
|---|---|---|
| 真相 | 每次运行的时间戳目录（run dir） | 累积的 MySQL 库 |
| 导出 | 全量重抓后独立归档 | 对库当前状态的一次查询/快照 |
| 跨命令衔接 | `latest` 软链 + run 目录里的 CSV | 库查询（期3 起） |

## 增量的支点：MediaCrawler 只能「拉最新 N 篇」

MC 的 creator 模式没有「跳过这些 id / 只要 X 之后」，所以增量在**我们这侧**做，三层：

1. **评论只补新（省最多）**：库里 `notes.comments_fetched_at` 非空且未过期的典型笔记
   直接跳过，只对没抓过/过期的发 `fetch_comments`。
2. **creator 稳态小 N + 去重**：仍拉最新 N（可调小），回来按 `note_id` 去重——新的插入、
   旧的只刷新互动数（顺手，不额外请求）。
3. **search 不进日常**：全量关键词搜索本就只在「进新领域」跑（`search` vs `track` 已分开）。

## 存储后端：本机 MySQL 独立库 `xhs_recon`

用 uni_atlas 同一台 MySQL、同一套凭据（`~/.my.cnf`），但独立 database——既能和
`study_abroad` 原生跨库 join（合数据做选题那步天然通），又不污染 uni_atlas 的表。
不用 SQLite：你本就常年开着 MySQL，SQLite 的「零依赖」优势不成立，反倒多个孤立文件。

「是否已拉」= 行上的时间戳列（即 uni_atlas 的 `source_pages.last_fetched_at` 那套）：

- `accounts(account_id PK, …, creator_fetched_at)` — NULL=从没拉过主页
- `notes(note_id PK, …, like/collect/comment_count, first_collected_at, last_collected_at,
  comments_fetched_at)` — `comments_fetched_at` NULL=从没抓过评论（增量判据）
- `comments(note_id, body_hash, …, PRIMARY KEY(note_id, body_hash))` — 正文哈希去重
- `creator_profiles(account_id PK, verify_type, fans, …)`

互动数每次再遇到就 UPDATE（不额外发请求）；`first_collected_at` 永不回改（幂等关键）。
构造 `MySQLStore` 即幂等建库建表（`CREATE DATABASE/TABLE IF NOT EXISTS`）。

## 测试口径：照搬 uni_atlas

uni_atlas 无 CI、DB 层不写离线单测，靠对真实本机 MySQL 手验。本重构照搬：

- **离线 CI**（`pytest`）：不碰 MySQL，管线测试全走 `store=None`（旧路径），保持全离线。
- **DB 层验收**：`scripts/integration_store.py` 对真库（一次性测试库 `xhs_recon_itest`
  建/删）跑幂等 upsert / 评论去重 / 增量判据 / refresh 窗 / 落库往返，与「真实采集不进 CI」
  同一套哲学。

## 已交付（期1 + 期2）

- **期1** `src/core/store.py`（Store 端口）+ `src/adapters/mysql_store.py`（MySQL 实现，
  pymysql + `~/.my.cnf`，`ON DUPLICATE KEY UPDATE`）。
- **期2** 接线：`config.py` 加 `StoreCfg(enabled/database)` 与 `comments.refresh_days`；
  `runtime.build_store` + `prepare` 返回 store；`_search/_sync/_comments` 三段接 upsert；
  评论段接 `notes_needing_comments` 增量过滤。
- **迁移/验收脚本** `scripts/backfill_store.py`（历史导出 → MySQL，幂等）+
  `scripts/integration_store.py`（真库验收）。
- **开关**：`store.enabled` 默认 **false** → 行为与旧版逐字节一致（全量、per-run 目录、latest
  软链全保留）；真实配置里置 `true` 即开增量。离线 199 测试绿、真库集成 5 项绿、ruff 绿。

### 开启方式（真实配置）

```yaml
store:
  enabled: true
  database: xhs_recon   # 本机 MySQL 独立库；凭据自 ~/.my.cnf
comments:
  enabled: true
  refresh_days: 0       # 只抓从没抓过的评论；>0 则超期重刷
```

首次切库前先回填历史（避免从空库重爬）：`uv run python scripts/backfill_store.py`。
