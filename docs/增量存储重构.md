# 增量存储：历史决策与当前实现

本文最初记录旧管线接入 MySQL 的设计。多平台模块化升级后，目标没有变化：数据持续累积、
重复内容幂等更新、watchlist 只处理到期账号、评论与媒体只补缺失项；实现位置已经迁入
`src/recon`，旧 `src/core/store.py`、`src/adapters/mysql_store.py` 和配套临时脚本不再是运行入口。

## 当前结构

- 应用端口：`src/recon/application/ports/repository.py`
- MySQL 实现：`src/recon/infrastructure/persistence/mysql/repository.py`
- 旧库迁移：`src/recon/infrastructure/persistence/mysql/legacy_import.py`
- 命令入口：`python -m src.recon.entrypoints.cli`

MySQL 使用 `creators`、`contents`、`content_comments`、`media_assets` 和
`content_keywords`。内容实体按平台与外部 ID 唯一；关键词归属单独存在 `content_keywords`，
因此相同帖子被不同关键词搜到时不会复制内容，也不会丢失来源区别。

## 增量规则

1. account/search/watchlist 分别通过独立 repository 端口保存，不互相调用。
2. watchlist 根据 `creator_fetched_at`、刷新天数和批大小选择本次到期账号。
3. 已知内容 ID 用于区分新增与已存在帖子；再次遇到的帖子更新互动等可变字段。
4. 评论按 `comments_fetched_at` 与刷新天数判断是否需要补抓。
5. backfill 只查询缺少本地媒体的内容，并通过同一 repository 幂等回写。
6. 旧表和历史媒体不删除；迁移命令支持先 dry-run，再执行导入。

## 验证方式

日常离线回归运行：

```bash
uv run ruff check src tests scripts web
uv run pytest -q
```

真实数据库迁移与采集不进入离线测试。执行前应使用 CLI 的迁移 dry-run，确认计数和目标库，
再运行正式迁移；真实 watchlist 验收同时检查实体计数、关键词归属和孤儿记录。
