## 背景

v0 三期交付后集成验收两次排查暴露同一短板：**全项目零日志**。`_run_crawler` 捕获 MediaCrawler 子进程全部输出后成功即丢弃（期3 曾因 247KB stderr 中一个非法字节炸管线，事后无据可查）；管线六个阶段无过程可见性（manual 集成跑十几分钟终端全程沉默，用户误判为卡死）。checklist 又禁 `print(`，正常路径被规定成哑巴。后续迭代与排障没法测试，先补日志。

**参考范式**：`~/VSCode/ws_2026/house-manager`（Go/slog）的三层结构——通用构造层（level/format 可配、组装点注入）＋ 业务约定层（上下文字段注入、错误→级别路由、脱敏唯一收口）＋ main 组装点接线。本需求按该哲学做 Python/stdlib 等价物，**不引第三方日志库**。

## 目标

1. **通用层**：各模块 `logging.getLogger(__name__)` 只发日志；级别/格式/去向仅在 pipeline 入口（组装点）配置一次；`--verbose` CLI 开 DEBUG。
2. **双输出**：控制台（人读，默认 INFO）＋ 每次运行一个日志文件（DEBUG 全量，`data/logs/run-<collected_at>.log`，gitignore）。
3. **子进程输出落盘**：`_run_crawler` 把 MediaCrawler stdout+stderr 写 `{save_path}/mediacrawler.log`——从此每次采集可事后复盘，`error` 不再只留末尾 300 字符。
4. **运行上下文字段**：日志行携带 run 级字段（collected_at/provider/keyword），Python 侧用 `LoggerAdapter` 或 `Filter` 注入（对应 applog.ContextHandler）。
5. **结果路由约定**：阶段结束统一记「成功 INFO / 预期内失败（FetchResult.error、降级）WARN / 异常 ERROR」（对应 applog.Result）。
6. **阶段可见性**：扩词/每次 search/聚合/打分/选典型/评论/导出各一行 INFO（条数+耗时），manual 集成脚本不再哑跑。

## 非目标

- 不引 loguru/structlog 等第三方库；不做 JSON 格式、日志轮转、集中收集（个人本地工具用不上，通用层留出扩展位即可）；
- 不改采集/导出业务逻辑与既有导出契约；
- **日志红线**：日志不得输出评论内容与任何评论者身份字段（脱敏收口哲学沿用——身份字段在 parser 边界已丢，日志层只记条数不记内容）。

## 定档提议

**标准级**。理由：横切 adapters/pipeline/scripts 三层、有接口设计决策（config `logging` 段、字段约定、红线口径），值得方案页+评审+checklist 一轮；但单期、无跨期契约，不够蓝图级。

**用户裁决**：可（2026-07-02，标准级）

## 验收口径

- fixture demo 跑一遍：控制台出各阶段 INFO；`data/logs/run-*.log` 存在且含 DEBUG；
- 真实采集跑一遍（manual）：`{save_path}/mediacrawler.log` 存在、含子进程完整输出；
- 采集失败路径记 WARN 且管线续跑（不假装口径不变）；
- 日志文本 grep 无评论 body/身份字段；`src` 仍无 `print(`；
- ruff/pytest 全绿；既有导出文件逐字节不变。
