## 背景

`run_research` 是 v0 立项时「一条命令端到端」的单体编排，三期追踪功能陆续焊在同一条流水线上。真实使用后暴露节奏错配（2026-07-03 实测定位）：

1. 三段天然频率不同——关键词搜索/creator 同步宜每周，评论只在深度分析时需要；现在每跑全量执行；
2. 评论段在 109 账号 × 2 典型笔记规模下必然超时报废（600s 上限，本日实测），但想跳过只能改 config 开关，想单独补采则完全做不到；
3. 单体编排也不利于测试与后续优化（creator 单会话改造、评论上限等都要在同一函数里动刀）。

## 目标

仿 Web 后端「路由 → handler → 共享 service」结构，将管线拆为 typer 子命令：

1. `search`：关键词搜索 → 聚合打分 → 选典型 → 导出搜索侧文件（广角，每周）；
2. `sync`：watchlist 合成（auto 名额读最近一次 search 的榜单文件）→ creator 拉取 → 专业度 → topic_feed（长焦，每周）；
3. `comments`：对最近一次结果的典型笔记补采评论并回填报告（深读，按需）；
4. `research`：三段按序组合，与现行为等价（兼容入口）。

跨命令衔接走**文件契约**（`data/exports/latest/` 的既有 CSV），不引入数据库/常驻状态。

## 非目标

- 不动 adapters / core 计算逻辑（B1/B2 修复、评论上限、creator 单会话属下一案）；
- 不做并发、不做定时任务（运行边界不变）；
- 旧入口 `python -m src.pipelines.run_research --config` 行为与导出契约零变化。

## 定档提议

**标准级**。理由：动 pipeline 结构 + 新增 CLI 契约 + 跨跑文件衔接语义，需要方案评审把关兼容性；但无蓝图必要（单期可完成，无跨期契约）。

**用户裁决**：可（2026-07-03，用户「先拆子路由，方便测试」）。

## 验收口径

- `research` 子命令与旧入口在同一 fixture 配置下导出逐字节一致（机械可验）；
- `search` 单独跑只产搜索侧文件；`sync` 单独跑在无搜索榜单时纯 manual 降级（警告可见）；`comments` 单独跑能对最近一跑补齐评论与报告；
- 全部子命令共用同一份 config，无新增配置段。

## 处置历史

- 2026-07-03 立案（源于真实使用的节奏错配与评论超时实测）；定档标准级，用户口头裁决可。
- 2026-07-03 方案 v2 两轮评审通过 → Gate 1 放行（拍板：补全式写回）→ Claude 本人实施（分工变更）→ 代码评审两轮（#1 阻塞：未登记文件删除卷入，补登后 #2 通过）→ **Gate 2 验收通过**（抽查档；exports_real 维持删除）。
