## 背景

当前项目已经具备 `pipelines → core → models`、由 adapters 实现端口的分层雏形，但平台扩展点尚未闭合：具体 provider 在组装根中以条件分支硬编码，平台专属配置与通用运行配置混放，编排通过运行时能力探测调用端口未声明的方法，且 MediaCrawler 适配器与主编排文件持续膨胀。若直接沿现状加入抖音搜索，会把平台差异继续扩散到配置、编排、存储和导出层，增加后续功能迭代与测试成本。

## 目标

- 建立面向多平台的稳定领域契约，统一表达平台、内容、创作者、搜索结果和能力差异。
- 把应用用例、平台插件、基础设施、CLI/组装根拆成边界清晰的模块，依赖方向可机械检查。
- 将现有小红书搜索、账号追踪、增量存储、导出和 Web 消费迁移到新架构，保持现有用户可见行为与数据可用。
- 建立显式的平台注册与能力声明机制；未来新增抖音平台时，主要新增平台包及配置，不修改领域核心和已有小红书实现。
- 补齐配置校验、契约测试、架构测试、端到端回归和迁移文档，使模块可以独立演进。

## 非目标

- 本需求不实现抖音真实搜索、登录、解析或反爬适配；抖音接入在架构底座稳定后单独立案。
- 不引入代理池、验证码破解、自动点赞/评论/私信或提高采集并发。
- 不借重构改变现有评分算法、选题算法和业务口径。
- 不一次性替换 MySQL、MediaCrawler、静态 Web 等已工作的外部依赖。
- 不承诺旧的 Python 私有导入路径永久兼容；CLI、配置及导出数据的兼容策略在蓝图 Gate 前单独拍板。

## 定档提议

**蓝图级**：改动横跨领域模型、应用编排、平台适配、配置、存储、导出与测试多个子系统，需要建立跨期稳定公共接口并分期迁移。

**用户裁决**：可（2026-07-16，按蓝图级推进）。

## 验收口径

- 开发蓝图明确目标模块图、依赖方向、公共契约、兼容策略、分期与每期回滚点，并通过独立方案评审。
- 架构测试可机械阻止领域/应用层反向依赖具体平台或基础设施。
- fixture 端到端流程与现有小红书离线回归全部通过；现有 CLI 主路径和导出消费路径按 Gate 决策保持兼容或提供明确迁移层。
- 存在一个不改领域核心即可注册的最小“第二平台假实现/契约夹具”，证明扩展点真实可用，而非仅把 XHS 目录改名。
- 文档给出新增平台的最短接入步骤、所需能力、配置 schema、契约测试与失败语义。
- 全量 `ruff`、`pytest`、工作台视图校验通过。
