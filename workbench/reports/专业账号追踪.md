## 背景

v0 三期交付的是「关键词搜索 → 热帖聚合」管线，与最初动机（**关注本领域专业账号及其最新发帖，总结后供其他项目选题**）存在三层偏差，2026-07-02 实测定位：

1. **搜索排序写死热度降序**：MediaCrawler `config/xhs_config.py:24` `SORT_TYPE = "popularity_descending"`，且 CLI 未暴露该参数，adapter 无法覆盖 → 结果全是历史爆款（demo 首条为 2024 年帖）；
2. **无账号维度采集**：管线只有 search 一跳，「账号」仅是热帖作者的副产品，从未访问账号主页拉最新发帖。MediaCrawler 具备 creator 模式（`--type creator --creator_id`）但 adapter 未接；
3. **打分选帖偏热度**：`account_ranker`（note_count/keyword_hit/interaction）与 `note_selector`（纯互动加权）均无时间维度；`Note.published_at` 已解析（`parsers.py:56`）但全程未使用。

## 目标

1. 搜索采集支持**按发布时间排序**（time_descending）并可配**时间窗过滤**（如近 30 天）；
2. 账号打分从「热度」转向**专业度**：垂直度（命中关键词笔记占比）、近期活跃度（时间窗内发帖数），互动量降为辅助信号；
3. **watchlist**：自动发现 top-N 入榜 + config 手动补充账号主页 URL（用户 2026-07-02 拍板：自动发现 + 手动补充）；
4. 对 watchlist 账号经 **creator 模式**抓取最新 N 篇笔记；
5. 汇总导出：按 账号 × 时间窗 的**选题素材**（markdown + jsonl），供其他项目（选题）直接消费。

## 非目标

- 只读拉取数据，不做关注/点赞/评论等任何写操作（沿用 v0 口径）；
- 不做选题生成本身（下游项目职责），只产出结构化素材；
- 不破坏 v0 既有导出契约（新增导出文件，不改旧文件格式与字段）；
- 不做多平台（仅 xhs）。

## 定档提议

**蓝图级**。理由：跨两个子项目（MediaCrawler 本地 fork 需暴露排序参数；xhs-recon 新增 adapter 接口 + 新 core 模块 + 新导出），且「时间维度修复」与「creator 追踪 + watchlist」天然分期、各期都够标准级。

**用户裁决**：可（2026-07-02，蓝图级）

## 验收口径

- 一条命令端到端：产出 账号榜（含专业度分项得分，落 `account_profile.csv`，可复算）、watchlist 每账号最新 N 篇按原样导出（`creator_notes.csv`，不活跃账号可能出窗，属预期）、选题素材导出；
- 时间口径机械可验：`topic_feed` 只收时间窗内笔记（全部 `published_at` 落窗内），出窗/缺时间的笔记不入 feed、在报告计数；
- fixture 降级路径不中断、测试不依赖真实小红书访问（沿用 v0 口径）；
- 已知风险验证点：xhs creator 模式需带 `xsec_token` 的主页 URL——搜索结果 JSONL 能否直接供出该 URL 需期方案 spike 定案。
