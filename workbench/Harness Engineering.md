# Harness Engineering

## 1. 这套流程解决什么问题

AI 可以显著提高编码速度，但商业项目不能只追求“代码写得快”。如果人只提出需求、不参与实施方案，项目的架构和演进方向会逐渐变成只有 AI 理解的黑箱；如果每个技术细节都由人审批，AI 带来的效率又会被大量确认工作抵消。

工作台流程希望在两者之间建立清晰边界：

> 人掌握需求、方案和风险边界，Agent 承担实施、检查和技术评审，用可验证结果证明交付质量。

它不是无人监督的自动开发，也不是由人逐步批准的流水线，而是一套“共同设计、边界内自主执行、关键变化重新决策”的协作方式。

## 2. 核心思想

### 人必须理解项目将如何变化

实施前，用户与 Agent 共同讨论目标、模块职责、接口、数据流、实施步骤、风险和回退方式。Agent 负责研究代码、提出选项并解释后果，用户负责补充业务背景、提出疑问和参与取舍。

用户不需要审核每个函数，但应能够理解：为什么这样改、系统会怎样变化、可能影响什么、失败后如何恢复。这样可以保证项目的长期技术方向仍由人掌握。

### Agent 在确定边界内连续完成工作

方案确定并通过技术评审后，命名、代码组织、测试结构、普通重构和评审修复等实施细节由 Agent 自主处理，不再逐项请求用户确认。

如果实施中需要改变目标、模块职责、公共接口、数据流、兼容或迁移策略，流程必须返回方案讨论。真实外部写入、发布、付费、数据删除等高风险操作，也必须由用户明确授权。

### 评审与实施相互独立

设计方案和实际代码都由独立 Reviewer 评审。Reviewer 不直接采信实施者的结论，而是根据代码、测试和运行结果重新检查。Reviewer 不绑定 Codex，可以使用经项目验证效果更好的 Claude Code、其他模型或人工评审者。

普通技术问题由 Agent 修复并复审，直至阻塞问题关闭。只有评审结论引发业务或方案层面的变化时，才重新交给用户决策。

### 以证据而不是声明证明完成

Agent 不能仅凭代码阅读或主观判断宣布任务完成。交付前必须运行本地检查，验证关键行为，并说明实际完成内容、检查结果、已知限制和回退方式。

真实测试发现的稳定问题应尽量补成回归测试，使一次人工经验转化为后续任务的永久保护。

## 3. 完整流程

```text
需求与现状研究
→ 用户与 Agent 共同制定方案
→ 独立 Reviewer 评审技术方案
→ Agent 实施并运行本地检查
→ 独立 Reviewer 评审代码
→ Agent 修复并复查
→ 按需由用户进行真实验收
```

### 第一阶段：共同制定方案

Agent 调研现有代码并提出推荐方案、备选方案和影响分析。用户参与讨论，确定业务取舍和系统演进方向。

该阶段不设置形式化批准动作。用户已经参与方案形成、关键问题已有结论且不存在未解决异议时，方案即可进入独立技术评审。

### 第二阶段：技术闭环

独立 Reviewer 先评审方案的可行性、完整性和测试策略。方案通过后，实施 Agent 按方案步骤实施；多步骤任务的每个步骤分别形成一个独立 commit，并在提交前完成对应检查。Reviewer 以 Git commit 和 diff 为主要事实来源，逐步核对实际改动与方案，再检查全部步骤组合后的整体结果。发现的问题在实施者与 Reviewer 之间完成修复和复查。

这一阶段通常不需要用户参与。只有方案边界或权限边界发生变化时，才暂停并请求用户决策。

### 第三阶段：真实验收

自动检查和代码评审通过后，技术验收即告完成。只有真实账号、外部平台、UI 观感、真实数据或业务判断无法自动验证时，才请用户进行人工测试。

Agent 应提供明确的测试步骤、预期结果和失败后的回退方式，避免用户重新阅读整个技术过程。

## 4. 实际案例：日志系统收敛

### 案例背景

该案例来自一个包含管理端、发布端和小程序端的后端项目。业务日志用于线上问题排查、用户操作定位和异常监控，通常需要记录请求编号、操作人身份、业务事件、执行结果和错误信息。

项目早期没有统一设计日志规范。不同模块在开发过程中分别复制或实现了自己的日志辅助代码，逐渐形成以下问题：

- 五套功能相近的日志工具同时存在，日志策略变更需要多处同步修改；
- 手机号脱敏存在多份实现，其中一份已经出现不同处理逻辑，带来隐私合规风险；
- 操作人信息在部分日志中是多个独立字段，在另一些日志中是压缩字符串，无法稳定地跨模块检索；
- 同名 `principal` 字段有时是字符串、有时是对象，接入 ES、Loki 等日志平台时可能发生字段类型冲突；
- 重复实现经过改名后难以通过普通搜索发现，维护人员无法确认一次修改是否覆盖完整。

本次重构的目标是建立一套共享日志能力，统一上下文字段、错误级别、身份结构和敏感信息处理，并逐步替换各模块的旧实现。重构不得改变业务接口、响应结果和既有事件名称，以避免影响现有排障习惯及潜在的日志消费方。

初步方案预计涉及约 430 个调用触点，需要拆成八个可验证、可回退的步骤。因此，这不是一次简单的代码整理，而是一次涉及公共基础设施、隐私处理和线上可观测性的中等规模重构，适合用于说明 Harness 如何控制变更风险。原始问题记录见[日志系统碎片化问题](https://app.notion.com/p/37b81496dc138122ae4edcbeeb9db646)。

### 第一步：用户与 Agent 共同确定方向

Agent 调研后提出建立统一的 `pkg/applog`，集中处理上下文字段、错误级别、手机号脱敏和 principal schema，并将迁移拆成八个可验证步骤。用户需要参与确定：

- principal 是否统一为 group 结构；
- test/prod 是否启用 source 信息；
- ContextHandler 采用包依赖还是在 `main.go` 组装；
- 是否确认现有日志面板和告警不依赖旧字段结构。

用户负责这些会影响长期架构和运行结果的选择，不需要决定每个函数如何实现。选择形成后不再追加形式化 Gate 1。

### 第二步：Reviewer 独立检查方案

独立 Reviewer 没有采信[方案 v1](https://app.notion.com/p/37b81496dc138152bbf2e5c300eaeaea)中的现状描述，而是直接使用 `grep`、`diff` 和 `wc` 检查仓库。[第一轮方案评审](https://app.notion.com/p/37b81496dc13810fadfbd770875f6877)发现了三项阻塞问题：

1. 手机号脱敏实现实际有 11 份，不是方案记录的 9 份；遗漏项不会被原验收命令发现。
2. 访问日志和业务日志对同一个 `principal` 字段分别输出字符串和对象，接入 ES/Loki 后可能发生字段类型冲突。
3. `add_source` 只有 local 环境为 true，test/prod 实际为 false，原方案基于错误配置设计了验证口径。

实施 Agent 据此修订方案，补全遗漏范围、统一 principal 结构并修正配置决策。[方案 v2](https://app.notion.com/p/37b81496dc138153b64de3f5e3a6df5d)经[第二轮方案评审](https://app.notion.com/p/37b81496dc13816ea887db8acec14f98)确认阻塞项关闭后进入实施。

这个阶段的价值不在于“另一个 AI 表示同意”，而在于 Reviewer 提供了可以由命令复现的反证。

### 第三步：分阶段实施和代码评审

实施 Agent 将八个迁移步骤分别形成八个独立 commit。每个 commit 只完成一个方案步骤，并在提交前通过对应的格式检查、静态检查、测试和构建。这样可以明确每一步实际修改了什么，也可以在单步出现问题时独立回退。

步骤 1—5 分别对应 `ce96524`、`f47200a`、`285a697`、`cea0441` 和 `2ad03c7`。完成后，[代码评审 #1](https://app.notion.com/p/37b81496dc1381b58391f47b80921448)不依据实施 Agent 的工作总结，而是直接检查 Git 记录：

- 使用 `git show <commit>` 逐项核对每个 commit 是否只包含对应步骤；
- 使用 `git diff 5574e4e..2ad03c7` 检查步骤 1—5 的整体影响；
- 将实际 diff 与方案中的范围、禁止事项和验收标准逐项对照；
- 对关键 commit 独立执行编译和静态检查，验证每一步均可单独成立和回退。

第一轮代码评审确认：

- 199 个既有事件名保持一致；
- 日志级别路由和脱敏行为得到测试覆盖；
- 修改范围没有扩展到日志之外；
- 每个阶段均可独立编译和回退。

Reviewer 同时发现 `WithGroup` 缺少独立测试。实施 Agent 在后续步骤中补充测试，再交回原 Reviewer 复查。

步骤 6—8 同样分别形成 `cf0c22a`、`5d7eaa3` 和 `66c4179` 三个 commit。全部步骤完成后，[代码评审 #2](https://app.notion.com/p/37b81496dc138116bbc2f76e0f9a30a9)先使用阶段 diff 检查步骤 6—8，再使用 `git diff 5574e4e..66c4179` 复查八个步骤的最终整体结果，并重新运行全部验收命令。

终轮评审确认七项检查全部通过：日志辅助代码 5→0、脱敏实现 11→1、principal 形态 3→1、260 个事件名集合保持一致，格式检查、静态检查、测试和构建全部通过。评审还识别出 shared-docs 子模块尚未推送，并将其列为合入前置条件。

在该流程中，独立 commit 是实施与评审之间的交接单位，Git diff 是评审的主要事实来源。方案说明“准备怎样改”，commit 和 diff 证明“实际怎样改”，测试与运行结果证明“改完是否正确”。

### 第四步：按需由用户验证真实效果

在修订后的 Harness 中，前述检查和代码评审通过即代表技术验收完成。用户只需在确有需要时抽查真实日志输出、检索效果或外部日志平台兼容性，不需要重新审核八个步骤的代码。

原流程最终形成了单独的[验收记录](https://app.notion.com/p/37b81496dc13814790cbdacf8f111f67)。在新流程中，该结果可直接汇总到当前任务记录，不再增加一次无实际测试内容的 Gate。

该案例体现了 Harness 的四项作用：用户控制架构选择，Agent 负责大规模实施，Reviewer 通过独立证据发现错误，自动检查负责证明结果。

## 5. 人与 Agent 的职责

| 参与者 | 主要职责 |
|---|---|
| 用户 | 提供业务背景；参与方案制定；决定重大取舍；授权高风险操作；完成必要的真实验收 |
| 设计与实施 Agent | 调研代码；提出并维护方案；实施功能；运行检查；修复评审问题；提供交付证据 |
| 独立 Reviewer | 评审技术方案和实际代码；复核关键风险；确认阻塞问题已经关闭；可以由 Claude Code、Codex、其他模型或人工承担 |

职责划分的原则是：用户不承担普通技术检查，Agent 不代替用户决定业务目标和重大系统方向。

## 6. AI 如何执行评审

AI 评审不是让实施 Agent 重新阅读一次自己的代码，而是把评审作为独立任务交给另一个干净上下文的 Reviewer。评审者不继承实施者的推理过程，只接收可核验材料：目标、共同方案、代码差异、项目约束和验证命令。

Reviewer 模型应由项目实际效果决定，而不是固定使用某一供应商。可以使用历史缺陷建立小型评审测试集，比较 Claude Code、Codex 或其他模型的缺陷发现率、误报和证据质量，再确定默认 Reviewer。独立上下文只能减少思维惯性，不能弥补模型本身的能力不足。

### 两次评审关注不同问题

| 评审阶段 | 主要检查内容 |
|---|---|
| 方案评审 | 方案是否可实施；接口和数据设计是否合理；依赖、异常、兼容和测试策略是否完整 |
| 代码评审 | 实际代码是否符合方案；业务逻辑和边界条件是否正确；测试是否充分；是否引入安全、数据或维护风险 |

### 评审者必须独立核验

Reviewer 应直接读取相关代码和实际差异，不以实施 Agent 的总结作为事实依据。对于可以运行的内容，应执行相关测试、复现命令或静态检查；对于无法自动证明的判断，应说明依据和剩余风险。

评审结果只记录可执行的问题，每个问题至少包含：

- 严重程度和代码位置；
- 可能造成的实际后果；
- 判断依据或复现结果；
- 建议的修复方向。

没有发现有效问题时，可以明确给出“无阻塞项”，不要求为体现工作量而制造意见。

### 问题必须形成闭环

实施 Agent 根据评审结果修改代码，并重新运行相关检查。原 Reviewer 随后复查修复差异，确认问题已经关闭且没有引入新的风险。阻塞问题未关闭时，任务不能进入交付阶段。

如果评审发现的是普通技术问题，由 Agent 自行完成闭环；如果需要改变业务目标、架构方向、公共接口或数据流，则返回用户重新决策。

### AI 评审的能力边界

独立上下文可以降低实施者的思维惯性，运行验证可以减少仅凭语言判断造成的误差，但 AI 评审不能保证发现所有问题。其效果取决于方案是否清楚、代码范围是否明确以及验证手段是否有效。

真实业务效果、外部平台行为和主观体验仍由用户验收。高安全风险或专业性较强的改动，可以在 AI 评审之外增加人工专项评审。

## 7. 如何控制 AI 修改风险

风险控制不依赖频繁点击“批准”，而是依赖以下机制：

- **方案边界**：目标、接口、数据流、影响范围和验收标准在实施前明确；
- **权限边界**：外部写入、发布、付费和不可逆操作需要单独授权；
- **独立评审**：方案与代码由实施者之外的 Reviewer 检查；
- **自动验证**：交付前必须运行统一的本地检查；
- **小步实施**：复杂功能分阶段完成，每一步都保持可验证和可回退；
- **异常升级**：重大偏差返回用户讨论，普通实现偏差由 Agent 处理。

因此，Agent 获得的是边界内的执行自主权，而不是不受约束的项目决策权。

## 8. 预期效果

这套流程希望同时获得三项结果：

1. 用户持续掌握项目的目标、架构方向和风险边界；
2. Agent 在方案确定后能够连续完成实施和技术闭环；
3. 交付质量由评审和运行证据保证，而不是依赖 Agent 的自我声明或用户的机械审批。

简而言之：

> 让人负责方向与责任，让 Agent 负责执行与验证，并在两者之间建立明确、可检查的边界。
