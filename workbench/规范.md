# 开发流程规范 v1

> 开发流程规范 v1：把重构工作台验证过的「立案 → 定档 → 方案 → 评审 → Gate → 实施 → 预飞 → 验收」骨架，迁移到**新功能开发**场景。差别只在三处：起点是**需求**而不是问题；方案核心是**接口与数据设计**而不是行为保持；预飞核验的是**新接口确实落地 + 没有越界/残留**而不是禁词归零。其余纪律一字不改。

任何要写进项目代码的功能开发都必须走本流程，不允许「想到哪写到哪」。

## 一句话流程

```
需求（立案查重 + 定档） → 方案（接口/数据/测试/验收 + checklist.yaml）
  → 方案评审（可证伪复核） → Gate 1（用户放行）
  → 实施（小步 commit，偏差即停） → 预飞自检（事实表全绿）
  → 代码评审（对照方案逐 commit） → Gate 2（用户验收，落验收记录、归档）
```

## 与重构流程的映射

| 重构流程 | 开发流程 | 说明 |
|---|---|---|
| 问题（Review Report） | **需求** | 立案单元：从「哪里不对」变成「要做什么」 |
| 重构蓝图 | **开发蓝图** | 跨多期的大功能整体设计 |
| 方案：基线 + 调用方盘点 + 表征测试 | 方案：**接口/数据设计 + 复用盘点 + 验收用例** | 从「保住旧行为」变成「定清新契约」 |
| 导出白名单 / 禁词归零 | **必含符号 / 禁留标记** | 预飞从「旧符号清零」变成「新接口落地 + 无越界残留」 |
| 重构报告 | **需求报告** | 全部档位需求总台账 |

## 待议项（不展开，立此存照）

- **需求与里程碑的关系**：多个需求合成一个发布里程碑时的批次管理；
- **演示与真实环境**：本项目 v0 阶段只跑通管线、默认 fixture，真实访问稳定性不纳入验收，后续是否单列一档「联调验收」。
