# 本仓操作规范

[← 返回规范主页](../规范.md)

在文件系统里管理开发过程文档，渲染成本地站点。核心机制：**元数据进 frontmatter，视图由脚本生成。**

## 三条铁律

1. **文件永不移动**——生命周期只体现在 frontmatter 的 `status` 字段，归档 = 改一个字段，不挪文件。
2. **生成文件只能生成，禁止手改**（首行带 `<!-- generated` 哨兵：`index.md`、`SUMMARY.md`、`需求报告.md`、`views/*`、`cases/*/index.md`、`assets/nav.js`）——视图与导航是查询结果，不是手工目录页。
3. **每次提交前跑 `python3 tools/build_views.py`**——frontmatter 缺字段、状态枚举非法、relates 悬空、视图漂移都会报错；`--check` 模式供 CI / pre-commit 用。

## 目录

```
reports/<需求标题>.md         # 需求库（文件名 = id）
cases/<过程文档名>/           # 过程文档目录（如 小红书竞品研究工具v0）
  开发蓝图.md                 # 仅蓝图级
  期<N>/
    方案v<N>.md
    checklist.yaml            # 本期检查清单参数表（预飞/评审共用的唯一事实源）
    方案评审<R>.md
    代码评审<R>.md
    验收记录.md
templates/                    # 建档模板（含 frontmatter 骨架与标准章节）
views/                        # 生成产物：active.md / by-case.md / all.md
tools/build_views.py          # 视图生成 + schema 校验（--check 供钩子/CI）
tools/build_site.py           # Markdown → 站点（site/，纯标准库 + assets 主题）
tools/serve_site.py           # 本地起站点
tools/preflight.py            # 预飞机械自检：命令 + checklist.yaml → 事实表
assets/                       # 站点主题（styles.css / app.js）+ 生成的 nav.js
需求报告.md                   # 生成：全部档位需求总台账
SCHEMA.md                     # frontmatter 字段与状态机定义
```

**命名总原则**：过程文档名 / 需求名 / 文档名用中文原名，不发明英文 slug。

## 怎么读（人与 AI 同一协议）

1. **看全局**：首页 `index.md`（活跃工作台）——还活着的东西全在这；全景/索引在侧栏「工作台」。
2. **看一个过程文档**：`cases/<case>/index.md`（生成）——按页内编号顺序读：**当前方案 →（旧版仅考古）→ 方案评审按轮 → 代码评审按轮 → 验收**。
3. **判断任何文档的现状**：只看 frontmatter 的 `status` / `verdict`，不从正文猜。
4. **AI/机器**：不必经站点——frontmatter 即数据库（字段定义在 `SCHEMA.md`），`views/all.md` 即全量索引，按 `type:` / `status:` / `case:` grep 直达。

## 怎么跑一个过程文档（规范流程 → 本仓操作）

| 规范步骤 | 本仓操作 |
|---|---|
| 1. 立案 | 查重后从 `templates/需求.md` 建 `reports/<需求标题>.md`，status=观察中；定档提议写入，用户一字裁决 |
| 1a. 轻量级 | 也立需求文件（`tier: 轻量级`，极简正文），不建方案页；合入后 status 直接拨 已交付 |
| 2. 写方案 | 建 `cases/<过程文档名>/期<N>/`（非分期 phase 固定 1），从 `templates/方案.md` 建 `方案v1.md`，参数表写 `checklist.yaml`；status=待评审 |
| 3. 方案评审 | `方案评审<R>.md`（verdict）；需修改 → 出 `方案v<N+1>.md`，旧版 status=需修改 |
| 4. Gate 1 | Gate 卡片在会话里给；决策落入方案「Gate 决策结果」章节；需求 status=开发中 |
| 5. 实施 | 功能分支按方案小步实施；完成后方案尾部补实施记录三章节，status=已实施 |
| 5.5 预飞 | `python3 tools/preflight.py --config cases/<过程文档名>/期<N>/checklist.yaml`，全绿才请评审 |
| 6. 代码评审 | `代码评审<R>.md`；预飞事实表可引用，判断项必须亲核 |
| 7. Gate 2 | `验收记录.md`；方案 status=已归档；需求 status=已交付 |
| 8. 回填 | 长期口径同步项目 README / AGENTS / CLAUDE.md（如需） |

每次改完跑 `python3 tools/build_views.py`（pre-commit 应拦漂移）。日常入口是 `index.md`。
