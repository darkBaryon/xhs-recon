# 三档分级与分期模式

[← 返回规范主页](../规范.md)

定档决定走多重的流程。**判据只看结构尺度与风险，不看「感觉重不重要」。**

## 三档判据

| 档位 | 判据 | 流程 |
|---|---|---|
| **蓝图级** | 跨多个子系统 / 新增对外公共接口 / 需要分多期交付的大功能 | 先出**开发蓝图**（整体设计 + 分期切分）→ 蓝图评审 → 逐期走标准级 |
| **标准级** | 单项目内多模块的新功能，接口和交付物明确，无跨系统改造 | 完整 8 步：需求 → 方案 → 评审 → Gate1 → 实施 → 预飞 → 代码评审 → Gate2 |
| **轻量级** | 少量文件、~200 行以内、无新增对外接口、看 diff 即可验收 | 简化流：需求一行 → 实施 → 测试/demo 证据 → 用户看 diff 点头 → 合入 |

## 定档程序

1. 设计职责在需求文件「定档提议」节给出**档位 + 一句话理由**（理由只引结构判据）；
2. 用户**一字裁决**（可 / 不行走 X 档）；
3. 裁决落回需求文件，需求升 `待计划`。

## 两根保险丝

- **就高不就低**：判据落在两档之间时取高档——多一轮评审的成本远小于设计失误返工。
- **升档不降档**：实施中发现实际尺度超出定档（接口面铺开了、要动到别的子系统），**立即停**按变更协议登记，由用户重新定档；不允许「悄悄按小档做完」。

## 标定尺寸（经验锚点，非硬阈值）

- 轻量级：单文件或紧邻几个文件、无新公共符号、无数据结构变更；
- 标准级：3–8 个模块、有新接口/数据模型、需要成套测试；
- 蓝图级：要画模块边界图、要排期、任一期都够得上一个标准级。

## 蓝图级分期结构

蓝图级先出 `开发蓝图`（整体目标 / 模块边界 / 接口契约草案 / 分期表 / 各期验收口径），蓝图评审通过后：

```
cases/<功能名>/
  开发蓝图.md          # 整体设计，status 走方案状态机
  期1/  方案v1.md  checklist.yaml  方案评审1.md  代码评审1.md  验收记录.md
  期2/  ...
```

每期独立走标准级 8 步；期与期之间：上一期未验收归档不开下一期实施。蓝图本身在最后一期验收后随之 `已归档`，期间若发现设计偏差出**蓝图 v2** 并重审（差量评审）。
