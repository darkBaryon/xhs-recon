# 评审规则

[← 返回规范主页](../规范.md)

评审是流程里唯一的「独立第二双眼睛」，质量地板由这几条硬规则托住。

## 亲跑复核

评审职责对方案/代码里的**每一个事实断言都要亲自复测**，不采信设计或实施的自测结论。预飞事实表可以引用，但凡涉及判断的项必须亲核（脚本能证明「命令退出 0」，不能证明「这个接口设计是对的」）。**评审记录里必须附验证记录**（跑了什么、看到什么）；无验证记录的评审作废重派。

## 可证伪质疑

每个章节至少提一个**可证伪的质疑**——能被一个具体事实证实或推翻的问题，而不是「建议优化一下」这类泛泛评语。

- ✗ 「错误处理可以更健壮」
- ✓ 「`xhs_cli_adapter` 在 `xhs` 命令不存在时返回什么？方案说降级 fixture，但 §5 接口签名里 `collect()` 没有表达失败路径——请指出降级在哪一层发生」

## 差量评审

第二轮起只评**增量**：v1→v2 改了哪、改对没有、有没有引入新问题；已通过的部分不重复刨。代码评审同理，按 commit 范围差量看。这样多轮评审不会线性膨胀。

## 小修不直通

评审结论 `需修改` 一律出 v+1 新文档，**哪怕只改一行**；旧版 `status: 需修改` 冻结不动。不允许「评审说改一下，设计直接在原文件改了当通过」——那样评审历史就断了。唯一例外：Gate 决策的落地允许在已通过文档原地修订，但必须新增「Gate 决策结果」章节并逐处标注来源。

## 范围哲学（多做即偏差）

开发场景的「白名单」是方案 §4 的**非目标**与 §5 的**接口设计**：

- 实施只能做方案列明的；**顺手多做的功能、多加的接口、多铺的抽象 = 偏差**，必须登记，由用户裁决去留。
- 「碰巧觉得以后可能用得上」不是保留理由——投机性泛化按死代码处理（见[实施纪律与终扫](实施纪律与终扫.md)）。
- 评审发现实现超出方案接口面（多导出了符号、多接了依赖），即便代码本身没 bug，也按偏差判 `需修改`。

## 评审记录要件

`方案评审<R>.md` / `代码评审<R>.md` 必填 `verdict`（通过 / 需修改 / 驳回），正文含：

- **结论**：一句话 + verdict；
- **逐章/逐 commit 核查**：每条带亲核证据；
- **可证伪质疑清单**：未关闭的标记为阻塞项；
- **建议栏**：评审中发现的新需求写这里（评审职责不立案，由设计统一立）。
