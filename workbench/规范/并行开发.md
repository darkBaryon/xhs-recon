# 并行开发

[← 返回规范主页](../规范.md)

> 本页管「多个功能怎么互不踩踏」（横向），与 8 步流程管的「一个功能怎么走到验收」（纵向）正交。背景：本仓由多个会话（Claude / Codex / 用户）并行作业，等价于「多名开发者共用一台机器」——团队里靠人手一台电脑天然获得的隔离，这里必须用 git worktree 手工补上。核心认识：**并行的难点不是分开，是安全地合回来**。

## 铁律五条

1. **一条功能线 = 一条分支 = 一个 worktree = 一个会话**。集成区（仓库主目录）**常驻 main**，只做三件事：合并、验收、跑 demo。任何功能开发都在自己的 worktree 里进行；任何会话不得在别人的目录里提交。
2. **动 git 前先对表**：`git branch --show-current`——共享目录的分支随时可能被别的会话切走，凭记忆提交/合并是事故之源（2026-07-02 实证：日志系统曾因此误合入他线分支）。
3. **main 是唯一汇合点，后合者解冲突**：功能线合 main 前，先在自己分支 `git merge main` 把先落地的改动拿进来、解掉冲突、CI 全绿，再合 main（`--no-ff`）——让 main 上的合并永远干净（合并队列的手动版）。合并动作只在集成区或临时 worktree 执行。
4. **契约先行**：两条线都要依赖的公共改动（models 字段、ports 签名、config 键、测试夹具）先**单独作为一个小 commit 落 main**，两条线各自基于它开工——把三方合并拆成两次双方合并。
5. **足迹相交则排队，生成物重跑不手合**：开工前问一句「这两个需求的改动足迹（要动的文件）相交吗？」相交（尤其 `run_research.py` 这类编排热点）就串行；生成文件（`workbench/index.md`、`views/*`、`SUMMARY.md`、`nav.js`、`需求报告.md`）冲突时随便取一边、合完重跑 `build_views.py`，永不手工合。

## 操作速查

```bash
# 开一条功能线（在集成区执行；目录建在仓库外侧，互为兄弟）
git worktree add ../wt-<功能名> -b feat/<功能名>

# 功能线合 main 前的自查（在功能 worktree 内）
git merge main            # 后合者解冲突
uv run pytest -q          # 全绿再申请合并

# 合并（在集成区，确认自己在 main 上）
git branch --show-current
git merge --no-ff feat/<功能名>
cd workbench && python3 tools/build_views.py   # 生成物重跑

# 收线
git worktree remove ../wt-<功能名>
```

## 补充口径

- **Feature flag 解耦合并与发布**：没做完的功能用配置开关关死（如 `comments.enabled: false`）也可以合 main——分支存活时间越短，冲突越小。
- 各会话的日常产物（`data/`）在各自 worktree 内互不相干，均为 gitignore 运行产物。
- 本页规则对人和 AI 会话同等生效；交接仍只认文档 + git diff。
