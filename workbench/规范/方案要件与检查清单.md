# 方案要件与检查清单

[← 返回规范主页](../规范.md)

## 双库与状态机

工作台两类文档对应两个「库」：

- **需求库**（`reports/`）：要做什么。状态机 `观察中 → 待计划 → 开发中 → 已交付`（+ `已作废`）。
- **过程文档库**（`cases/`）：怎么做、做得对不对。方案/蓝图状态机 `草拟中 → 待评审 → 需修改 → 已通过 → 已实施 → 已归档`。

字段与枚举的强制定义在 [SCHEMA.md](../SCHEMA.md)，由 `build_views.py` 校验。

## 方案标准章节

从 `templates/方案.md` 起稿，**九节缺一不可**（无内容也要写「无 / 不适用」）：

1. **给用户的摘要**：≤10 行人话；至少一组 before/after（现状怎样 → 做完怎样）；拍板清单（每项一句话利弊，无则明写「无」）；不用未定义私有术语。
2. **起点**：起点 commit / 当前文件盘点——一切 diff、复测、对账以它为参照（非 git 仓库则盘点起点文件列表）。
3. **现状与约束盘点**：写方案当时实测，带 `文件:行号`。必须含**复用盘点**：本功能要新建的能力，现有代码里有没有可复用的出口？避免重复造轮子。约束（只读边界、敏感字段、不碰的范围）逐条列。
4. **目标 / 非目标**：目标 = 本期要交付的功能点；**非目标显式列出不做的范围**（这是范围哲学的合同，见[评审规则](评审规则.md)）。
5. **接口与数据设计**：新增/变更的对外接口签名、数据模型（dataclass / schema / 表结构）、模块边界与依赖方向。这是开发方案的核心——把契约定清，实施只是填充。
6. **实现步骤**：小步、每步可独立 CI 全绿 commit；步骤号与 commit message 对应。
7. **测试策略**：单测 / 集成 / 验收用例；行为断言优先于实现细节断言；列出验收用例的输入与期望输出；标注盲区。
8. **验收标准 + 检查清单 YAML**：必须机械可核验。参数表落同目录 `checklist.yaml`（唯一事实源，预飞/评审共用），此处只写人话解释。
9. **风险与禁止事项 / 回滚口径**：本期最值得知道的风险；越界禁止项（如本项目禁止代理池/验证码/自动互动）；未合主线前怎么弃子。

## 检查清单 YAML（方案与评审之间的合同）

`checklist.yaml` 把「验收标准」翻译成机器可跑的参数表，逻辑固定、参数每期换。四类检查：

```yaml
必过命令:                # 退出码须 0；expect_empty=true 时 stdout 也须空（grep/diff 类）
  - { name: lint,  run: "ruff check ." }
  - { name: tests, run: "PYTHONPATH=. pytest -q" }
  - { name: demo,  run: "python -m src.pipelines.run_research --config configs/sample.yaml" }
必存产物:                # 实现完成后这些路径必须存在
  - data/exports/report_input.md
必含符号:                # 本期新增接口/导出必须落地（grep 命中 ≥1）
  - { grep: "class FixtureAdapter", in: "src/adapters" }
禁留标记:                # 调试残留 / 越界功能，全仓须 0 命中
  - { grep: "proxy_pool", in: "src" }
  - print(            # 调试输出残留
```

预飞机械执行（见[流程、自检与 Gate](流程、自检与Gate.md)）：**①** 实施完自检；**②** 评审拿事实表只做判断；**③** 合入后沉淀为常驻守护。把评审里的体力活前置消化，评审拿到的是「事实表 + 待判断项」。
