# 流程、自检与 Gate

[← 返回规范主页](../规范.md)

## 流程（状态机，蓝图级/标准级）

```
1. 立案      设计职责查重后建需求；定档提议 → 用户一字裁决；
            升「待计划」
2. 写方案    设计职责出 v1（含摘要 / 接口数据设计 / 复用盘点 / checklist.yaml），
            status=待评审
3. 方案评审  评审职责出「方案评审 #N」；需修改 → 设计出 v+1 回到 3
            （标准级返工 1 轮封顶 = 评审至多 2 轮，#2 仍需修改升蓝图级）
4. Gate 1   用户确认；需求 → 开发中
5. 实施      实施职责在功能分支按方案小步实施；完成后在方案页追加实施记录三章节
5.5 预飞自检 跑 checklist 脚本（项目级命令 + 本期 YAML 参数表），全绿才请评审
6. 代码评审  评审职责对照方案逐 commit 查 diff（含偏差章节对账），出
            「代码评审 #N」；需修改 → 实施职责修复，轮次+1 回到 6（差量评审）
7. Gate 2   用户验收；设计职责写「验收记录」、合入主线、方案 → 已归档、
            需求 → 已交付
8. 回填      长期口径同步到项目 README / AGENTS / CLAUDE.md（如需）
```

- 步骤多可分批「实施 → 代码评审」循环：评审页注明覆盖的步骤/commit 范围；上一批阻塞未关闭不开始下一批实施。
- 轻量级走简化流：需求一行 → 实施 → 测试/demo/diff 证据 → 用户看 diff 点头 → 合入。

## 实施记录三章节（实施职责追加到方案页）

实施完成后追加这三节，齐备即把 `status` 流转 `已实施`，这是触发代码评审的信号：

- **实施记录**：每步 commit hash + 一句话说明 + 验收标准逐项自查；
- **实施偏差记录**：与方案不符处（哪怕很小；无则写「无」）——注意：**方案明示条目未执行也属偏差**，必须登记（漏做却写「无偏差」即为失实）；
- **蓝图验证备忘**：仅蓝图级分期功能必需（接口契约是否够用、是否建议修蓝图）；其余写「不适用」。

## 预飞机械自检

实施完成、交评审之前，必须先跑一遍 `tools/preflight.py`（项目级命令 + 本期 `checklist.yaml`），全绿才能流转 `已实施` 请评审；不绿先修。

目的：把评审里的体力活（逐条跑命令、grep 接口有没有落地、grep 有没有调试残留或越界功能）前置消化，评审拿到「事实表 + 待判断项」；同时杜绝「自我感觉良好」式交付——自查盲区由机械检查兜底。

## Gate 卡片与用户职责边界

设计职责每次请求 Gate 必须附卡片：

- **结论一句话**（这次要放行/验收的是什么）；
- **拍板项**（用户唯一不可替代的职责：业务取舍 / 不兼容变更 / 花钱；无则明写「无」）；
- **抽查菜单**（1–3 条现成命令或查看点，供用户选用，可不用）；
- **风险提示**（本次最值得知道的一条）。

用户职责边界：定档 = 读一句话理由 + 一字裁决；Gate 1 = 读摘要 + 处理拍板项 + 确认评审结论为通过；Gate 2 = 三档任选（轻验收 = 看指标表 / 抽查 = 跑菜单一条 / 实测 = 起服务或跑 demo 点两下，涉及对外行为或数据落库时推荐实测）。技术正确性由评审职责把关，用户不重复把关。
