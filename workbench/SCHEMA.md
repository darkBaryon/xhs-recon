# Frontmatter Schema

所有 `reports/` 与 `cases/` 下的 `.md` 必须以 YAML frontmatter 开头。**frontmatter 即数据库**：现状只看字段，不从正文猜；`build_views.py` 强制校验，不合规提交即拦。

## 公共字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `title` | ✓ | 文档标题，沿用命名约定（`<简称> 期<N> 方案 v<N>` 等） |
| `type` | ✓ | 见下方类型枚举 |
| `status` | 按类型 | 见下方状态机 |
| `created` | ✓ | `YYYY-MM-DD` |

## 需求（reports/）

立案单元，对应一个「要做的功能」。

- `type: 需求`
- `id`：与文件名一致，如 `小红书竞品研究工具v0`
- `status`：`观察中 → 待计划 → 开发中 → 已交付`；另有 `已作废`
- `tier`：定档结果（`蓝图级 / 标准级 / 轻量级`）；定档前可暂缺，需求报告归入「未定档」
- `relates` 由过程文档反向引用，需求本身不写
- 终态：`已交付`、`已作废`

轻量级需求正文极简（背景一句 / 目标 / 验收口径 / diff 量级），不建方案页。

## 过程文档（cases/）

| type | 必填附加字段 | status |
|---|---|---|
| `开发蓝图` | `case`, `relates` | 同方案状态机 |
| `方案` | `case`, `phase`, `version`, `relates` | `草拟中 → 待评审 → 需修改 → 已通过 → 已实施 → 已归档` |
| `方案评审` | `case`, `phase`, `round` | 无状态机，必填 `verdict`（通过/需修改/驳回） |
| `代码评审` | `case`, `phase`, `round` | 同上 |
| `验收记录` | `case`, `phase` | 无状态机 |

- `relates`：列表，元素为需求 `id`；蓝图/方案必填，评审/验收可省（沿用所属方案的关联）。
- `phase`：分期功能从 1 递增；非分期功能固定 `1`。
- `version` / `round`：方案版本号 / 评审轮次，从 1 递增。
- 终态：方案/蓝图的 `已归档`。

## 视图定义（views/，生成）

| 生成物 | 内容 |
|---|---|
| `index.md` | 站点首页 = 活跃工作台（所有非终态文档，按 case 分组） |
| `views/active.md` | 同首页，仓内浏览用 |
| `views/by-case.md` | 每个过程文档全量文档树 + 关联需求反链 |
| `views/all.md` | 全量平铺表，按 created 排序 |
| `需求报告.md` | 全部档位需求总台账，按 tier 分组，含关联过程文档反链 |
| `cases/<case>/index.md` | 单个过程文档主页（阅读顺序） |
| `SUMMARY.md` / `assets/nav.js` | 站点侧栏 / 顶部下拉导航数据 |

首行带 `<!-- generated by tools/build_views.py` 哨兵的文件**只能由脚本生成**，禁止手改。
