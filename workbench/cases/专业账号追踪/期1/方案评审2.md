## 新引入问题扫查（v2 增量部分）

1. **§6 第二断言「开窗配置导出为全 0 配置的确定性子集/重排」——字节级不成立，须按投影读法落地**。亲核推演：window=3650 时 5 条 fixture 全在窗内（最老 2024-05-17，约 777 天），notes/accounts/account_rank 三件与全 0 逐字节相同；但**半衰开启**时 `note_selector` 的 `selection_reason` 从 `"top by interaction"`（`note_selector.py:26`）变为 `"top by interaction×recency"`（§4 明写），`note_score` 数值也随衰减变——而 `exporter.py:160-162` 把这两列都写入 `typical_notes.csv`，`exporter.py:54` 把 `note_score {:.0f}` 写入 `report_input.md` → 这两个文件按字节行比对「子集/重排」**必挂**，「两组断言均机械可验」按字面不成立。存在唯一合理的机械读法：**按 note_id / account_id 投影比集合**（选中的笔记集相同、账号集相同），该读法确定性成立且可执行。判**建议不判阻塞**，理由：建议1 的核心回归守护（全 0 双配置字节对比 + 既有期望值断言）已正确落地、足以兜住「默认行为不变」；此断言是 v2 附加的第二层验证，不动接口/步骤/checklist，修法唯一且局部——按下方建议栏口径实施、代码评审按此核对即可。
2. **起点漂移一格（非 v2 之过，记录在案）**：评审 #1 时 `git log -1` = f727dc3，本轮 HEAD 已至 6f1bdea；`git diff --stat f727dc3 6f1bdea` 仅 `README.md +2/-1`，不含 no-touch 监控的三个文件，checklist 命令语义不受影响。实施分支实际基于 HEAD（6f1bdea），实施记录写明即可。
3. 其余增量（§2 复用盘点段、§3 非目标补「机械兜底见 checklist」、§7 验收补 no-touch 一句）与正文/checklist 对账无矛盾。
