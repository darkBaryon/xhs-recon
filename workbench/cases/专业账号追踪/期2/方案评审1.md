**可证伪质疑 Q-S4（阻塞项）：FetchResult.ok 与部分失败时 notes 消费路径**

方案 §4 部分失败口径：`error = "creator fetch failed: id1,id2"`（非 None），则 `result.ok == False`；但 `result.notes` 非空（成功账号已保留）。

方案 §4 pipeline 步骤5 说「用 creator notes 的 nickname 回填 watchlist」，步骤5 没有条件分支——即无论 ok 为 True 或 False，都消费 `result.notes`。这是方案隐含的行为：**pipeline 必须在 ok=False 时仍读取 result.notes**（而非将其视为空）。

问题：方案未在接口规格层明确「ok=False 时 notes 字段含部分成功结果」的语义，也未在 pipeline 步骤4 写出分支——「部分失败：成功保留、失败入 error」在 §4 adapter 描述里，但 pipeline 步骤里没有呼应「当 ok=False 时仍取 notes」这一非直觉行为。

现有 `FetchResult.ok` 语义（models.py:74-76）：`error is None → ok`。ok=False 通常意味着「操作失败」，调用方倾向于跳过 notes。方案没有解释「creator 场景下 ok=False 但 notes 非空是正常态」这一设计决策，实施者可能将 ok=False 当全失败处理，静默丢弃部分成功的 notes。

**判定：阻塞项**——方案 §4 需补充：pipeline 步骤4 明确写出「不论 ok 值，始终消费 result.notes 做 creator_notes.csv；ok=False 时额外 warning + 日志 error 字段」；或考虑让 adapter 仅在全失败时才设 error（与 ok=False 语义对齐），部分失败用 warning 日志而非 error 字段。若不在方案层钉死，实施时极易静默丢数据。

## 阻塞项（不改会出错）

### 阻塞1：部分失败时 ok=False 但 notes 非空——pipeline 消费路径须在方案层钉死

**问题**：`MediaCrawlerAdapter.fetch_creator_notes` 部分账号失败时返回 `FetchResult(notes=<成功部分>, error="creator fetch failed: id1,id2")`，此时 `result.ok == False`（因为 `error is not None`）。现有系统中 ok=False 通常意味着「跳过 notes」（_apply_time_window 第67行 `if not result.ok: filtered_results.append(result)` 直接追加，aggregate 后续仍消费——但 creator notes 不走 aggregate，pipeline 须直接访问 `result.notes`）。

方案 §4 pipeline 步骤4 仅写「NotImplementedError → 降级」，没有写「ok=False 时仍消费 result.notes」。步骤5 直接说「用 creator notes 的 nickname 回填」，暗示不论 ok 都取 notes，但这是隐含推断，不是接口规格。

**风险**：实施者若按 ok=False=全失败处理（符合现有惯例），将跳过 notes 导致 creator_notes.csv 静默为空——与方案「成功保留」口径相悖，且无测试能直接捕获（单测可能只测全成功或全失败）。

**要求 v2 做的事（二选一）**：
1. 在 §4 pipeline 步骤4 显式写「无论 ok 值，始终消费 result.notes 导出 creator_notes.csv；仅当 ok=False 时额外输出 warning」；同时在 §6 单测策略里补「MediaCrawlerAdapter 部分失败用例：notes 保留且长度 = 成功账号数」。
2. 或重新定义部分失败 adapter 行为：仅全失败时 error 非 None（ok=False）；部分失败时 error=None（ok=True）但 command 字段或 raw_text 记录失败 id 供日志使用——这样 ok 语义一致但需修 §4 adapter 描述。
