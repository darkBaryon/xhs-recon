# 专业账号追踪 开发蓝图 v2

## 给用户的摘要

把 v0 的「关键词搜热帖」升级为「盯住本领域专业账号的最新发帖」：搜索改按时间排序+时间窗过滤（不再全是陈年爆款）；账号从热帖副产品变成一等公民——自动发现+手动补充组成 watchlist，用 MediaCrawler creator 模式取每个账号的最新 N 篇；最后按 账号×时间窗 导出选题素材（md+jsonl）给下游选题项目直接消费。

- Before：搜「留学辅导」→ 2024 年的万赞爆款 + 一堆蹭热点的杂账号。
- After：一条命令 → 专业账号榜（垂直度+活跃度打分，`account_profile.csv` 可复算）+ 每账号最新发帖 + 窗内选题素材 `topic_feed`。

拍板清单：

- **分期方式**：期1 时间维度修复 → 期2 creator 拉取与 watchlist → 期3 专业度打分与选题素材导出（三期，每期标准级大小；嫌慢可把期2/期3 合并，代价是单期评审面变大）。
- **MediaCrawler fork 改法**：给 CLI 加 `--sort_type` 可选参数透传（缺省不覆盖其配置默认值，最小 diff），本地 fork 无上游同步压力。
- **用量口径**：watchlist 总量上限默认 10（auto_top_n + manual），creator 每账号默认取 10 篇——个人自用，低频小范围。

## v1→v2 变更（供差量评审）

回应「蓝图评审 #1」（verdict 需修改，2 阻塞 + 4 建议）：

1. **关闭阻塞1（验收口径静默降级）**：源需求验收口径已回写（见需求处置历史 2026-07-02）——原「每账号最新 N 篇 `published_at` 均落窗内」与「最新 N 篇」语义冲突（不活跃账号必然出窗），改为：`creator_notes.csv` 按原样导出；**`topic_feed` 仅收窗内笔记（机械可验：全部 `published_at` 落窗内）**，出窗/缺时间的笔记不入 feed、在报告计数。§4 分期表验收口径与之逐条一致，期1 fixture 断言窗过滤机械可验、真实跑手动抽查。
2. **关闭阻塞2（分项得分缺导出载体）**：契约草案自相矛盾处修正——旧 5 文件（含 `account_rank.csv` 7 列）确实不动；专业度分项落**新增 `account_profile.csv`**（期3），列含 `vertical_ratio / recent_note_count / 综合分`，可复算。

5. **吸收知情项**：search 模式 `CRAWLER_MAX_NOTES_COUNT` 有下限钳制 20（`core.py:133-134`），creator 流无钳制——记入 §3 事实清单。

## 1. 整体目标 / 非目标

**目标**：交付需求 5 点——时间排序+时间窗、专业度打分、watchlist（自动+手动）、creator 取最新 N 篇、选题素材导出。研究核心继续平台无关。

**非目标**（与需求一致）：只读拉取、不做任何写操作；不做选题生成本身（下游项目职责）；不破坏 v0 既有 5 文件导出契约（只新增）；仅 xhs 单平台。

## 2. 模块边界与依赖方向

沿用 v0 铁律：`pipelines → core → models`；`adapters` 实现 `core` 端口、由 `pipelines` 注入；`core` 无 "xhs"、不碰原始 JSONL。本功能全部改动落在既有格子里，不新增层：

```
configs/*.yaml ──► pipelines.run_research           # 编排追加：时间窗过滤 → watchlist 合成 → creator 拉取
                        │
                        ▼
        ┌──────────────── core ────────────────────┐
        │ keyword_expander  aggregator              │
        │ account_ranker(+专业度 v1，期3)            │
        │ note_selector(+时间衰减，期1)              │
        │ watchlist(新，期2；只吃归一化 id)           │
        │ exporter(+4 文件，期2/3)                   │
        └───────────────┬───────────────────────────┘
                        ▼
                  models(+AccountRank 分项字段)   adapters
                                                  ├ fixture_adapter(+creator fixture，期2)
                                                  └ mediacrawler_adapter(+sort 透传，期1；+creator 命令，期2)

MediaCrawler fork：cmd_arg/arg.py 加可选 --sort_type 透传 config.SORT_TYPE（期1，唯一的 fork 改动，缺省不覆盖）
```

## 3. 接口契约草案（跨期稳定，不轻改）

已 spike 并经蓝图评审 #1 逐条亲核成立的外部事实（2026-07-02）：

- `SearchSortType` 枚举：`general` / `time_descending` / `popularity_descending`（`media_platform/xhs/field.py:55-62`）；CLI 目前不暴露，仅 `config/xhs_config.py:24` 写死 + `core.py:155` 消费——透传参数只需动 `cmd_arg/arg.py` 一处。
- creator 模式 `--creator_id` **接受纯 24 位十六进制 user_id，xsec_token 可选**（`help.py:319-348`）；搜索 JSONL 每行含 `user_id`（`store/xhs/__init__.py:116`）→ 搜索发现的账号可直接喂 creator，无需额外一跳。
- creator 落盘 `xhs/jsonl/creator_contents_*.jsonl`，**字段与 search 完全一致**（同一 `update_xhs_note`）→ `parsers` 全复用；数量受 `CRAWLER_MAX_NOTES_COUNT` 控制。
- **知情**：search 流对 `CRAWLER_MAX_NOTES_COUNT` 有下限钳制 20（`core.py:133-134`）；creator 流无钳制，`notes_per_account: 10` 按字面生效。
- **知情**：creator 流按主页顺序分页截断（`client.py:618-649`），**置顶笔记会占据最前位**——「最新优先」不严格成立，对策见 §5。
- 登录态与 search 共用（qrcode/cookie 均可）。

契约：

```python
# 端口（ports.py）新增一个方法，基类默认 NotImplementedError（向后兼容，同期3 评论先例）：
ResearchAdapter.fetch_creator_notes(
    account_ids: list[str], limit: int, collected_at: str
) -> FetchResult      # operation="creator_notes"；notes/accounts 复用现有模型

# search 端口签名不变；排序不进端口——它是 adapter 构造参数（fixture 无排序概念）。

# models：只增不改。
AccountRank 新增分项（期3，default 兼容旧调用）：
    vertical_ratio: float = 0.0      # 垂直度 = creator 最新 N 篇中命中领域关键词占比
    recent_note_count: int = 0       # 时间窗内发帖数（活跃度）
WatchAccount（期2，新模型）：
    WatchAccount(account_id, nickname, source)   # source: "auto" | "manual"
# 「主页 URL→user_id」归一化在 adapters.parsers（平台专属）；core.watchlist 只吃归一化 id。

# config 契约（新增段，旧段不动）：
search:    { sort: time_descending, window_days: 30 }   # window_days=0 不过滤；sort 缺省不传→不覆盖 fork 配置
watchlist: { auto_top_n: 5, manual: [<user_id 或主页 URL>], max_total: 10 }
creator:   { notes_per_account: 10 }

# 导出契约：旧 5 文件（accounts/notes/account_rank/typical_notes/report_input）逐字节口径不变，新增 4 件：
watchlist.csv          # account_id,nickname,source（期2）
creator_notes.csv      # 列同 notes.csv，按原样导出，不做窗过滤（期2）
account_profile.csv    # account_id,nickname,vertical_ratio,recent_note_count,profile_score（期3，分项可复算）
topic_feed.md + topic_feed.jsonl   # 仅窗内笔记；出窗/缺时间不入 feed、报告计数（期3，下游消费的正式出口）
```

## 5. 风险与未决项

- **creator 模式未实测**：spike 只核了代码路径（纯 user_id 可用、落盘文件名、字段一致），没真跑过——期2 方案前置 spike 一次真实 creator 拉取，若实测纯 user_id 走不通，则降级为 manual 提供完整主页 URL，auto 部分推后。
- **置顶笔记破坏「最新优先」**：creator 流按主页顺序截断，置顶（多为旧的招牌帖）会挤占 N 篇名额——对策：`notes_per_account` 取值留余量（默认 10 已含余量），下游 `topic_feed` 窗过滤兜底；不做置顶识别（期3 若实测干扰大再议）。
- **auto top-N 窗内样本不足**：时间窗内搜索结果太少时，自动发现凑不满 top_n——兜底口径：取实际数量入 watchlist，不硬凑，报告标注「auto 实际 K/N」。
- **时间序的相关性噪声**：`time_descending` 下最新结果会混入蹭词水帖——靠关键词扩展 + 窗内互动/命中过滤兜底；打分精度沿用 v0 口径「允许各期调，不算偏差」。
- **creator 笔记无 source_keyword**：creator JSONL 的 `source_keyword` 为空，`parsers` 的 keyword 参数如何归一（`source_keywords=[]` 或标 watchlist 来源）在期2 方案定案。
- **拉取节奏**：watchlist × 每账号一次会话，串行单并发、总量上限 10 兜底；不做定时任务（跑一次算一次）。
- **published_at 零星缺失**：个别笔记 `time` 为空按 parser 既定降级，窗过滤把缺时间的笔记归为「窗外」并计数（细节期1 方案定）。
