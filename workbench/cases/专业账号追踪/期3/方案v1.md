## 给用户的摘要

期3 收官整个蓝图：给 watchlist 账号算**专业度分项**（垂直度 = creator 最新发帖命中领域关键词占比；活跃度 = 窗内发帖数），落 `account_profile.csv` 可复算；把窗内的 creator 笔记整理成**选题素材** `topic_feed.md`（人读）+ `topic_feed.jsonl`（下游选题项目直接消费）。旧文件（含期2 两件）逐字节口径不变，`report_input.md` 零改动。

- Before：期2 拉回 42 条 creator 笔记躺在 csv 里——哪个账号更专业、哪些帖新鲜可用，还得人眼扫。
- After：同一条命令 → 每账号垂直度/活跃度/综合分一张表 + 只含窗内新帖的选题素材包，出窗/缺时间的笔记不混入、计数留痕。

拍板清单：**无**（用量、窗口、导出契约均蓝图 Gate 已定；打分权重默认值属「允许各期调」精度口径，不构成取舍）。

## 3. 目标 / 非目标

**目标**：

1. `AccountRank` 新增分项字段（default 兼容旧调用，蓝图契约）+ `account_ranker.profile_accounts`：对 watchlist 账号计算 `vertical_ratio` / `recent_note_count` / `profile_score`；
2. 导出 `account_profile.csv`（列 `account_id,nickname,vertical_ratio,recent_note_count,profile_score`，蓝图逐字，可复算）；
3. 导出 `topic_feed.md` + `topic_feed.jsonl`：仅窗内 creator 笔记，出窗/缺时间不入 feed、在 feed 头部与日志计数（机械可验）；
4. pipeline 接线 + `ranking.weights` 新增 `vertical`/`activity` 两键（有内置默认，不配行为一致）。

**非目标**：不动 `report_input.md`（蓝图钉死）；不动采集面——adapters/fork/端口零改动，不追加任何拉取；不做置顶笔记识别（期2 实测已捕获现象，窗过滤天然滤掉 2023 旧置顶，暂无需额外机制）；不做中文分词/语义匹配（垂直度 v1 用文本包含命中，打分精度蓝图口径「允许各期调，不算偏差」）；不做选题生成（下游职责）；search 侧 `rank_accounts` 公式不动。

## 4. 接口与数据设计

```python
# src/models.py（只增不改；default 全零 = 旧构造/旧测试零改动）
AccountRank 新增：
    vertical_ratio: float = 0.0     # creator 最新 N 篇中命中领域关键词的占比（蓝图契约字段）
    recent_note_count: int = 0      # 时间窗内发帖数（蓝图契约字段）
    profile_score: float = 0.0      # 综合分（蓝图 account_profile.csv 既有列，落模型是承载所需的
                                    # 最小扩展——蓝图只列前两项，此为契约微扩，记入蓝图验证备忘）

# src/core/account_ranker.py（新增，不动 rank_accounts）
DEFAULT_PROFILE_WEIGHTS = {"vertical": 10.0, "activity": 1.0}
def profile_accounts(
    watchlist: list[WatchAccount], creator_notes: list[Note],
    domain_keywords: list[str], window_days: int, now_iso: str,
    weights: dict[str, float] | None = None,
) -> list[AccountRank]
# 每个 watchlist 账号一行（保持 watchlist 顺序），account_id/nickname 取自 WatchAccount，
# search 侧字段留默认 0（account_profile.csv 只导 5 列，不受影响）：
#   命中判定：note.title/body/tags 任一（casefold 后）包含任一 domain_keyword → 命中
#   vertical_ratio = 命中条数 / 该账号 creator 笔记数；0 条笔记 → 0.0
#   recent_note_count = filter_notes(该账号笔记, window_days, now_iso).kept
#     （复用 time_window；window_days<=0 = 不过滤 = 全部计入，与 search 段语义一致；
#      缺 published_at 按既定归 missing_time，不计入）
#   profile_score = w["vertical"] * vertical_ratio + w["activity"] * recent_note_count
# 纯函数、确定性、可测；core 无平台知识

# src/core/exporter.py（export_all 新增可选参数，None = 不写 = 旧行为；照 watchlist 先例）
export_all(..., account_profiles: list[AccountRank] | None = None,
                topic_feed: list[Note] | None = None,
                topic_feed_stats: WindowFilterStats | None = None,
                topic_feed_window_days: int = 0)
# account_profile.csv：account_id,nickname,vertical_ratio,recent_note_count,profile_score
#   （蓝图逐字 5 列；ratio 保留 4 位小数，score 保留 2 位——定死格式便于复算对账）
# topic_feed.jsonl：每行一条窗内笔记（json.dumps ensure_ascii=False）：
#   { account_id, nickname, source, note_id, title, body, tags: [...], url,
#     published_at, collected_at, like_count, collect_count, comment_count }
#   nickname/source 由 watchlist 参数（export_all 既有）按 account_id 映射；
#   互动三计数原样带出（creator 来源为 0=未采集，口径同期2，下游知情）
# topic_feed.md：头部一行机械对账统计
#   「窗口 N 天 · 入 feed K 条 · 出窗 X · 缺时间 Y · 账号 M」
#   随后按 watchlist 顺序分组：## nickname（account_id）→ 每条「- 日期 标题 url」
# 排序：账号间按 watchlist 序，账号内按 published_at 降序（确定性，e2e 可字节断言）

# src/pipelines/run_research.py（接线，creator 拉取之后、export_all 之前；
#                                watchlist 为 None 时整段跳过 = 旧行为零变化）
# 1. profiles = profile_accounts(watchlist, creator_notes, keywords,      # :113 的全集
#                                window_days, collected_at,
#                                config.get("ranking", {}).get("weights"))  # 与 rank_accounts 同源
# 2. feed_notes, feed_stats = filter_notes(creator_notes, window_days, collected_at)
#    日志一行：topic_feed kept/out_of_window/missing_time（对账口径）
# 3. export_all 追加传 account_profiles / topic_feed / topic_feed_stats / window_days

# config 契约（无新段；两份 sample 同步加两键，缺省=内置默认，行为一致）
ranking:
  weights: { note_count: 10, keyword_hit: 5, interaction: 0.01,
             vertical: 10.0, activity: 1.0 }    # 后两键期3新增，供 profile_score 复算
```

依赖方向不变：`pipelines → core → models`；profile 计算归 `account_ranker`（蓝图 §2 模块图「account_ranker(+专业度 v1，期3)」），窗过滤复用 `time_window`，无新层无新依赖。

## 6. 测试策略

- 单测（行为断言优先）：
  - `profile_accounts`：3 篇中 2 篇命中 → `vertical_ratio==0.5` 类可手算断言；窗内 2 篇 → `recent_note_count==2`；`profile_score == 10*ratio + 1*count` 复算；0 条 creator 笔记账号三值全 0 且仍出行；命中不区分大小写；tags 命中与 body 命中等价；
  - `exporter`：不传新参目录**不出现**三新文件（旧行为回归）；`account_profile.csv` 列头逐字；jsonl 逐行 `json.loads` 可解析且键齐；md 首行统计与 stats 一致；
  - pipeline：无 `watchlist` 段时 `profile_accounts` 不被调用（mock 断言）、三新文件不产出。
- e2e（挂点 `tests/test_pipeline_e2e.py:128` 既有双配置手法）：开 watchlist 跑 fixture → 三新文件存在；**旧 5 + 期2 两件与期2 版本产出逐字节一致**（`report_input.md` 零改动的机械证明）；新增用例：pin `collected_at` 下 jsonl 全部窗内 + 三计数对账（creator fixture 4 条的时间分布若与 pin 值不配合，允许在本期给 fixture **新增**时间错开的行——只增不改既有 4 行，避免破坏期2 断言；实施时按 pin 值定，偏差记录留痕）。
- 手动验收（Gate 2）：真实数据不必重采——直接对期2 实测产物复算：取 `data/exports/20260702T163155/creator_notes.csv` 任一账号，人工数命中/窗内条数，对照本期重跑（fixture 或真实均可）的 `account_profile.csv` 分项可复算；`topic_feed` 不含「路觅」那条 2023 置顶旧帖（窗过滤兜底的实证）。
- 盲区：垂直度文本包含命中对中文表述变体（如「留学生辅导」vs「留学辅导」子串关系）偏宽松，属 v1 精度口径（蓝图「允许各期调，不算偏差」），下游用分项列可自行重权。

## 8. 风险与禁止事项

- **时间序蹭词水帖混入 feed**：窗内笔记未做互动/语义过滤，水帖会进 feed——蓝图既定风险，靠 `account_profile` 分项供下游按账号加权自滤；本期不加过滤规则（多做即偏差）。
- **fixture 时间分布与 pin 值耦合**：e2e 窗内断言依赖 creator fixture 的 `published_at` 与 pin 的 `collected_at` 相对关系——§6 已定口径：只增不改既有夹具行，新增行时间错开覆盖出窗/缺时间分支。
- **profile_score 落模型属契约微扩**：蓝图 §3 只列两分项字段，综合分列却在 csv 契约中——本方案取「模型承载」解法（最小、可测），实施后记入蓝图验证备忘，由评审判定是否需修蓝图（预判：不需，csv 契约本身已含该列）。
- 禁止：动 `report_input.md` 任何字节口径；动 adapters/端口/fork；动 `rank_accounts` 公式与旧导出 7 件列；加过滤规则、分词依赖、新配置段；任何采集行为变化。
