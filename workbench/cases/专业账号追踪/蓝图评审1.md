## 结论

verdict: **需修改**。蓝图对 MediaCrawler 的 5 条 spike 事实断言经亲核**全部成立**（枚举三值、SORT_TYPE 写死、CLI 未暴露、纯 24 位 ID 可用、creator 落盘同 21 字段），端口/模型扩展向后兼容、分期尺寸与依赖顺序合理。但有 2 条阻塞：**①验收口径被单方面放宽**——需求写「每账号最新 N 篇 `published_at` 均落在时间窗内——机械可验」，蓝图三期验收无一承接（期1 降为「中位数+手动」），且未回写需求；**②期3「账号榜含分项得分」缺导出载体**——与「旧 5 文件逐字节口径不变」及需求非目标「不改旧文件格式与字段」自相矛盾，契约草案必须定清分项得分落在哪个文件。

### §4 分期尺寸与依赖

对照 `规范/三档分级与分期模式.md` 标准级锚点（3–8 模块、有新接口/数据模型、成套测试）：期1 = fork 1 处 + adapter 参数 + config 段 + pipeline 过滤 + note_selector 衰减 ≈ 5 个改动面；期2 = 端口方法 + 双 adapter 实现 + watchlist 模块 + 2 个导出 ≈ 6 面（本蓝图最重一期，仍在 8 以内）；期3 = ranker 改造 + 模型字段 + topic_feed 导出 ≈ 4 面。**三期均落在标准级带内，无超标也无凑数**。依赖顺序期1→期2→期3 合理：时间维度是 watchlist 活跃度与选题素材时间窗的前置。验收口径除阻塞1 外，机械/手动的标注诚实（真实采集手动、fixture 机械，沿 v0 判例）。

## 可证伪质疑

1. **【阻塞】需求的机械验收口径被三期悄悄放没了。** 可证伪点已证实：需求 `reports/专业账号追踪.md` 验收第 1 条「watchlist 每账号最新 N 篇（`published_at` 均落在配置时间窗内——机械可验）」；蓝图 §4 期1 =「中位数落窗内（手动验收）」、期2 =「每账号最新 N 篇（手动验收）」、期3 = 无窗内断言——「均落窗内+机械可验」在蓝图任何一期都找不到承接。且该需求口径本身与「最新 N 篇」语义冲突：不活跃账号（30 天没发帖）的最新 N 篇必然出窗，蓝图 §5「缺时间归窗外」也只覆盖缺失不覆盖出窗。正确处置不是静默降级，而是**回写需求**（如改为「窗内笔记数按账号统计并导出，出窗不算失败」或「creator 拉取后按窗过滤再导出」），使两份文档验收基线一致。未回写=阻塞（同 v0 蓝图评审4 判例）。
2. **【阻塞】期3「账号榜含分项得分（机械可验）」的导出载体不存在。** 可证伪点已证实：`src/core/exporter.py:134-157` account_rank.csv 列固定 7 项（无 vertical_ratio/recent_note_count）；蓝图 §3 导出契约承诺「旧 5 文件逐字节口径不变」、需求非目标写「不改旧文件格式与字段」；而新增 3 件中 watchlist.csv 只有单一 `score` 列（§3 第 85 行）、creator_notes.csv「列同 notes.csv」、topic_feed 是素材不是榜。四下里没有一个文件能装下分项得分——要么给 account_rank.csv 加列（违约，须显式豁免并回写需求）、要么新增账号榜文件（契约草案漏列）。蓝图级契约「跨期稳定不轻改」，这个洞必须在 v2 补上而不是期3 方案再说。
3. **【已关闭】spike 5 条外部事实是否属实？** 证伪测试：逐条打开 `../MediaCrawler` 对应文件比对行号与内容（见逐章核查 §3 第 1-6 条）。结果全部成立，且我补核了蓝图没写的 star-import 链路（base_config.py:144）确认「只动 arg.py 一处」不是虚言。关闭。
4. **【非阻塞，须在 v2 澄清】watchlist 在 core，URL→user_id 解析放哪？** 可证伪点：蓝图 §2 图把 watchlist 划入 core，§4 期2 交付含「parse URL→user_id」；若该解析实现在 core.watchlist，则 `xiaohongshu.com/user/profile` 模式匹配进 core，违反 §2 自述铁律「core 无 "xhs"」。v2 应一句话注明：URL 解析归 adapters（可对照 `help.py:336-346` 同款正则）或 pipelines 注入，core.watchlist 只吃纯 account_id。随阻塞项修订一并处理。
5. **【新风险，建议列入 §5】置顶笔记破坏「最新优先」。** 事实基础：`client.py:618-649` `get_all_notes_by_creator` 按主页分页顺序截断到 CRAWLER_MAX_NOTES_COUNT，**无任何时间排序**；xhs 主页置顶笔记排最前，有置顶的账号其「最新 N 篇」会被陈年置顶帖占席（N=10 时占 1-2 席就是 10-20% 污染），同时加剧质疑1 的出窗问题。期2 spike 建议专挑一个带置顶的账号实测，并决定是否在本项目侧按 published_at 重排/过滤。
6. **【新风险，建议列入 §5】auto 发现在时间窗下的样本不足兜底未定义。** 蓝图 §5 列了「时间序相关性噪声」但只讲选帖过滤；time_descending + 窗过滤后，窗内去重账号可能少于 auto_top_n（冷门领域或窗设 7 天时很现实），此时 watchlist 合成是照发不误、放宽窗、还是报告降级？行为未定义会在期2 变成实施自由裁量。建议 §5 登记、期2 方案定案（如「不足时如实入榜并在报告标注 auto 实得数」）。

## 建议栏（不立案，交设计职责）

- **回写需求验收口径**（对应阻塞1）：把「均落窗内——机械可验」修正为与「最新 N 篇」语义相容的口径（窗内计数导出/拉取后过滤，二选一），蓝图 §4 各期验收同步对齐。
- **补契约载体**（对应阻塞2）：在 §3 导出契约明确分项得分落点——推荐新增文件（如 account_rank_v1.csv 或并入 watchlist.csv 加列），避免动旧 5 文件；若决定给 account_rank.csv 加列，须同步回写需求非目标的「不改旧文件字段」措辞。
- **v2 注明 URL 解析归属**（对应质疑4）：一句话即可，core.watchlist 只收纯 account_id。
- **§5 补 2 条风险**（对应质疑5/6）：置顶笔记、auto 样本不足兜底。
- **用量口径知情项**（非改动要求）：search 模式 `CRAWLER_MAX_NOTES_COUNT` 有下限钳制 20（`core.py:133-134`），adapter 名义 limit<20 时实抓 20/关键词；creator 流无钳制，`notes_per_account: 10` 真实有效。
- 期1 方案落地 `--sort_type` 时，建议参数缺省值设为「不覆盖」（None/空跳过赋值），避免 fork 默认值反向覆盖 xhs_config，保持「最小 diff」承诺。
