# 专业账号追踪 开发蓝图 v1

## 给用户的摘要

把 v0 的「关键词搜热帖」升级为「盯住本领域专业账号的最新发帖」：搜索改按时间排序+时间窗过滤（不再全是陈年爆款）；账号从热帖副产品变成一等公民——自动发现+手动补充组成 watchlist，用 MediaCrawler creator 模式抓每个账号的最新 N 篇；最后按 账号×时间窗 导出选题素材（md+jsonl）给下游选题项目直接消费。

- Before：搜「留学辅导」→ 2024 年的万赞爆款 + 一堆蹭热点的杂账号。
- After：一条命令 → 专业账号榜（垂直度+活跃度打分）+ 每账号近 30 天最新发帖 + 选题素材文件。

拍板清单：

- **分期方式**：期1 时间维度修复 → 期2 creator 采集与 watchlist → 期3 专业度打分与选题素材导出（三期，每期标准级大小；嫌慢可把期2/期3 合并，代价是单期评审面变大）。
- **MediaCrawler fork 改法**：给 CLI 加 `--sort_type` 参数透传（最小 diff，不动其默认配置文件），本地 fork 无上游同步压力。
- **用量口径**：watchlist 总量上限默认 10（auto_top_n + manual），creator 每账号默认取 10 篇——个人自用，低频小范围。

## 1. 整体目标 / 非目标

**目标**：交付需求 5 点——时间排序+时间窗、专业度打分、watchlist（自动+手动）、creator 抓最新 N 篇、选题素材导出。研究核心继续平台无关。

**非目标**（与需求一致）：只读拉取、不做任何写操作；不做选题生成本身（下游项目职责）；不破坏 v0 既有 5 文件导出契约（只新增）；仅 xhs 单平台。

## 2. 模块边界与依赖方向

沿用 v0 铁律：`pipelines → core → models`；`adapters` 实现 `core` 端口、由 `pipelines` 注入；`core` 无 "xhs"、不碰原始 JSONL。本功能全部改动落在既有格子里，不新增层：

```
configs/*.yaml ──► pipelines.run_research           # 编排追加：时间窗过滤 → watchlist 合成 → creator 采集
                        │
                        ▼
        ┌──────────────── core ────────────────────┐
        │ keyword_expander  aggregator              │
        │ account_ranker(+专业度 v1，期3)            │
        │ note_selector(+时间衰减，期1)              │
        │ watchlist(新，期2)  exporter(+3 文件，期2/3)│
        └───────────────┬───────────────────────────┘
                        ▼
                  models(+AccountRank 分项字段)   adapters
                                                  ├ fixture_adapter(+creator fixture，期2)
                                                  └ mediacrawler_adapter(+sort 透传，期1；+creator 命令，期2)
                                                    └ parsers（复用，creator JSONL 同 21 字段）
MediaCrawler fork：cmd_arg/arg.py 加 --sort_type 透传 config.SORT_TYPE（期1，唯一的 fork 改动）
```

## 3. 接口契约草案（跨期稳定，不轻改）

已 spike 钉死的外部事实（2026-07-02，文件:行号见需求处置历史与期方案）：

- `SearchSortType` 枚举：`general` / `time_descending` / `popularity_descending`（`media_platform/xhs/field.py:55-62`）；CLI 目前不暴露，仅 `config/xhs_config.py:24` 写死 + `core.py:155` 消费——透传参数只需动 `cmd_arg/arg.py` 一处。
- creator 模式 `--creator_id` **接受纯 24 位十六进制 user_id，xsec_token 可选**（`help.py:319-348`）；搜索 JSONL 每行含 `user_id`（`store/xhs/__init__.py:116`）→ 搜索发现的账号可直接喂 creator，无需额外一跳。
- creator 落盘 `xhs/jsonl/creator_contents_*.jsonl`，**字段与 search 完全一致**（同一 `update_xhs_note`）→ `parsers` 全复用；数量受 `CRAWLER_MAX_NOTES_COUNT` 控制，最新优先。
- 登录态与 search 共用（qrcode/cookie 均可）。

契约：

```python
# 端口（ports.py）新增一个方法，基类默认 NotImplementedError（向后兼容，同期3 评论先例）：
ResearchAdapter.fetch_creator_notes(
    account_ids: list[str], limit: int, collected_at: str
) -> FetchResult      # operation="creator_notes"；notes/accounts 复用现有模型

# search 端口签名不变；排序不进端口——它是 adapter 构造参数（fixture 无排序概念）。

# models：只增不改。
AccountRank 新增分项（期3，default 兼容旧调用）：
    vertical_ratio: float = 0.0      # 垂直度 = creator 最新 N 篇中命中领域关键词占比
    recent_note_count: int = 0       # 时间窗内发帖数（活跃度）
Watchlist 条目（期2，新模型）：
    WatchAccount(account_id, nickname, source)   # source: "auto" | "manual"

# config 契约（新增段，旧段不动）：
search:    { sort: time_descending, window_days: 30 }   # window_days=0 不过滤
watchlist: { auto_top_n: 5, manual: [<user_id 或主页 URL>], max_total: 10 }
creator:   { notes_per_account: 10 }

# 导出契约：旧 5 文件逐字节口径不变，新增 3 件：
watchlist.csv                       # account_id,nickname,source,score
creator_notes.csv                   # 列同 notes.csv
topic_feed.md + topic_feed.jsonl    # 账号×时间窗选题素材（下游项目消费的正式出口）
```

## 5. 风险与未决项

- **creator 模式未实测**：spike 只核了代码路径（纯 user_id 可用、落盘文件名、字段一致），没真跑过——期2 方案前置 spike 一次真实 creator 拉取，若实测纯 user_id 走不通，则降级为 manual 提供完整主页 URL，auto 部分推后。
- **时间序的相关性噪声**：`time_descending` 下最新结果会混入蹭词水帖——靠关键词扩展 + 窗内互动/命中过滤兜底；打分精度沿用 v0 口径「允许各期调，不算偏差」。
- **creator 笔记无 source_keyword**：creator JSONL 的 `source_keyword` 为空，`parsers` 的 keyword 参数如何归一（`source_keywords=[]` 或标 watchlist 来源）在期2 方案定案。
- **采集节奏**：watchlist × 每账号一次抓取 = 多次浏览器会话，串行单并发、总量上限 10 兜底；不做定时任务（跑一次算一次）。
- **published_at 零星缺失**：个别笔记 `time` 为空按 parser 既定降级，窗过滤把缺时间的笔记归为「窗外」并在日志/报告计数（细节期1 方案定）。
