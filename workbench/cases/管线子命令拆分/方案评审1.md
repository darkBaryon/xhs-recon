### §4 接口与数据设计

**Q1（阻塞）— sync 的旧 5 件空表头设计与 README 推荐顺序自相矛盾**

按 §4/§8 设计推演 `search → sync → comments` 三命令序列：

1. `cli search`：产出 `accounts/notes/account_rank/typical_notes/report_input`（真实数据）+ 软链更新 `latest → search 跑目录`；
2. `cli sync`：`ranks = load_ranks_csv(latest/account_rank.csv)` 读到 search 的真实榜单 → `auto` 名额生效 → 产出 `watchlist/creator_notes/account_profile/topic_feed`（真实数据）**+ `accounts/notes/account_rank/typical_notes/report_input` 传空**（§4 原文：「本跑无搜索数据：accounts/notes/ranks/typical 传空 → 旧 5 件为空表头」）→ 新建独立 `run_dir`（§4「每命令独立 collected_at/run_id」）→ `_update_latest_link` 把 `latest` 指向**这个 typical_notes 为空表头的 sync 跑**（§8 原文自陈：「`latest` 会指向这个『不完整』跑」）；
3. `cli comments`：按 §4「`run_dir = latest 实际目录`；`typical = load_typical_csv(run_dir)`」——此时 `latest` 已被第 2 步的 sync 跑接管，`typical_notes.csv` 是空表头（0 行）→ `comments` 命令按 §4「latest 缺失/无 typical → 报错退出」的口径直接失败。

即：**只要用户按 §8 承诺的 README 推荐顺序操作，第三步必然失败**，因为第二步（sync）用空 typical 覆盖了 `latest` 指向。§8 把这个后果自我描述为「属预期链路」并归因于「README 写明推荐顺序」，但推演结果显示这条推荐顺序在方案自身设计下根本走不通——不是「预期的降级」，是「设计与其自身推荐用法互斥」。

亲核结论：**成立，阻塞**。这不是「sync 单独跑该不该产空表头」的口味问题，是「sync 产空表头 + sync 无条件抢占 latest」两个决定叠加后，直接废掉 comments 命令对 sync 产物的可达性，而「补全 sync/search 之后的评论」正是源需求验收口径第三条明文要求的能力。v2 必须二选一交代清楚：
  - (a) `sync` 不产生新的 `latest` 指向 / 不覆盖已有的 5 件旧文件（例如复用当前 `latest` 指向的运行目录，只新增 4 件 watchlist 侧文件到该目录——与「comments 写回原跑目录」同构，可复用同一套「写回」机制）；或
  - (b) `comments` 读取口径改为「最近一次含非空 `typical_notes.csv` 的运行」而非字面 `latest`（需要额外的目录扫描/索引逻辑，复杂度上升）；或
  - (c) 明确推翻「search → sync → comments」推荐顺序，改为「search → comments → sync」或强制文档提示「sync 后必须重新 search 才能再 comments」——但这与源需求「sync 独立、按需 comments」的动机相悖，且需要用户重新裁决。

方案 §8 已经预见到这条因果链（自己写出了「latest 会指向这个『不完整』跑」），但止步于「属预期链路」的自我合理化，未意识到这与同一节写的「README 写明推荐顺序 search→sync→comments」直接矛盾。判定为**方案自相矛盾**，需 v2 回答上述二选一（或提出第三种口径）并同步改 §4 sync 伪代码、§8 风险描述、checklist 若涉及验收脚本。

**Q2（阻塞）— `load_ranks_csv` 对空 `note_ids` 的往返规则未写明，朴素实现会产生数据形变**

`account_rank.csv` 的 `note_ids` 列由 `exporter.py:246` `_join(r.note_ids)` 生成，`_join` 即 `PIPE.join(xs)`。对 `note_ids=[]` 的账号（`relevant_note_count=0`），`_join([])` 产出空字符串 `""`。若 `load_ranks_csv` 按直觉实现 `row["note_ids"].split(PIPE)` 做反序列化，则：

```
PIPE.join([]).split(PIPE) == ['']   # 非 []，已亲测
```

即空列表往返后变成"一个空字符串元素的列表"，`AccountRank(note_ids=[''])` 而非 `AccountRank(note_ids=[])`——类型意义上仍是 `list[str]`（pydantic 校验不会报错），但**语义/数据完整性走样**。方案 §5 步骤 2 明确承诺「rank/typical CSV 写出→读回往返一致」为该步验收标准之一，若测试 fixture 恰好覆盖到 0 条相关笔记的账号（并非不可能——manual watchlist 账号未必总有 relevant note），该往返测试会因这个未言明的边界规则而失败或产出错误数据。

当前看，`note_ids` 本身在 `sync`/`comments` 两条消费路径里都不被读取使用（`build_watchlist` 只用 `account_id/nickname/account_score`；`_write_report` 只用 `nickname/account_id/relevant_note_count/keyword_hit_count/avg_interaction/account_score`），**本期不会因此产生错误行为**，但 `artifacts.py` 作为组装层的通用反序列化契约，若不在方案里钉死"空字符串 → 空列表"这条分割规则，是遗漏而非有意——属于「不改会出错」的候选（往返单测直接可证伪）。

**判定：阻塞**。v2 须在 §4 `load_ranks_csv` 签名注释或 §6 测试策略中显式写明：`note_ids` 列反序列化时空字符串须映射为 `[]`（`"" if not raw else raw.split(PIPE)` 一类规则），并在往返单测里覆盖"0 条相关笔记账号"这一边界样例。

**Q3（阻塞）— cli 子命令自行生成 `collected_at` 时的 `_now_iso` monkeypatch 生效面未钉死**

现有全部 e2e 测试统一用 `monkeypatch.setattr("src.pipelines.run_research._now_iso", lambda: ...)`（亲核 `tests/test_pipeline_e2e.py` 全部 9 处命中，路径全部固定为 `src.pipelines.run_research._now_iso`）。

`cli research` 子命令按 §4「与 `run_research()` 同一代码路径（直接调用）」——不管 `cli.py` 用 `from src.pipelines.run_research import run_research` 还是模块属性访问，`run_research()` 函数体内对 `_now_iso()` 的调用永远在**其定义所在模块** `run_research.py` 的全局命名空间里解析，与调用方 `cli.py` 的 import 方式无关（Python 闭包对全局名字的查找绑定到定义模块，不是调用模块）。**因此 `cli research` vs 旧入口的逐字节等价性 e2e，现有 pin 手法直接适用，无需额外钉死——此点判定关闭，不阻塞**。

但 `cli search`/`cli sync`/`cli comments` 三个独立子命令需要在 `cli.py` 自己的函数体里生成各自的 `collected_at`（它们不经过 `run_research()`，而是直接调 `_search_stage`/`_sync_stage`/`_comments_stage`，§4 三个阶段函数签名都要求调用方传入 `collected_at`）。若 `cli.py` 实现时选择 `from src.pipelines.run_research import _now_iso` 后在自己的函数体里调用 `_now_iso()`，这次调用解析的是 **`src.pipelines.cli` 命名空间里那个 import 时刻绑定的引用**，而非 `src.pipelines.run_research` 模块当前的属性值——对 `src.pipelines.run_research._now_iso` 的 `monkeypatch.setattr` patch 不会影响这个引用（Python `from...import` 语义：绑定的是当时的对象引用，不是模块属性的别名）。

方案 §6「e2e（pin `_now_iso` 手法照旧）」对 `search`/`sync`/`comments` 三条新 e2e 只字面沿用旧手法，未说明这三个新命令的 `collected_at` 生成点该用什么方式导入、测试该 patch 哪个字符串路径。若实施者按最顺手的 `from ... import _now_iso` 写法，`cli search`/`sync`/`comments` 的新 e2e 会在断言真实时间戳导出内容时得到**未被 pin 的真实墙钟时间**（不是崩溃，是悄悄不生效——比 Q1/Q2 更隐蔽），直接影响这三条命令自身导出内容的确定性验证，也间接影响「其余文件字节不变」这类断言的可靠性。

**判定：阻塞**。v2 须在 §4 明确 `cli.py` 中 `search`/`sync`/`comments`/`research` 四个命令生成 `collected_at` 的统一方式（建议：一律通过模块限定访问，如 `from src.pipelines import run_research as rr; rr._now_iso()`，或干脆复用 `run_research._now_iso` 这个函数对象本身而不重新绑定名字，以保证已有/新增测试可统一 `monkeypatch.setattr("src.pipelines.run_research._now_iso", ...)`），并在 §6 写明新 e2e 的 patch 目标路径。

其余接口设计核查：`SyncArtifacts` 用 pydantic BaseModel 但不进 `models.py`——有先例支撑（`WindowFilterStats` 同样定义在 `src/core/time_window.py` 而非 `models.py`，亲核确认），属组装层/模块局部数据载体的既有惯例，**不构成对"models 只增不改"约束的旁路**，判定关闭不阻塞。✅

**comments 写回原跑目录的"无损性"核实**：`_write_report`（`exporter.py:32-61`）仅使用 `AccountRank` 的 `nickname/account_id/relevant_note_count/keyword_hit_count/avg_interaction/account_score` 五个字段（亲核 `exporter.py:49-58` 循环体），不触及 `note_ids/vertical_ratio/recent_note_count/profile_score`；`avg_interaction`/`account_score` 走 CSV 的 `str(float)` 写出与 `float(str)` 读回，10 万次随机浮点验证格式化后（`:.0f`/`:.2f`）零字节差——**重写 `report_input.md` 在数值/字段层面确认无损，此点关闭不阻塞**。✅

## 阻塞项（不改会出错）

### 阻塞1：sync 的旧 5 件空表头 + latest 覆盖，废掉自身推荐的 search→sync→comments 使用顺序

见 §4 核查 Q1 详细推演。核心链条：`sync` 独立跑产出空表头的 `typical_notes.csv` → `_update_latest_link` 把 `latest` 指向这次不完整的跑（方案 §8 自陈）→ `comments` 按字面 `latest` 读 `typical_notes.csv` 拿到 0 行 → 报错退出。而这正是源需求验收口径第三条明文要求的能力入口。

**要求 v2 做的事**：在 §4/§8 明确二选一（或提出第三种口径）解决该矛盾——例如 `sync` 复用/写回当前 `latest` 指向的运行目录（与「comments 写回原跑目录」同构处理，只新增 4 件 watchlist 侧文件，不覆盖已有 5 件）而非新建独立跑目录顶替 `latest`；并同步修订 §4 sync 伪代码、§8 风险条目、需要新增的 README 推荐顺序表述。

### 阻塞2：`load_ranks_csv` 对空 `note_ids` 的反序列化规则未写明

见 §4 核查 Q2。`PIPE.join([]).split(PIPE) == ['']`（已实测），若不显式处理会导致 0 条相关笔记账号的往返数据形变，与 §5 步骤 2 承诺的「CSV 写出→读回往返一致」矛盾。

**要求 v2 做的事**：在 §4 `load_ranks_csv` 部分或 §6 测试策略里写明空字符串→空列表的显式映射规则，并要求往返单测覆盖 0 条相关笔记账号样例。

### 阻塞3：cli 子命令自生成 `collected_at` 的 `_now_iso` import/patch 口径未钉死

见 §4 核查 Q3。`cli research` 因与 `run_research()` 共享同一模块命名空间而安全，但 `cli search`/`sync`/`comments` 若各自在 `cli.py` 内用 `from...import _now_iso` 方式取用，既有 e2e pin 手法对其不生效，新增 e2e 可能悄悄用真实墙钟时间而非 pin 值，破坏确定性断言。

**要求 v2 做的事**：在 §4 明确四个 cli 子命令生成 `collected_at` 的统一实现方式（建议走模块限定访问而非裸 `from...import` 重新绑定名字），并在 §6 写明新增 e2e 对应的 `monkeypatch` 目标路径。

## 建议栏（实施期落实，不返工）

1. **§5 显式说明 checklist 与逐步 CI 的关系**：建议在 §5 开头补一句「本节『每步可独立 CI 全绿』指现有 `pytest`/`ruff` 套件；checklist.yaml 的 `demo-cli` 等全量命令在步骤 3（`cli.py` 落地）后才具备可跑条件，交评审前的终检以 checklist 全量为准（沿 `实施纪律与终扫.md` 口径）」，避免实施者在步骤 1-2 误跑 checklist 得到预期外的红。

2. **`SyncArtifacts` 落位建议顺带写清楚模块路径**：方案 §4 只说「pipelines 内部载体」，建议明确落在 `run_research.py`（与阶段函数同文件，符合"组装层私有"定位）还是新开 `src/pipelines/sync_artifacts.py`——不影响接口正确性，但影响 checklist 之外的代码组织一致性，实施期口头/commit 里交代一下即可。

## 结论

方案整体思路清晰：接口设计遵循既有依赖方向（`cli → run_research 阶段函数/artifacts → core → models`），checklist 的 no-touch 机械门槛把范围哲学卡得很死，无 B1/B2 或其他下一案内容夹带，行号断言全部亲核精确。comments 写回原跑目录的拍板项在数值精度层面确认无损（10 万次随机浮点验证）；`SyncArtifacts` 模型放置有先例支撑；`cli research` 与旧入口的等价性 pin 手法本身安全。

但方案存在**三处「不改会出错」级别的阻塞**：(1) `sync` 的空表头+latest 覆盖设计与其自己承诺的 README 推荐顺序 `search→sync→comments` 直接自相矛盾，按方案字面设计推演，`comments` 命令在推荐顺序下必然报错退出，这与源需求验收口径正面冲突；(2) `load_ranks_csv` 对空 `note_ids` 列的反序列化规则未写明，朴素实现会产生 `['']` vs `[]` 的数据形变，与方案自己承诺的往返一致性测试矛盾（已实测证实）；(3) `cli` 新增三个独立子命令自行生成 `collected_at` 时的 `_now_iso` import/patch 口径未钉死，存在悄悄使新 e2e 测试非确定的风险。三项均可被具体命令/推演证伪，且均要求 v2 在对应章节明确交代，不影响已核实为安全的其余接口设计部分。

verdict **需修改**，方案 v1 冻结，进入 v2 修订。
