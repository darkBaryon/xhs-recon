## 给用户的摘要

打通「创作者主页档案」这条已经端到嘴边的数据：fork 补一行让 jsonl 写出档案（数据早已抓到，只是出口没接），xhs-recon 解析后新导出 `creator_profiles.csv`——watchlist 每个账号一行**官方主页数据**（简介/粉丝/互动/标签含认证类/IP）。机构判定从「昵称猜」升级为「平台数据人判」，零额外采集请求。

- Before：判断「予橙」是不是正经机构，只能看昵称带不带「辅导」。
- After：`cli sync` 顺带产出每账号的简介（"XX教育科技旗下"）、粉丝量级、职业认证标签——扫一眼表就能定。

拍板清单：**无**（零额外请求、只新增文件、fork 改动一行且模式有先例——无新取舍；判定融合进打分属另案）。

## 2. 现状与约束盘点（写方案当时实测，spike 2026-07-03 源码亲核）

- **数据链路已全通只缺出口**（fork 侧）：creator 模式入口 `get_creators_and_notes` 在拉笔记前调 `get_creator_info`（`media_platform/xhs/core.py:199`），解析主页 `window.__INITIAL_STATE__` 的 `userPageData`（`extractor.py:52-69`）；`save_creator`（`store/xhs/__init__.py:184-230`）组装字段：`user_id/nickname/gender/avatar/desc/ip_location/follows/fans/interaction/tag_list`（tag_list 为 `{tagType: name}` 的 JSON 串，职业认证即在其中）+ `last_modify_ts`；分发到 `store_creator`——**`XhsJsonlStoreImplement.store_creator` 为空 `pass`**（`store/xhs/_store_impl.py:115`），csv/json 分支同样为 pass，仅 Mongo 分支实写（`:342`）。
- **写出命名规则**：`AsyncFileWriter._get_file_path` → `{crawler_type}_{item_type}_{日期}.{ext}`（`tools/async_file_writer.py:43`）——creator 模式 + `item_type="creators"` → **`xhs/jsonl/creator_creators_<日期>.jsonl`**；写 jsonl 的先例 `store_content` 即 `write_to_jsonl(item_type="contents")`（`_store_impl.py:109`），照抄。
- **零额外请求**：档案请求发生在既有 creator 流程内（每账号本来就要发），本案不新增任何网络行为。
- xhs-recon 侧现状：`fetch_creator_notes` 每账号独立子目录 + glob `creator_contents_*.jsonl`（`src/adapters/mediacrawler_adapter.py:185-191`）——档案文件与其同目录，加一个 glob 即可；`FetchResult` 无档案载体（`src/models.py:66-85`）——扩可选字段（默认空列表，向后兼容，先例：期2 `fetch_creator_notes` 端口默认实现）；计数归一化复用 `parsers.normalize_count`（`parsers.py:16-28`，"1.1万"→11000 已有单测）；导出复用 `_write_csv` + `export_watch_side` 可选参数模式（`exporter.py`，None=不写先例）。
- **复用盘点**：无重复造轮点——解析新建 `parse_creator_profiles_jsonl_lines`（схema 与 21 字段笔记完全不同，不复用 `parse_jsonl_lines`）；其余全部复用既有出口。
- 约束：`core` 无 "xhs"（档案解析归 adapters）；旧 12 件导出契约零变化（只新增 `creator_profiles.csv`）；fork 最小 diff（一行函数体，不动 `save_creator` 字段组装）；`account_profile.csv` 蓝图五列不碰。

## 3. 目标 / 非目标

**目标**：

1. **[fork]** `XhsJsonlStoreImplement.store_creator` 补写出（一行，`item_type="creators"`）；
2. 新模型 `CreatorProfile` + `FetchResult.profiles` 可选字段（默认 `[]`）；
3. `parsers.parse_creator_profiles_jsonl_lines`：档案行 → `CreatorProfile`（计数归一、`tag_list` JSON 串 → dict、缺字段降级空值）；
4. `MediaCrawlerAdapter.fetch_creator_notes` 顺带读回档案（glob `creator_creators_*.jsonl`，缺文件 → `profiles=[]` 不报错——软信号口径，旧版 fork 也能跑）；`FixtureAdapter` 新增可选 `creator_profiles_path`；
5. 导出 `creator_profiles.csv`（列：`account_id,nickname,fans,follows,interaction,tags,desc,ip_location`；tags 以 `类型:名称` 对、`|` 连接）；`export_watch_side` 加可选参数（None=不写）；
6. `_sync_stage`/cli `sync` 接线（`SyncArtifacts` 加字段）。

**非目标**：不做认证自动判定/打分融合（另案）；不对搜索候选逐个查档案（用量口径）；不改 fork 的 `save_creator` 字段组装与 csv/json 分支；不改旧 12 件契约；不加配置段（档案随 creator 拉取自动产出，无需开关——watchlist 未启用时本就不跑 creator）。

## 6. 测试策略

- 单测：parser 四组（见 §5-3）；adapter 双实现各两组；exporter 列头/None 不写/tags 连接格式；
- e2e（既有 pin 手法）：`cli search → sync` 链上断言 `creator_profiles.csv` 存在、行数与档案夹具吻合、**其余 12 件字节不变**；fixture 未配档案路径时文件为空表头（软降级机械可验）；
- 手动验收（Gate 2）：真实跑测试组（3 账号）→ `creator_profiles.csv` 3 行，抽 1 个机构号对照小红书主页：粉丝量级一致、tags 含主页可见的认证类标签（fork 改动的端到端实证）；
- 盲区：fork 侧无自动化测试（该仓不进 CI，步骤 1 手动验证 + Gate 2 真实跑双覆盖）；`userPageData` 的字段形态若遇平台改版会整体缺失——软信号口径已兜（profiles 空、主产物不受影响）。

## 8. 风险与禁止事项

- **认证字段的确切形态未实测**：spike 只证了 `tag_list` 通道存在，专业号认证的 tagType 具体叫什么（profession/official/其他）要等步骤 1 真实落盘才见分晓——若认证不在 tag_list 而在 `basicInfo` 独立字段，本案先交付现有字段（简介/粉丝已够人判），追加字段留下一案（fork 只动一行的口径不破）；
- **平台改版风险**：`__INITIAL_STATE__` 结构变化会让档案整体缺失——软信号口径（profiles 空不报错）保证主管线不受影响；
- **双仓提交**：fork 与 xhs-recon 分别 commit，代码评审两边都查（期1 先例）；
- 禁止：fork 除 `store_creator` 一行外任何改动；对候选账号新增档案请求；改旧 12 件契约列；加配置段。

## 9. 回滚口径

xhs-recon 弃分支即回滚；fork 的一行 commit 独立 revert 干净（留着也无害——只是多写一个没人读的文件）。已 commit 去留由用户裁决。
