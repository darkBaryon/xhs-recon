### §4 接口与数据设计

**Q1（阻塞）— `FixtureAdapter` 新参数 `creator_profiles_path` 未接入配置读取链，§6 承诺的 e2e 无法达成**

方案 §4 只写了 `FixtureAdapter.__init__(..., creator_profiles_path: str | None = None)` 这一层类接口改动，但**没有提及 `_build_adapter`（`run_research.py:37-61`）要同步改**。亲核该函数：

```python
def _build_adapter(config: dict) -> ResearchAdapter:
    ...
    creator_path = config.get("creator_fixture_path")   # 顶层键，仅这一个
    ...
    return FixtureAdapter(
        config["fixture_path"], comments_path=comments_path, creator_path=creator_path
    )
```

全函数没有任何位置读取或透传 `creator_profiles_path`。这意味着：即便按方案 §4 末尾「fixture 夹具：`tests/fixtures/creator_creators_sample.jsonl`」新增了夹具文件，只要 `_build_adapter` 不新增一行 `profiles_path = config.get("creator_profiles_path")` 并透传给 `FixtureAdapter(creator_profiles_path=profiles_path)`，任何通过 config 驱动的路径（包括 §6 承诺的 e2e：「`cli search → sync` 链上断言 `creator_profiles.csv` 存在、行数与档案夹具吻合」）都不可能拿到非空档案——`FixtureAdapter` 实例的 `creator_profiles_path` 永远是构造函数默认值 `None`，`profiles` 恒为 `[]`。

这不是「实施时顺手就能补上」的琐碎遗漏——它恰好卡在方案自己承诺的**验收门槛**上（§6「行数与档案夹具吻合」不可能通过），且是「哪个文件要改」这一范围哲学核心问题的一部分（`checklist.yaml` 必含符号只检查 `creator_profiles_path` 在 `fixture_adapter.py` 里出现，不检查 `_build_adapter` 是否真正读取转发，机械清单捕捉不到这个缺口）。

**判定：阻塞**。v2 须在 §4 补一条 `run_research.py` `_build_adapter` 的改动：新增顶层配置键（建议与既有 `creator_fixture_path` 同构命名，如 `creator_profiles_path`），读取后透传给两个 provider 分支下都要构造的 `FixtureAdapter`（含 `mediacrawler` 不可用降级分支，当前该分支第 58 行也漏了 `creator_path` 之外的新参数）。

**Q2（阻塞）— 方案「12 件」「13 件」文件计数与实测/checklist 三方不一致**

方案通篇（§2/§3/§5/§7/§8）反复断言「旧 12 件导出契约」，加 `creator_profiles.csv` 后「demo 产物 13 件」。实测：

```
uv run python -m src.pipelines.cli research --config configs/sample.yaml
✓ 导出 11 个文件 → data/exports/20260703T005856
```

（accounts / notes / account_rank / typical_notes / watchlist / creator_notes / account_profile / topic_feed_jsonl / topic_feed / comments / report_input，共 11 个 `paths` 键。）

即当前基线是 **11 件，不是「旧 12 件」**；加 `creator_profiles.csv` 之后应为 **12 件，不是「13 件」**。方案数字全篇错位一档。

同时 `checklist.yaml` 的「必存产物」列表本身只列了 **11 条**（10 条老文件 + `creator_profiles.csv`），其中**遗漏了 `comments.csv`**——`configs/sample.yaml` 的 `comments.enabled: true` + `fixture_path` 配置本就会产出 `comments.csv`（已实测确认存在于 11 件基线中）。checklist 头部注释写「必存产物...demo 跑完后须存在（13 件）」，与其自身罗列的 11 条数目也对不上。

按评审规则「亲跑复核」的要求，`checklist.yaml` 是「方案与评审之间的合同」「唯一事实源」——它本身携带的数字断言（13 件）与其罗列的条目数（11 条）就自相矛盾，且实测证明真实基线数目（11）、方案叙述数目（12/13）、checklist 罗列数目（11+缺1=有效10+1）三者互不相等。这不是无关痛痒的笔误：`comments.csv` 缺失在必存产物清单里意味着即便未来该文件被意外破坏，checklist 也不会报警——机械核验出现真实缺口。

**判定：阻塞**。v2 须：(a) 修正全篇「12 件/13 件」为准确数字（11 件基线 + `creator_profiles.csv` = 12 件）；(b) `checklist.yaml` 必存产物补上 `data/exports/latest/comments.csv`，并把头部注释的数字改成与实际罗列条目数一致。

**其余接口设计核查**：`CreatorProfile` 模型字段与 `save_creator` 落盘字段一一对应（`account_id←user_id`/`fans`/`follows`/`interaction`/`desc`/`ip_location`；`gender`/`avatar`/`last_modify_ts` 显式丢弃），无过度设计。`export_watch_side`/`export_all` 加可选参数的「None=不写」模式与既有 4 个已有可选参数（`watchlist`/`creator_notes`/`account_profiles`/`topic_feed`）完全同构，`export_all` 委托透传（264-390 行的 `paths.update(export_watch_side(...))` 结构）确认可以只加一个新形参、无需改内部逻辑。✅

## 结论

方案对 fork 侧数据链路的核心断言（`get_creator_info` 时序、`extractor` 返回形态、`save_creator` 字段组装、`store_creator` 空 `pass`、`crawler_type_var` 文件名推导）**全部亲核成立**，「零额外请求」「fork 只动一行」的范围收敛判断可信，`export_watch_side`/`export_all` 的 None-可选参数接线模式与既有先例同构，`SyncArtifacts` 透传两处调用点已被方案明确提及、接线完整。`tag_list` 缺失时的崩溃风险经实测复现，但确认是 `save_creator` 现状既有、与本次改动无关、且爆炸半径被 xhs-recon 侧现有子进程隔离机制兜住——不构成阻塞，仅建议登记。

但方案存在**两处「不改会出错」级别的阻塞**：(1) `FixtureAdapter` 新增的 `creator_profiles_path` 参数没有配套设计 `_build_adapter` 的配置读取与透传逻辑，导致方案自己在 §6 承诺的「`creator_profiles.csv` 行数与档案夹具吻合」e2e 在当前设计下不可能通过；(2) 方案全篇反复使用的「旧 12 件/加后 13 件」文件计数与实测基线（11 件）、`checklist.yaml` 实际罗列条目数（11 条，且遗漏 `comments.csv`）三方互不一致，机械核验清单本身存在真实缺口。两项均可被具体命令/实测证伪，且均要求 v2 在对应章节明确交代，不影响已核实为可信的其余部分（fork 数据链路断言、导出接线模式、`SyncArtifacts` 透传完整性、B006 无冲突）。
