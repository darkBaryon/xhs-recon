### Q-1【已关闭】：期3 遗留 e2e 测试与日志系统结合后是否向 `data/logs/` 写文件？

**质疑**：`tests/test_pipeline_e2e.py` 的 `_cfg` 函数未设 `logging` 键，`run_research` 读 `config.get("logging", {})` 返回 `{}`，`configure_logging` 用默认值 `file_enabled=True`、`dir="data/logs"`。则 3 个 e2e 测试每次运行都会在真实 `data/logs/` 留文件，而非 tmp_path 隔离。

**验证**：
1. 删除 `data/logs/`，单跑 `pytest tests/test_pipeline_e2e.py -v`；
2. 跑后 `ls data/logs/`：存在 `run-2026.log`（来自 monkeypatch 注入"2026"的 e2e 测试）和 `run-2026-07-02T*.log`（`test_pipeline_end_to_end` 无 monkeypatch，产实时时间戳文件）；
3. 单跑 `pytest tests/test_pipeline_logging.py tests/test_logging_setup.py` 后 `data/logs/` 不存在——确认只有 e2e 测试有此副作用。

**结论**：确认。期3 e2e 测试在日志系统落地后产生磁盘副作用（每次 pytest 向 `data/logs/` 写文件）。影响评估：
- 不污染 git（`data/logs/` 在 .gitignore）；
- 不影响测试正确性（所有测试通过）；
- 副作用使 preflight 的"必存产物 `data/logs`"在 pytest 后天然满足，稍弱化了 demo 步骤的独立性；
- 日志文件会随时间累积（每次 CI/pytest 都追加），但 gitignored 无运维压力。

**是否阻塞**：不构成阻塞（测试全通过，不污染 git）。记为非阻塞建议（见建议栏）。

### Q-2【已关闭】：`file_enabled` 判断用 `is False` 而非 `not`，是否存在行为边界？

**质疑**：`logging_setup.py` L58 `if cfg.get("file_enabled", True) is False:`——当传入 `file_enabled=0` 或 `file_enabled="false"` 时，`0 is False → False`、`"false" is False → False`，不会关闭文件 handler，与预期语义不符。

**验证**：
- Python 实验：`False is False → True`；`0 is False → False`；`"false" is False → False`。
- 实际使用场景：配置来自 `yaml.safe_load`，YAML 的 `false` 解析为 Python `False`，`True is False → False`，`False is False → True`。YAML 场景下 `is False` 等价于 `== False`。
- 若程序化调用 `configure_logging({"file_enabled": 0}, ...)`，文件 handler 不会被关闭（silently creates file）。

**结论**：YAML 配置场景安全，行为符合预期。程序化传 `0`/`"false"` 的边界不稳健，属 code smell，非功能缺陷。不构成阻塞。记为非阻塞建议。

### Q-3【已关闭】：`configure_logging` 签名中 `cfg: dict | None` 接受 None，是否意味着某调用路径传 None？

**质疑**：方案 §4 签名写 `cfg: dict`，实现写 `cfg: dict | None`，方案 `run_research` 传 `config.get("logging", {})` 永远是 `dict`，那 `None` 分支为什么存在？是否暗示了方案外的调用路径？

**验证**：亲读 `configure_logging` 函数体 L43 `cfg = cfg or {}`——如果调用方直接传 `None`，`cfg or {}` 会把它规范化为 `{}`，使函数表现等同 `cfg={}`。`run_research.py` 传 `config.get("logging", {})` 永远是 dict。`None` 分支是防御性设计，不暗示任何方案外调用路径。接口面未扩展（`cfg: dict | None` 比 `cfg: dict` 更宽容，是 additive）。

**结论**：非偏差，非阻塞。`None` 接受是防御性设计，不扩展接口面。已关闭。

### Q-4【已关闭】：`run_research` 中 search 阶段 log 顺序是否与方案 §4 伪码一致？
