## 可证伪质疑及验证记录

### Q-A（新质疑）：N2 字节级对照测试——collected_at 时间戳天然导致两次运行文件不等？

**质疑**：v2 §6 说「同 fixture 输入、日志各开关组合，6 导出文件逐字节相等」。`run_research` 在函数入口调用 `_now_iso()`（返回 `datetime.now(tz=timezone.utc).isoformat()`）作为 `collected_at`，该值传入 `parse_note`/`parse_account`/`parse_comment`，最终写入 `notes.csv:collected_at`、`accounts.csv:first_seen_at/last_seen_at`、`comments.csv:collected_at`。两次 run_research 调用之间哪怕相差毫秒，这三个文件的内容必然不同，字节级对照天然失败。v2 方案描述未提及需固定时间戳注入。

**亲核路径**：

1. 读 `src/core/exporter.py`：`notes.csv` 导出列含 `collected_at`（L112），`comments.csv` 含 `collected_at`（L169），`accounts.csv` 含 `first_seen_at`/`last_seen_at`（L85-86），这三个字段全部来自 `run_research` 的 `collected_at = _now_iso()`。

2. Python 实验：
```python
from datetime import datetime, timezone
import time
t1 = datetime.now(tz=timezone.utc).isoformat()
time.sleep(0.001)
t2 = datetime.now(tz=timezone.utc).isoformat()
# t1 = '2026-07-02T01:59:23.573212+00:00'
# t2 = '2026-07-02T01:59:23.574717+00:00'
# t1 == t2 → False
```
两次调用必不等，且 sleep 0ms 也不等（微秒差异）。

3. 既有 e2e 测试反查：`tests/test_pipeline_e2e.py` 中做字节级比较的测试（`test_pipeline_comments_disabled_matches_phase2_outputs`，L78）已经用了 `monkeypatch.setattr("src.pipelines.run_research._now_iso", lambda: "2026")` 固定时间戳（L79）；同样，`test_pipeline_end_to_end_with_comments`（L40）也用了同样的 monkeypatch（L41）。说明项目早已掌握正确注入手法。

**结论**：v2 方案 N2 字节级对照测试的描述**不完整**——未说明需要 monkeypatch 固定 `_now_iso`（或等价地注入固定 `collected_at`）才能令测试成立。但这不是设计级漏洞，因为：(a) 既有 e2e 测试已有标准范式；(b) 实施者对 `_now_iso` monkeypatch 并不陌生；(c) 方案评审规则允许「小瑕疵可在实施期以明确指令落实」。**列为实施期须落实建议（见下）**，不打回。

### Q-B（B1 亲核实验）：handler.name 属性及前缀过滤是否真实可行？

**质疑**：Python stdlib `logging.Handler` 的 `name` 属性是否可赋值可读？前缀过滤能否保证陌生 handler 存活、xhsrecon.* 恰一套？

**实验**（在项目环境 python3 直接执行）：

```
=== 初始状态（陌生 handler）===
  handler name=None
  handler name='pytest_caplog_handler'
=== 第一次 configure_logging 后 ===
  handler name=None
  handler name='pytest_caplog_handler'
  handler name='xhsrecon.console'
  handler name='xhsrecon.file'
  已移除: 'xhsrecon.console'
  已移除: 'xhsrecon.file'
=== 幂等重调后（期望：陌生存活，xhsrecon.* 恰一套）===
  handler name=None
  handler name='pytest_caplog_handler'
  handler name='xhsrecon.console'
  handler name='xhsrecon.file'

xhsrecon.* handler 数量: 2（期望 2，console+file）
陌生 handler 数量: 2（期望 2，stranger_unnamed+stranger_named）
B1 幂等方案验证：正确

StreamHandler 默认 name: None
set_name 后: 'test_name'
name 属性可读写: 是（通过 set_name/get_name property）
```

**结论**：`handler.name` 属性在 Python stdlib logging 中可通过 `set_name()` 赋值、通过 `h.name` 读取；前缀过滤「只移除 name 以 `xhsrecon.` 开头的 handler」逻辑正确——无名（name=None）和他名 handler 不受影响存活；xhsrecon.* 幂等重调后恰一套。B1 方案实现路径成立。

### Q-C（B2 可证伪质疑）：keyword 消息体写法能否被 §6 测试断言机械验证？

**质疑**：v2 B2 取舍说「keyword 作为消息体首个参数，可 grep 性等价」，并在 §6 单测中补了「search 行含 kw=」断言。这个断言能否真的用 caplog 机械验证，还是只是说说？

**验证**：读 v2 §4 接线伪代码：
```python
log.info("search kw=%s p%d: %d notes in %.1fs", kw, page, len(r.notes), dt)
```
caplog 断言写法：
```python
assert any("kw=" in r.message for r in caplog.records if r.levelno == logging.INFO)
```
这个断言是可测试、可机械验证的——caplog.records 包含 LogRecord 对象，`r.getMessage()` 返回格式化后的消息字符串，`kw=` 是字面前缀，字符串包含关系判断确定性强。

**结论**：B2 keyword 消息体写法的可 grep 性可被测试断言验证，取舍论证可落地。B2 关闭有效。

## 差量引入新问题核查

### 落盘移位到 search()/fetch_comments()——fixture_adapter 无 save_path

v2 N1 将落盘从 `_run_crawler` 移到 `search()`/`fetch_comments()` 内。`_run_crawler` 仅在 `MediaCrawlerAdapter` 中存在，`FixtureAdapter` 没有此方法，也无 save_path 概念。

**核查**：v2 方案 §3 目标3 说「`search()`/`fetch_comments()` 在子进程返回后总是落盘」，但 §2 约束盘点中 `fixture_adapter` 标注为「静默」。实施步骤 §5 第3条说「adapter 内部新增模块级 logger」，并未要求 FixtureAdapter 也落盘 mediacrawler.log。结合代码：FixtureAdapter 无 `_run_crawler`，无子进程，落盘逻辑天然只在 MediaCrawlerAdapter 内实现，与 FixtureAdapter 解耦。

**结论**：方案对两个 adapter 的分工隐含清晰（只有 MediaCrawlerAdapter 有子进程/save_path），不算新问题；实施者可自行判断，属正常设计边界。无阻塞。

### mkdir exist_ok 修正

v1 只说 `parents=True 兜底`，N1 评审指出缺 `exist_ok=True`。v2 伪代码已明确：`save_path.mkdir(parents=True, exist_ok=True)`。已关。
