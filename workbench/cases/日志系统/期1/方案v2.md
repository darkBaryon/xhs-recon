# 日志系统 期1 方案 v2

## 给用户的摘要

给 xhs-recon 补一套 **stdlib logging** 的工程化日志：管线每个阶段一行 INFO（条数+耗时）、每次运行落一个 DEBUG 全量日志文件、MediaCrawler 子进程输出完整落盘到采集目录、失败按「成功 INFO / 预期内失败 WARN / 异常 ERROR」统一路由。参考 house-manager 的三层哲学（通用构造 / 业务约定 / 组装点接线），Python 等价物落地。

- Before：全项目零日志，子进程输出用后即丢，manual 集成十几分钟终端沉默，排障无据可查。
- After：控制台能看到管线在干什么；`data/logs/run-*.log` 与 `{save_path}/mediacrawler.log` 可事后复盘；**导出文件逐字节不变**（日志纯旁路）。

拍板清单：**无**（技术底线沿用：不引第三方库、日志不出现评论内容/身份字段；均在需求已定）。

## v1→v2 变更（供差量评审）

回应方案评审 #1（verdict 需修改，2 阻塞 + 4 建议）：

- **B1（阻塞）幂等实现**：§4 改为**只移除自己打标的 handler**——configure_logging 创建的 Handler 一律 `handler.name = "xhsrecon.console" / "xhsrecon.file"`，幂等 = 按 `name.startswith("xhsrecon.")` 移除后重建；**绝不 `root.handlers.clear()`**（会误删 pytest caplog 的 LogCaptureHandler，导致 §6 全部 caplog 断言失效）。§6 增补专项测试：configure_logging 之后 root 上预置的陌生 handler 仍存活。
- **B2（阻塞）keyword 字段取舍**：需求目标 4 的三字段中 `keyword` **不进 RunContextFilter**——理由：Filter 是 run 级单例，keyword 随 search 循环逐次变化，塞进 Filter 需要可变全局态（丑且有测试串扰风险）。取舍：run 级字段（run_id/provider）走 Filter；**keyword 作为消息体首个参数**出现在每条 search/评论相关日志行（`"search kw=%s p%d ..."`），可 grep 性等价。此为对需求目标 4 字面的**显式偏差**，已在本节登记（house-manager 的 ctx 注入对应 request 级字段，xhs-recon 无 request 概念，run 级即对齐）。
- **N1**：`mediacrawler.log` 落盘点从 `_run_crawler` 内移到 **`search()`/`fetch_comments()` 内**（拿到 `rc, out` 之后），并先 `save_path.mkdir(parents=True, exist_ok=True)`——覆盖 rc≠0 且 MediaCrawler 未建目录的情形；`_run_crawler` 保持纯执行职责（单测注入点不变）。
- **N2**：§6 旁路证明改为**新增专项断言**：同一 fixture 输入分别以 `file_enabled: true/false`、`--verbose` 开/关跑两遍，6 个导出文件逐字节相等（现有 e2e 只测评论开关，不能证明日志旁路）。
- **N3**：§2 行号修正 `run_research.py:47-88`。
- **N4**：checklist 层次红线 grep 加注：`import src.pipelines...` 写法可绕过 `from src.pipelines` 的 grep，机械检查为下限，import 面貌由代码评审亲核。

**约束**：

- 依赖铁律不破：`logging_setup` 放 **pipelines 层**（组装点），core/adapters 不 import 它，只用 stdlib `logging.getLogger(__name__)`——core 仍平台无关、零配置。
- 日志纯旁路：**不改任何导出内容与业务逻辑**；旁路性由 §6 新增的字节级对照测试机械证明。
- 不引第三方库；不做轮转/JSON/集中收集（通用层留扩展位：`configure_logging` 单点换 Handler 即可）。

## 3. 目标 / 非目标

**目标**：

1. `logging_setup.configure_logging(logging_cfg, verbose, run_id, provider)`：控制台 Handler（默认 INFO，`--verbose`→DEBUG）+ 文件 Handler（`{dir}/run-{run_id}.log`，DEBUG 全量，`file_enabled` 可关）；统一 formatter 含 `run_id`/`provider`；幂等按 handler 名前缀 `xhsrecon.` 收口（B1）。
2. 管线阶段 INFO：扩词结果、每次 search（keyword 入消息体/页/条数/耗时）、聚合（笔记/账号数）、打分、选典型（条数）、评论（条数或跳过原因）、导出（文件数）。
3. `search()`/`fetch_comments()` 在子进程返回后**总是**落盘 `{save_path}/mediacrawler.log`（成功失败都写，先 mkdir exist_ok）；rc≠0 记 WARN。
4. `log_result` 结果路由：`FetchResult.ok`→INFO；`error`→WARN；未捕获异常自然 ERROR（不吞）。
5. 红线专项测试：日志输出（caplog+日志文件）不含评论 body 与 `nickname/user_id/ip_location/avatar` 键值。
6. config 扩展 `logging: {level, dir, file_enabled}`（缺省 info / data/logs / true）；CLI 加 `--verbose`。

**非目标**：不改导出契约与 5+1 文件内容；不动 search/aggregate/rank/select/评论采集逻辑；不引第三方；不做轮转/JSON；`workbench/` 工具脚本不在本期范围；keyword 不做 Filter 级注入（见 B2 取舍）。

## 4. 接口与数据设计

**`src/pipelines/logging_setup.py`（新增，组装点专属）**：

```python
HANDLER_PREFIX = "xhsrecon."

class RunContextFilter(logging.Filter):
    def __init__(self, run_id: str, provider: str): ...
    def filter(self, record):          # 注入 record.run_id / record.provider
        record.run_id = self.run_id; record.provider = self.provider; return True

FORMAT = "%(asctime)s %(levelname)-7s %(name)s [%(provider)s %(run_id)s] %(message)s"

def configure_logging(cfg: dict, *, verbose: bool, run_id: str, provider: str) -> None:
    root = logging.getLogger()
    # 幂等（B1）：只移除 name 以 HANDLER_PREFIX 开头的旧 handler，再重建；
    # 绝不 root.handlers.clear()——保 pytest caplog 的 LogCaptureHandler 存活。
    # console = StreamHandler(name="xhsrecon.console", level=DEBUG if verbose else cfg level)
    # file    = FileHandler(name="xhsrecon.file", {dir}/run-{run_id}.log, level=DEBUG)，
    #           file_enabled=false 跳过；建目录失败 → 控制台 WARN 一行，不抛（旁路保证）。
    # 两 handler 均挂 RunContextFilter(run_id, provider)；root.setLevel(DEBUG)。

def log_result(logger, fr) -> None:
    # fr.ok → INFO "%s ok: %d notes %d accounts %d comments"（只有条数）
    # fr.error → WARN "%s failed: %s"（error 文本；不打 raw_text，不打评论内容）
```

## 8. 风险与禁止事项 / 回滚口径

- **日志红线**：任何 logger 调用不得传评论 body/身份字段——评审与红线测试双守；`log_result` 是唯一失败记录口，不打 `raw_text`。

- **caplog 兼容（B1）**：幂等只动 `xhsrecon.*` 前缀 handler；实施中若发现任何测试因 handler 管理互相串扰，立即停并按变更协议登记。
- **旁路保证**：日志失败不得影响管线（文件 Handler 建目录失败降级控制台 WARN，不抛）；旁路性由字节级对照测试机械证明。
- **回滚**：全部在 `feat/logging`，弃分支即回期3 态；config `logging` 段缺省行为=有日志但不影响任何产出。
