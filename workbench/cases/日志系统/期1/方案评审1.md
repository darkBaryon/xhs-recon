### Q2：`configure_logging` 幂等声称「测试友好」——实际是否如此？（**阻塞**）

**质疑**：方案 §4 说「幂等：重复调用先清旧 handler（测试友好）」。若实现为 `root.handlers.clear()`，会怎样？

**验证**：

```python
# 查看 pytest LogCaptureHandler 安装机制
from _pytest.logging import catching_logs
# catching_logs.__enter__ 调用:
root_logger.addHandler(self.handler)  # 注入 LogCaptureHandler
# catching_logs.__exit__ 调用:
root_logger.removeHandler(self.handler)
```

`catching_logs` 是 pytest 用于 `caplog` 捕获的 context manager，在测试开始时 `root_logger.addHandler(LogCaptureHandler)`，在测试结束时 `removeHandler`。若 `configure_logging` 在测试体内被调用时执行 `root.handlers.clear()`，**LogCaptureHandler 被删除，后续 `caplog.records` 为空**，方案 §6 所有 caplog 断言（阶段 INFO/WARN 路由、红线 grep）全部失效。

**结论**：「先清旧 handler」若实现为 `clear()` 会破坏 caplog，与方案「测试友好」的自述矛盾。正确实现是只删自己管理的 handler（按类型过滤或保存引用）。方案须补充正确的幂等实现策略。**阻塞**。

### Q3：需求目标4 要求 `keyword` 上下文字段——方案是否落地？（**阻塞**）

**质疑**：需求目标4 明确说「日志行携带 run 级字段（`collected_at/provider/keyword`）」三字段并列。方案 `RunContextFilter(run_id, provider)` 只有两个，keyword 去哪了？

**验证**：读 `workbench/reports/日志系统.md` 目标4：

> 日志行携带 run 级字段（collected_at/provider/keyword），Python 侧用 `LoggerAdapter` 或 `Filter` 注入

方案 §4 `RunContextFilter.__init__(self, run_id: str, provider: str)` 无 keyword 参数；`FORMAT` 也只含 `%(provider)s %(run_id)s`，无 keyword。方案 §4 「每次 search: log.info('search %s p%d...', ...)」中 keyword 只出现在消息体，不是结构化字段。

**结论**：keyword 未注入到 Filter/Formatter，降级到消息体参数，与需求并列表述不符。keyword 在 search 循环内逐次变化，不适合 run 级 Filter 这是事实，但方案未解释这个取舍。需设计方补充：要么解释为何 keyword 做消息参数而非结构化字段（并说明需求可被满足），要么补充 keyword 注入（如 per-search LoggerAdapter）。**阻塞**。

### Q4：`mediacrawler.log` 落盘——rc≠0 时 `save_path` 是否存在？（非阻塞，实施注意项）

**质疑**：方案伪代码 `(save_path / "mediacrawler.log").write_text(out, ...)`，但若 rc≠0 且 MediaCrawler 启动失败，save_path 目录可能根本未创建，`write_text` 会 `FileNotFoundError`。

**验证**：`_run_crawler` 当前签名 `(self, cmd) -> tuple[int, str]`，不接受 save_path 参数；落盘逻辑属于 `search()`/`fetch_comments()` 的调用方，在这两个方法里 save_path 是局部变量。rc≠0 时 MediaCrawler 可能崩溃于进程启动阶段，save_path 目录未创建。

方案 §4 说「需 parents=True 兜底」，但未说 `exist_ok=True`；`Path.mkdir(parents=True)` 在目录已存在时抛 `FileExistsError`，需同时加 `exist_ok=True`。

**结论**：需实施时在落盘前加 `save_path.mkdir(parents=True, exist_ok=True)`。方案伪代码模糊了落盘的具体函数位置，实施者需自行决定。**非阻塞，建议在 §4 伪代码中澄清落盘位置（在 search()/fetch_comments() 而非 _run_crawler 内）**。

## 阻塞项

**B1（§4/§6，幂等实现描述错误）**：方案说「幂等：重复调用先清旧 handler（测试友好）」，若实现为 `root.handlers.clear()` 会删除 pytest 的 `LogCaptureHandler`，导致所有 caplog 断言失效。v2 须明确幂等的正确实现：只清自己管理的 handler（按类型识别或保存引用），保留其他 handler（如 pytest 的 LogCaptureHandler）。

**B2（§3/§4，需求目标4 keyword 字段未落地）**：需求目标4 明确要求「`collected_at/provider/keyword`」三字段注入，方案 `RunContextFilter` 只注入两个（run_id/provider），keyword 降级到消息体参数。v2 须说明取舍理由（keyword 随 search 循环变化，不适合 run 级 Filter），或补充 per-search `LoggerAdapter` 方案将 keyword 作为结构化字段。

## 非阻塞建议

**N1（§4 伪代码）**：明确 `mediacrawler.log` 落盘逻辑放在 `search()`/`fetch_comments()` 内（而非 `_run_crawler`），并补充 `save_path.mkdir(parents=True, exist_ok=True)` 以覆盖 rc≠0 时目录未创建的情况。

**N2（§6）**：「复用既有 e2e 断言」改为明确表述：新增专项测试，同一 fixture 配置下有/无 `configure_logging` 的两次 `run_research` 导出文件字节一致；或说明通过代码审查确认 `export_all` 不经过 logging 路径（等价旁路证明）。

**N3（§2 行号）**：`run_research.py:46-67` 改为 `:47-88`（评论段和导出段均在范围内）。

**N4（checklist）**：补充 `import src.pipelines` 写法的防护提示（实施纪律注释），提醒实施者不使用 `import <module>` 写法绕过 grep 检查。
