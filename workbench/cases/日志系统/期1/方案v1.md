# 日志系统 期1 方案 v1

## 给用户的摘要

给 xhs-recon 补一套 **stdlib logging** 的工程化日志：管线每个阶段一行 INFO（条数+耗时）、每次运行落一个 DEBUG 全量日志文件、MediaCrawler 子进程输出完整落盘到采集目录、失败按「成功 INFO / 预期内失败 WARN / 异常 ERROR」统一路由。参考 house-manager 的三层哲学（通用构造 / 业务约定 / 组装点接线），Python 等价物落地。

- Before：全项目零日志，子进程输出用后即丢，manual 集成十几分钟终端沉默，排障无据可查。
- After：控制台能看到管线在干什么；`data/logs/run-*.log` 与 `{save_path}/mediacrawler.log` 可事后复盘；**导出文件逐字节不变**（日志纯旁路）。

拍板清单：**无**（技术底线沿用：不引第三方库、日志不出现评论内容/身份字段；均在需求已定）。

## 2. 现状与约束盘点

**现状（谁需要说话而说不了）**：

- `src/pipelines/run_research.py:46-67` `run_research`：六阶段（扩词→搜索→聚合→打分→选典型→评论→导出）零过程输出，仅 CLI 末尾 `typer.echo` 产出路径。
- `src/adapters/mediacrawler_adapter.py:115-124` `_run_crawler`：`subprocess.run` 捕获子进程全部输出，成功即丢；失败仅 `error` 留末尾 300 字符（期3 曾 247KB stderr 无据可查）。
- `src/adapters/fixture_adapter.py` / `parsers.py`：静默。
- `checklist` 惯例禁 `src` 内 `print(` —— 正常路径无合法发声通道。
- `scripts/integration_mediacrawler.py`（manual）：全程哑跑，期3 验收时用户误判卡死。

**house-manager 范式 → Python 映射**（需求已调研，`pkg/logger`+`pkg/applog`+main 组装）：

| house-manager | 本方案 |
|---|---|
| `pkg/logger.New/Init`（level 可配、组装点注入） | `src/pipelines/logging_setup.py: configure_logging(cfg, verbose)`，仅 pipeline 入口调用 |
| `applog.ContextHandler`（ctx 注入 request_id） | `RunContextFilter(logging.Filter)` 注入 `run_id`（=collected_at）与 `provider` |
| `applog.Result`（错误→级别路由唯一实现） | `log_result(logger, fetch_result)`：ok→INFO，`error`→WARN（预期内失败，不假装口径） |
| 脱敏唯一收口（MaskPhone） | 日志红线：只记评论**条数**，永不记评论 body/任何身份字段 |
| main.go 组装点 | `run_research()` 开头接线（与 `_build_adapter` 同哲学，core/adapter 只 `getLogger(__name__)` 发日志） |

**约束**：

- 依赖铁律不破：`logging_setup` 放 **pipelines 层**（组装点），core/adapters 不 import 它，只用 stdlib `logging.getLogger(__name__)`——core 仍平台无关、零配置。
- 日志纯旁路：**不改任何导出内容与业务逻辑**；`comments.enabled:false` 与期3 导出逐字节一致。
- 不引第三方库；不做轮转/JSON/集中收集（通用层留扩展位：`configure_logging` 单点换 Handler 即可）。

## 3. 目标 / 非目标

**目标**：

1. `logging_setup.configure_logging(logging_cfg, verbose, run_id, provider)`：控制台 Handler（默认 INFO，`--verbose`→DEBUG）+ 文件 Handler（`{dir}/run-{run_id}.log`，DEBUG 全量，`file_enabled` 可关）；统一 formatter 含 `run_id`/`provider` 字段。
2. 管线阶段 INFO：扩词结果、每次 search（关键词/页/条数/耗时）、聚合（笔记/账号数）、打分、选典型（条数）、评论（条数或跳过原因）、导出（文件数）。
3. `_run_crawler` 子进程输出**总是**落盘 `{save_path}/mediacrawler.log`（成功失败都写）；rc≠0 记 WARN。
4. `log_result` 结果路由：`FetchResult.ok`→INFO；`error`→WARN；未捕获异常自然 ERROR（不吞）。
5. 红线专项测试：日志输出（caplog+日志文件）不含评论 body 与 `nickname/user_id/ip_location/avatar` 键值。
6. config 扩展 `logging: {level, dir, file_enabled}`（缺省 info / data/logs / true）；CLI 加 `--verbose`。

**非目标**：不改导出契约与 5+1 文件内容；不动 search/aggregate/rank/select/评论采集逻辑；不引第三方；不做轮转/JSON；`workbench/` 工具脚本不在本期范围。

- 机械参数落同目录 `checklist.yaml`。

## 8. 风险与禁止事项 / 回滚口径

- **日志红线**：任何 logger 调用不得传评论 body/身份字段——评审与红线测试双守；`log_result` 是唯一失败记录口，不打 `raw_text`。
- **层次红线**：core/adapters 只 `getLogger`，禁止 import pipelines 的 logging_setup（否则依赖倒置）。
- **旁路保证**：日志失败不得影响管线（文件 Handler 建目录失败降级控制台 WARN 一行，不抛）。
- **回滚**：全部在 `feat/logging`，弃分支即回期3 态；config `logging` 段缺省行为=有日志但不影响任何产出。
