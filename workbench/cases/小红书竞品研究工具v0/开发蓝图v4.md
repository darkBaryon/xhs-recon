# 小红书竞品研究工具 v0 开发蓝图 v4

## 给用户的摘要

本蓝图钉三样：模块边界 + 跨期稳定的接口契约 + 技术栈与数据源。研究核心（关键词/聚合/打分/选择/导出）平台无关、可复用；小红书专属的采集**与解析**隔到可替换 adapter 后。

- Before：期1 仅 fixture 管线，「真实采集」数据源未定。
- After：期2 数据源经横向调研**定为 MediaCrawler**——`mediacrawler_adapter` 触发其采集、产 JSONL、复用期1 `parsers`（同 21 字段），失败/无 MediaCrawler 降级 fixture；`core` 仍 0 处 "xhs"。

拍板项（均已确认）：

- **分期方式**：期1 fixture 管线（已交付）、期2 真实 MediaCrawler 采集、期3 评论（可选）。
- **稳定契约范围**：`ResearchAdapter` 端口、领域模型、`Exporter` 输出契约 跨期稳定。
- **技术栈**：Python 3.11+ / uv / pydantic / typer。
- **数据源（v4 新拍板）**：**MediaCrawler**（浏览器爬虫，search 一次返回含正文的全字段）。理由见下节调研结论。
- 评论默认只存 `body / note_id / like_count / collected_at`，不存评论用户身份字段。

## v3→v4 变更（供差量评审）

期2 开工 spike 阶段，发现蓝图 v3 假设的「独立 `xhs` 命令」不成立，经横向工具调研后由用户拍板数据源。逐条：

1. **数据源定为 MediaCrawler**（替换 v3 模糊的「真实 `xhs` CLI」）。**调研结论**：小红书官方 search API **本身不返回正文**——所有轻量逆向工具（`xiaohongshu-cli`/`redbook`/`ReaJason/xhs`/`XHS-Downloader`）的 search 都只给卡片，正文须逐篇详情请求；**唯一 search 一次即含正文的是 MediaCrawler**（浏览器抓取、内部已补详情），代价是 Playwright 浏览器 + 登录。用户在知情下选 MediaCrawler 路线（数据最全、省去逐篇补全）。
2. **§2/§4 adapter 改名与形态**：`xhs_cli_adapter` → `mediacrawler_adapter`（subprocess/runner 触发 MediaCrawler 采集 → 读其 JSONL → 复用 `parsers`）。
3. **§3 契约无改动**：MediaCrawler 输出即期1 已对齐的 21 字段，`Note` 模型字段（body/tags/published_at/url）**全部有真实数据、零缺口**；期1 `parsers` 基本直接复用。
4. **§1 非目标澄清**：明确「不自研爬虫、不做并发/代理池/反风控；仅调用现成 MediaCrawler 做低频小范围采集」，与需求非目标及 MediaCrawler README 自述一致。
5. **§5 风险更新**：引入 Playwright 浏览器 + 登录依赖；真实采集不可自动化测试（测试吃 fixture，真实跑通手动验收）。
6. 撤回 v3 期2 的「`xhs` 命令」假设与「无 xhs 命令降级」措辞，改为「MediaCrawler 不可用/采集失败时降级 fixture」。
7. xiaohongshu-cli 在 spike 中安装并验证（确认其 search 不返正文），结论排除，非数据源。

## 1. 整体目标 / 非目标

**目标**：把需求 7 步实现为一条可配置、可测试、可复用的本地管线；研究核心不依赖小红书。

**非目标**（蓝图层显式不做）：**自研爬虫**、并发/增量采集、代理池/反风控/验证码、持久化数据库、任何互动写操作。采集仅**调用现成 MediaCrawler 做低频小范围**抓取。

## 2. 模块边界与依赖方向

```
configs/sample.yaml ──► pipelines.run_research (typer CLI)    # 编排：读 config，注入 adapter，串各步
                              │  依赖（注入 adapter）
                              ▼
        ┌─────────────────── core ───────────────────┐
        │ keyword_expander  aggregator  account_ranker│  研究核心：平台无关、可复用
        │ note_selector     exporter                  │  只认 models 与 ResearchAdapter 端口
        └──────────────┬──────────────────────────────┘  无解析、无原始 JSONL、无 "xhs" 字样
                       │ 依赖
                       ▼
                 models (pydantic)          adapters (实现 ResearchAdapter 端口)
            Account/Note/Comment/…    fixture_adapter   mediacrawler_adapter
                                      └ parsers: MediaCrawler JSONL → models（xhs 专属，归一化在此，两 adapter 共用）
```

**依赖方向铁律**（不变）：`pipelines → core → models`；`adapters` 实现 `core` 端口、由 `pipelines` 注入；`core` 永不 import adapter、不出现 "xhs"、不碰原始 JSONL。

- `fixture_adapter`（期1，已交付）与 `mediacrawler_adapter`（期2）消费**同一份 MediaCrawler JSONL 形状**，**共用 `parsers`**——fixture 即 MediaCrawler 的离线样本。

## 3. 接口契约草案（跨期稳定，不轻改）

期1 已落地，期2 不改契约。要点（详见期1 方案 §4）：

```python
# models（pydantic）：Account / Note / Comment / AccountRank / TypicalNote / FetchResult
#   Note 字段在 MediaCrawler 源下全部有数据：body←desc、tags←tag_list、published_at←time、url←note_url
# 端口：ResearchAdapter.search(keyword, page, limit, collected_at) -> FetchResult
#       fetch_comments(...) 期3 才实现（基类默认 NotImplementedError）
# FetchResult(notes[], accounts[], raw_path, raw_text, error, command, ...)；.ok = not error
# 导出契约：accounts/notes/account_rank/typical_notes.csv + report_input.md
```

| 期 | 交付 | 验收口径 | 依赖/前置 |

每期独立走标准级 8 步；上一期未验收归档不开下一期实施。

## 5. 风险与未决项

- **MediaCrawler 调用方式**（期2 最大不确定）：输出格式已知（21 字段），但「adapter 怎么触发它跑、传关键词、定位输出 JSONL」需期2 方案现状盘点实测（线索：`data/.mediacrawler_xhs_runner.py` 的 import+config+asyncio 模式、MediaCrawler 的 CDP/cookie 登录模式）。
- **浏览器 + 登录依赖**：真实采集要 Playwright/CDP + 扫码或 cookie 登录；**无法自动化测试**——期2 测试吃 fixture 验证解析/降级，真实采集靠手动验收。
- **合规与自律**：MediaCrawler 声明仅供学习研究；本工具默认低频小范围，禁并发/代理池/反风控；不存评论用户敏感字段。
- 打分公式 v0 简单加权，允许各期调，不算偏差。
