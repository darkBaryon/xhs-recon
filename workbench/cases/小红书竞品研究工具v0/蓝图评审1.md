## 结论

蓝图模块边界与依赖方向铁律设计扎实、忠于源需求，**但接口契约草案有两处字段无真实数据来源、且期1 引用的 fixture 路径在仓库里根本不存在**——这两条会让期1「现在就能跑通验证」的承诺落空，因此 verdict = **需修改**。

阻塞项清单：

1. **B1**：期1 验收口径依赖 `tests/fixtures/search_contents_sample.jsonl`，该文件不存在（连 `tests/` 目录都没有）；唯一真实样本在仓库外 `data/xhs/jsonl/search_contents_2026-06-23.jsonl`。期1「fixture 管线现在就能跑通」的前提不成立。
2. **B2**：`Account.profile_url` 与 `Account.note_count` 在真实搜索样本里**没有任何源字段**可映射，契约被标注为「跨期稳定、各期不轻改」，但其来源未在蓝图说明（只能由 user_id 拼接 / 在结果集内计数，二者语义都不等于字段名所承诺的）。

## 逐章核查

**摘要 / §1 目标·非目标**：忠于源需求。源需求 7 步（扩展关键词 / 调 xhs 只读搜索 / 提取 notes·accounts / 去重打分 / 选典型笔记 / 可选评论 / 导出）逐一落到模块：keyword_expander→1，adapter.search+parsers→2·3，account_ranker→4，note_selector→5，fetch_comments→6，exporter→7。非目标（无爬虫/无反风控/无互动/不存评论用户敏感字段/不做旧兼容）与蓝图 §1 非目标一致。亲核：`grep` 源需求与蓝图非目标逐条对照，无遗漏、无超范围扩张。

**§2 模块边界与依赖方向**：`pipelines→core→models`、adapter 由 pipelines 注入、core 不 import 具体 adapter 且「无 xhs 字样」——方向正确，且「core 无 xhs 字样」确实是可 grep 的机械判据。**但该判据只能抓字面 "xhs" 与显式 import，抓不到结构性耦合**：若 core 的 parser 直接消费 `liked_count="1万"`、`tag_list="a,b,c"`、`time=ms` 这类 MediaCrawler/xhs 专属编码（见验证记录），core 就隐性绑定了 xhs 数据形状，grep 永远是 0 命中却并不平台无关。判据需补一条「core 只接收已规范化的 models 对象，原始字段解析归 adapter/parsers 边界」。

**§3 接口契约**：拿真实样本逐字段核对（见验证记录）——
- `Note`：title←title、body←desc、tags[]←tag_list(逗号串需 split)、url←note_url、like_count←liked_count、collect_count←collected_count、comment_count←comment_count、published_at←time(ms)、source_keywords[]←source_keyword(单串)、raw_path(自造指针)，**均可映射**。
- `Account`：account_id←user_id、nickname←nickname **可映射**；**profile_url、note_count 无源字段**（样本 user 侧只有 user_id/nickname/avatar），是 B2。
- `Comment` 刻意不含用户身份字段——与源需求非目标一致，正确。
- `FetchResult` 失败装载不抛、`.ok=not error`——回应了「降级在哪一层」，设计自洽。

**§4 分期表**：按「能否本地验证」切分合理，期间串行（上一期未验收不开下一期）符合规范。但期1 验收锚在不存在的 fixture 上（B1）。期2 spike-first（探针不进主线）、无命令降级 fixture——忠于源需求验收口径，正确。

**§5 风险与未决项**：已点出 xhs 输出格式未知、fixture 含 PII、打分公式 v0 简单加权。**PII 风险被低估**：风险栏只说「至少评论侧」脱敏，但搜索样本的 note 级 user_id/nickname/avatar 已进入 Account 并会导出（accounts.csv）。源需求非目标仅禁评论用户敏感字段，故存 note 作者身份不算越界——但 avatar 字段既无模型位也无用途，应明确丢弃。

## 可证伪质疑

1. **（§4/期1，阻塞 B1）** 期1 验收要 `fixture_adapter` 吃 `tests/fixtures/`，请指出 `tests/fixtures/search_contents_sample.jsonl` 当前在哪？亲核：`ls tests/fixtures/` 报 No such file or directory，`find . -name search_contents*` 在仓库内 0 命中。未关闭 → 阻塞。
2. **（§3/Account，阻塞 B2）** `Account.profile_url` 从真实搜索 JSONL 的哪个字段来？亲核：全 20 条记录的 key 并集里无 profile/url 类用户字段，仅 user_id/nickname/avatar。若答「由 user_id 拼 xiaohongshu.com/user/profile/{id}」则该拼接逻辑属 xhs 专属、不该藏在被标「平台无关」的 models 里。未关闭 → 阻塞。
3. **（§2 依赖铁律）** core 的 account_ranker 计算 avg_interaction 时如何处理 `liked_count="1万"`？亲核：样本首条 liked_count 即 `"1万"`（其余为纯数字串）。若 core 直接 `int(liked_count)` 会抛 ValueError；若 core 内置「万/千→数值」解析，则 core 绑定了中文平台编码，与「平台无关」矛盾。请指明 万/千 归一化发生在 adapter 还是 core——这决定 §2 铁律是否真守得住。
4. **（§3/source_keywords）** 真实样本 `source_keyword` 是单字符串（本样本全为「留学辅导」），模型用 `source_keywords[]` 复数列表——多关键词命中同一 note 时谁负责合并去重？parsers 还是 ranker？契约未表达聚合责任。可由「同一 note_id 跨关键词出现」证伪，但本单样本 20 条 user_id 全不同、无法触发，盲区需在期1 方案补测试数据。
5. **（§4/期1，账号去重）** 期1 验收含「对账号去重打分」，但唯一真实样本 20 条 user_id 全部唯一（亲核：distinct=20/20），跑通也证明不了去重逻辑。期1 测试需构造重复作者的合成 fixture，否则去重是未被验收覆盖的空转。

## 建议栏（不立案）

- avatar 字段在三个模型里都无位置也无用途，建议期1 方案显式「采集即丢弃」并加 grep 守护，避免无意中导出头像 URL。
- 蓝图 §3 注「形状吸收自已删除的可行性草稿」——草稿已删，profile_url/note_count 等无源字段的来源论证随之丢失，建议把字段→源映射表固化进期1 方案，作为契约的可核验附录。
- `Note.tags` 源是逗号拼接串，`tag_list` 还带话题词残留（desc 里有 `#留学[话题]#` 形式），tag 清洗规则建议在期1 方案钉死。
