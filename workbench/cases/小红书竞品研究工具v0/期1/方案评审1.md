## 可证伪质疑

1. **[阻塞·已证伪]** §1 称 data 三目录忽略于 `.gitignore:18-20`。实测：`grep -n data/ .gitignore` → raw=16/processed=17/exports=18，文件共 18 行，第 20 行不存在。引用错误，订正为 16-18。
2. **[阻塞·已证伪]** §7 散文与 checklist.yaml 是否完全一致？实测 checklist 禁留标记仅 `xhs`(src/core)/`print(`/`proxy_pool`/`captcha`(均 src)，**无任何自动互动符号 grep**，与 §7“无自动互动符号残留”承诺冲突。机器表是唯一事实源，须对齐。
3. **[已关闭]** `class Note` 经 core(aggregator/selector/exporter) 流转，其 `url` 值是否会让 `grep "xhs" in src/core` 假阳性？实测 5 条 note_url 均含 "xiaohongshu" 不含 "xhs"，且 grep 作用于源文件非运行数据——不破坏守护。关闭。
4. **[已关闭，记建议栏]** §2 称 README “述其为数据源”指 MediaCrawler。实测 README 无 “MediaCrawler” 字样，仅言 xhs CLI；但 21 字段形状对齐真实样本属实。归因失实不影响设计，降级为建议。
5. **[已关闭]** 现 fixture 能否触发去重/跨关键词合并测试？实测不能（5 异作者/单关键词）——故方案靠步骤7 合成 fixture，安排成立、闭环。

## 建议栏（不立案）

- §2 删去或改写“（README.md 述其为数据源）”的 MediaCrawler 归因，改为“对齐实测 21 字段 xhs search_contents 形状”，避免错挂出处（质疑4）。
- §7 若选择补 grep 而非删承诺：自动互动符号难以稳定 grep（点赞/评论/私信无统一标识符），建议明确该红线由“core/adapter 期1 无任何网络写代码 + 期2/3 再设守护”兜底，而非期1 强加一条模糊 grep——由设计职责裁定。
- `keyword_expander` 的 `synonyms` 配置项蓝图未具名（蓝图允许扩展策略各期演进，故非偏差），但建议方案点明“synonyms 为 v0 扩展策略、可演进”，以免后续误判为契约。
