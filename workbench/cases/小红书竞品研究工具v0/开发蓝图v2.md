# 小红书竞品研究工具 v0 开发蓝图 v2

## 给用户的摘要

本蓝图**只钉三样**：模块边界 + 跨期稳定的接口契约 + 技术栈；实现细节留给各期方案。目的是让研究核心（关键词/聚合/打分/选择/导出）成为**平台无关、可被后续功能直接调用的库**，把小红书专属的采集**与解析**逻辑隔到一个可替换的 adapter 后面。

- Before：一个脚本里把「调 xhs」「解析」「打分」「导出」混在一起，换个数据源或复用某一步都要重写。
- After：`pipeline` 编排 → 依赖 `core` 各步 → `core` 只认领域模型 `models` 和一个 `ResearchAdapter` 端口；xhs 是端口的一个实现，可换 fixture、可换别的平台。**raw→models 的解析归 adapter，`core` 永不碰原始 JSONL。**

拍板项（均已确认）：

- **分期方式**：按「能否本地验证」切——期1 fixture 管线（现在就能跑通验证），期2 真实 `xhs` CLI（本机暂无命令，需先 spike），期3 评论（可选）。
- **稳定契约范围**：`ResearchAdapter` 端口、领域模型 `Account/Note/Comment`、`Exporter` 输出契约 三者定为跨期稳定；其余（打分公式、关键词扩展策略）允许各期演进。
- **技术栈**：Python 3.11+ / uv / pydantic / typer（见下节）。
- 评论默认只存 `body / note_id / like_count / collected_at`，不存评论用户身份字段。

## v1→v2 变更（供差量评审）

回应「蓝图评审 #1」（verdict 需修改），逐条落地：

1. **解析归属定为 adapter**（关闭评审阻塞「解析层未钉死」+ 发现#1「编码耦合」）：`parsers`（raw JSONL→models）从 `core` 移入 `adapters`；`core` 只消费 models。§2 依赖图与铁律、§3 端口契约据此重写。
2. **pydantic 取代 dataclass**：§3 模型改为 `pydantic BaseModel`（`.model_dump()`）；交互计数 `"1万"→int`、`tag_list` 逗号串→list 等归一化在 adapter 边界由字段 validator 完成。
3. **`Account.profile_url` / `note_count` 来源补全**（关闭评审 B2）：profile_url 由 `user_id` 拼接、note_count 为聚合派生，均在 §3 注明。
4. **`source_keywords[]` 聚合语义**（关闭评审发现#3）：源 JSONL 每条仅单 `source_keyword`，跨关键词命中同 note/account 时由 `core.aggregator` 合并去重，§2/§3 注明。
5. **`fetch_comments` 基类默认**：注明基类默认 `raise NotImplementedError`，期1/期2 adapter 不必实现。
6. 新增 **技术栈** 节。
7. 撤销评审 B1（「fixture 不存在」）：经设计复核，`tests/fixtures/search_contents_sample.jsonl` 确存在于仓库根（5 条），评审 cwd 误判，不构成变更。

## 1. 整体目标 / 非目标

**目标**：把需求 7 步实现为一条可配置、可测试、可复用的本地管线；研究核心不依赖小红书。

**非目标**（蓝图层显式不做）：真实访问稳定性保证、并发/增量采集、持久化数据库、任何互动写操作、任何反风控手段。

## 2. 模块边界与依赖方向

```
configs/sample.yaml ──► pipelines.run_research (typer CLI)    # 编排：读 config，注入 adapter，串各步
                              │  依赖（注入 adapter）
                              ▼
        ┌─────────────────── core ───────────────────┐
        │ keyword_expander  aggregator  account_ranker│  研究核心：平台无关、可复用
        │ note_selector     exporter                  │  只认 models 与 ResearchAdapter 端口
        └──────────────┬──────────────────────────────┘  无解析、无原始 JSONL、无 "xhs" 字样
                       │ 依赖
                       ▼
                 models (pydantic)          adapters (实现 ResearchAdapter 端口)
            Account/Note/Comment/…    xhs_cli_adapter   fixture_adapter
                                      └ parsers: raw JSONL → models（xhs 专属，归一化在此，两 adapter 共用）
```

**依赖方向铁律**：`pipelines → core → models`；`adapters` 实现 `core` 定义的 `ResearchAdapter` 端口，由 `pipelines` 注入。

- **`core` 永不 import 具体 adapter、永不出现 "xhs" 字样、永不触碰原始 JSONL** —— raw→models 的解析（含字段映射与归一化）只发生在 `adapters/parsers`。此时「core 无 xhs 字样」的 grep 判据才**等价于**平台无关（v1 的判据抓不到「core 认死 xhs 字段形状」的隐性耦合，故 v2 把解析整体下沉到 adapter）。
- `fixture_adapter` 与 `xhs_cli_adapter` 消费同一份 xhs JSONL 形状，**共用 `adapters/parsers`**；fixture 即「同形状的离线样本」。

## 3. 接口契约草案（跨期稳定，各期填充不轻改）

```python
# models —— 共享词汇（pydantic BaseModel；.model_dump() 出 dict；validator 在 adapter 边界归一化）
Account(account_id, nickname, profile_url, source_keywords[], note_count, first_seen_at, last_seen_at)
   # profile_url = adapter 由 user_id 拼 https://www.xiaohongshu.com/user/profile/{user_id}
   # note_count  = 派生：core.aggregator 按 account 聚合其相关 note 数
   # source_keywords[] = 跨关键词合并去重（源 JSONL 每条仅单 source_keyword；aggregator 填）
Note(note_id, account_id, title, body, tags[], url, like_count, collect_count,
     comment_count, published_at, collected_at, source_keywords[], raw_path)
   # like_count/collect_count/comment_count: int —— adapter 把 "1万"/"921" 等本地化串归一化为整数
   # tags[] ← 逗号串 tag_list 拆分；body ← desc；url ← note_url；published_at ← time（均在 adapter）
Comment(body, note_id, like_count, collected_at)          # 刻意不含用户身份字段
AccountRank(account_id, nickname, relevant_note_count, keyword_hit_count,
            avg_interaction, account_score, note_ids[])
TypicalNote(account_id, note_id, title, url, note_score, selection_reason, ...)

# adapters 端口 —— 采集+解析的边界；raw→models 在此；失败不抛，装进 FetchResult
class ResearchAdapter(ABC):
    provider_name: str
    def search(keyword, page, limit, collected_at) -> FetchResult         # 返回已解析的 notes/accounts
    def fetch_comments(note_id, limit, collected_at) -> FetchResult        # 基类默认 raise NotImplementedError；期3 才实现

FetchResult(provider, operation, collected_at, keyword, page, note_id,
            notes[], accounts[],                  # adapter 已解析的 pydantic models —— core 只读这两个
            raw_path, raw_text, error, command)    # .ok = not error；raw_path/raw_text = 原始留痕便于复算
```

> 这些形状吸收自已删除的可行性草稿（设计可用、代码弃用），各期方案按此填充实现并补测试。

## 5. 风险与未决项

- **xhs CLI 输出格式未知**（最大不确定）：期2 开工前必须 spike，拿真实输出再定 `xhs_cli_adapter`；期1 的 `parsers` 先对齐现有 MediaCrawler JSONL 字段（已实测 21 字段）。
- **交互计数为本地化字符串**：实测样本 `liked_count="1万"`、`collected_count="921"`，需在 adapter 边界归一化（`万/千` 进位、纯数字串、可能的 `999+`）。期1 方案给解析用例与期望输出。
- **fixture 含 PII**：`tests/fixtures/` 样本与导出需脱敏策略（至少评论侧），期1 方案细化。note 作者 `nickname` 按需求刻意保留（非敏感字段口径）。
- 打分公式（账号/笔记相关性）v0 用简单加权，不追求精度；允许各期调，不算偏差。
