## 可证伪质疑

1. **【阻塞】`_build_command` 为何前缀 `uv run`？方案 §4/§2 全程只说 `python main.py`，此前缀未登记。**
   证据：live 跑 `' '.join(cmd)` = `uv run python main.py --platform xhs …`；亲读方案 v2 §2 line 39（spike 入口 `main.py:100-110`）、§4 line 72-73 伪码 `python main.py …`——均无 `uv run`。`subprocess.run(cmd, cwd=mediacrawler_dir)` 以 MediaCrawler 目录为 cwd，命令是否需 `uv run` 取决于 MediaCrawler 是否以 uv venv 管理依赖，方案未交代、实施未登记。**可证伪点**：若 `../MediaCrawler` 无 uv 工程（无其 `uv.lock`/不以 uv 跑），则 `uv run` 会改解释环境甚至失败 → 真实采集起不来；若确以 uv 管理则前缀必要但仍须登记。集成验收 manual 未跑（`⏸ 待用户登录`），故此前缀**正确性未经任何运行验证**。判 `需修改`：要么登记偏差由用户裁决，要么回退为方案的 `python main.py`。

2. **【非阻塞】未知 provider（如配置笔误 `mediacrwler`）静默落 FixtureAdapter，原 `raise ValueError` 被删——会不会让"以为在真采、实际跑 fixture"无声发生？**
   证据：亲读 `run_research.py:29-43`，`provider` 非 `mediacrawler` 一律 fall-through `return FixtureAdapter`。原 `main` 版对非 `fixture` `raise ValueError`。新版与方案 §4 伪码一致（伪码末行即静默 FixtureAdapter），故**非偏差**；但语义上 typo 的 provider 会静默降级、与方案 §8"采集失败不假装"诚实口径精神略有张力。非阻塞（符合方案字面），记建议栏：可在 fall-through 前对未知非空 provider `log` 一行警示。

3. **【非阻塞·已闭环】异常/超时路径无单测覆盖。**
   证据：方案 §5 步骤3、§6 明示"subprocess 非 0/**超时**/无输出 → error FetchResult；单测覆盖"。亲核 `test_mediacrawler_adapter.py`：仅覆盖非 0（`test_search_nonzero_exit_is_error`）、空输出（`test_search_empty_output_is_error`）、成功；**`except (OSError, subprocess.SubprocessError)` 分支（含 TimeoutExpired）无单测**。我独立 live 注入 `OSError`/`TimeoutExpired` 验证该分支确返回 `ok=False`、`error="run failed: …"`，**行为正确**，故非阻塞；但方案点名的"超时"覆盖缺一条测试。记建议栏：补一条 monkeypatch 抛 `TimeoutExpired` 的断言测试即闭合。
