## 可证伪质疑

1. **【非阻塞】「每次 search 用唯一 save_path 隔离本次产出」做不到逐 search 隔离，只做到逐 RUN 隔离——口径轻微夸大。**
   证据：**亲核 `src/pipelines/run_research.py:37,46-48`** —— `collected_at = _now_iso()` 在 `run_research` 里**每次 run 只算一次**（line 37），随后 `for kw … for page … adapter.search(kw, page, limit, collected_at)`（line 46-48）把**同一个 collected_at** 传给本 run 内全部 kw×page 次 search。v2 §4（line 70-71）的 `save_path = <out_dir>/<collected_at 安全化>` 因此**在一个 run 内对所有 search 都相同** → 多关键词/多页时，第 N 次 search 的 subprocess 仍追加进同一个 `search_contents_<日期>.jsonl`，其 glob 会读到本 run 前 N-1 次 search 的行。即「隔离本次产出」对**单次 search** 不成立，只对**跨 run** 成立。
   为何非阻塞：(a)**亲核 `src/adapters/parsers.py:40-41,85-86`** —— `_row_keyword` 优先取行内 `source_keyword`、仅缺失时回退入参 keyword，真实 MediaCrawler 行自带 `source_keyword`（评审#1 已实测同形），故跨 search 误读不会错配关键词归属；(b)**亲核 `src/core/aggregator.py:29-31`** —— notes 按 `note_id` 去重，重复读到的同一行被 dedup。两者叠加 → **最终 5 文件导出不受影响**。但单次 `FetchResult.notes` 会是累积集而非本 search 净产出，与 v2 文字「隔离本次产出」名实有微小落差。
   建议（不阻塞）：把唯一性粒度对齐到 search（save_path 拼入 keyword/page，或 collected_at 取到秒/纳秒级且每 search 重算），或把 v2 文字改为「逐 run 隔离同日追加」以名实相符。任一即可，记入建议栏。

2. **【非阻塞·待实施验证】并发=1 下同 run 多 subprocess 写同一 jsonl 无锁竞争？**
   证据：`async_file_writer.py:58` 的 `asyncio.Lock` 只在**单进程内**生效；v2 每 search 起独立 subprocess。但因 `--max_concurrency_num 1` 且 pipeline 串行 `for` 循环（run_research.py:46-48 顺序 append），同一时刻只有一个 MediaCrawler 进程在写，无并发写同一文件。offline 不可证、manual 验收覆盖，登记备查、不阻塞。

## 建议栏（不立案）

- **save_path 唯一性粒度**（同质疑1）：当前 `<out_dir>/<collected_at>/` 只逐 run 唯一。若期望逐 search 净产出，建议实施时 save_path 拼入 `kw/page`；否则把方案文字降级为「逐 run 隔离」。最终导出虽不受影响，但口径宜对齐，避免实施按「逐 search 隔离」字面理解后困惑。属实现细节，由设计决定是否收口。
- 评审#1 建议栏两条（首次 `uv run playwright install` + 登录态前置；同日追加跨跑累积）仍适用；v2 的唯一 save_path 已在「跨 run」维度处理后者，集成脚本 README 可顺带提示首次环境前置。
