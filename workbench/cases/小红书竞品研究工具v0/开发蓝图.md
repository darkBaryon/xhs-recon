# 小红书竞品研究工具 v0 开发蓝图 v1

## 给用户的摘要

本蓝图**只钉两样**：模块边界 + 跨期稳定的接口契约；实现细节留给各期方案。目的是让研究核心（关键词/解析/打分/选择/导出）成为**平台无关、可被后续功能直接调用的库**，把小红书专属的采集逻辑隔到一个可替换的 adapter 后面。

- Before：一个脚本里把「调 xhs」「解析」「打分」「导出」混在一起，换个数据源或复用某一步都要重写。
- After：`pipeline` 编排 → 依赖 `core` 各步 → `core` 只认领域模型 `models` 和一个 `ResearchAdapter` 端口；xhs 是端口的一个实现，可换 fixture、可换别的平台。

拍板项：

- **分期方式**：按「能否本地验证」切——期1 fixture 管线（现在就能跑通验证），期2 真实 `xhs` CLI（本机暂无命令，需先 spike），期3 评论（可选）。是否认可？
- **稳定契约范围**：`ResearchAdapter` 端口、领域模型 `Account/Note/Comment`、`Exporter` 输出契约 三者定为跨期稳定；其余（打分公式、关键词扩展策略）允许各期演进。
- 评论默认只存 `body / note_id / like_count / collected_at`，不存评论用户身份字段。

## 1. 整体目标 / 非目标

**目标**：把需求 7 步实现为一条可配置、可测试、可复用的本地管线；研究核心不依赖小红书。

**非目标**（蓝图层显式不做）：真实访问稳定性保证、并发/增量采集、持久化数据库、任何互动写操作、任何反风控手段。

## 2. 模块边界与依赖方向

```
configs/sample.yaml ──► pipelines.run_research        # 编排：读 config，串起各步
                              │  依赖（注入 adapter）
                              ▼
        ┌─────────────────── core ───────────────────┐
        │ keyword_expander  parsers  account_ranker   │  研究核心：平台无关、可复用
        │ note_selector     exporter                  │  只认 models 与 ResearchAdapter 端口
        └──────────────┬──────────────────────────────┘
                       │ 依赖
                       ▼
                     models            adapters (实现 ResearchAdapter 端口)
            Account/Note/Comment/…      xhs_cli_adapter   fixture_adapter
```

**依赖方向铁律**：`pipelines → core → models`；`adapters` 实现 `core` 定义的 `ResearchAdapter` 端口，由 `pipelines` 注入。**`core` 永不 import 具体 adapter，也永不出现 "xhs" 字样**——这是复用性的机械判据（预飞可 grep 守护）。

## 3. 接口契约草案（跨期稳定，各期填充不轻改）

```python
# models —— 共享词汇（dataclass，附 to_dict）
Account(account_id, nickname, profile_url, source_keywords[], note_count, first_seen_at, last_seen_at)
Note(note_id, account_id, title, body, tags[], url, like_count, collect_count,
     comment_count, published_at, collected_at, source_keywords[], raw_path)
Comment(body, note_id, like_count, collected_at)          # 刻意不含用户身份字段
AccountRank(account_id, nickname, relevant_note_count, keyword_hit_count,
            avg_interaction, account_score, note_ids[])
TypicalNote(account_id, note_id, title, url, note_score, selection_reason, ...)

# adapters 端口 —— 采集与解析的边界；失败不抛、装进 FetchResult
class ResearchAdapter(ABC):
    provider_name: str
    def search(keyword, page, limit, collected_at) -> FetchResult
    def fetch_comments(note_id, limit, collected_at) -> FetchResult   # 期3 才必需

FetchResult(provider, operation, collected_at, keyword, page, note_id,
            raw, raw_text, error, command)   # .ok = not error；失败路径显式化

# exporter 输出契约 —— 文件名与列稳定，供 LLM 端消费
exports: accounts.csv  notes.csv  account_rank.csv  typical_notes.csv  report_input.md
```

> 这些形状吸收自已删除的可行性草稿（设计可用、代码弃用），各期方案按此填充实现并补测试。

## 4. 分期表

| 期 | 交付 | 验收口径 | 依赖/前置 |
|---|---|---|---|
| **期1** fixture 管线 | models + keyword_expander + parsers + account_ranker + note_selector + exporter + pipeline + 测试；`fixture_adapter` 吃 `tests/fixtures/` | `run_research --config sample.yaml` 端到端产出全部导出文件；测试全绿；core 无 "xhs" 字样 | 现有 JSONL 样本 |
| **期2** 真实 xhs CLI | `xhs_cli_adapter`（shell 调 xhs、存 raw）+ 无命令自动降级 fixture | 有 xhs 时真实跑通；无 xhs 时降级不中断；raw 落 `data/raw/` | 先 spike 摸清 xhs 输出格式（探针，不进主线） |
| **期3**（可选）评论 | 典型笔记评论读取 + 导出，仅 `body/note_id/like_count/collected_at` | 评论并入 report_input；预飞 grep 0 处用户身份字段 | 期2 |

期与期之间：上一期未验收归档不开下一期实施。每期独立走标准级 8 步。

## 5. 风险与未决项

- **xhs CLI 输出格式未知**（最大不确定）：期2 开工前必须 spike，拿真实输出再定 `xhs_cli_adapter` 解析；期1 的 `parsers` 先对齐现有 MediaCrawler JSONL 字段。
- **fixture 含 PII**：`tests/fixtures/` 样本与导出需脱敏策略（至少评论侧），期1 方案细化。
- 打分公式（账号/笔记相关性）v0 用简单加权，不追求精度；允许各期调，不算偏差。
