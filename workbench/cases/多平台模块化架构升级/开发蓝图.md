# 多平台模块化架构升级 开发蓝图 v1

## 给用户的摘要

把当前“名义分层、实际仍围绕 XHS 生长”的项目，升级为**多平台模块化单体**：领域与用例只认统一内容模型和细粒度能力端口，小红书/未来抖音各自在独立平台包里完成采集与翻译，MySQL、CSV、Web 属于基础设施。

- Before：新增平台要修改全局配置、`runtime` 分支、主编排、模型、存储和导出；部分能力靠 `hasattr()` 猜。
- After：新增平台主要新增 `platforms/<name>/`、注册插件并通过契约测试；不修改领域核心，缺失能力显式报错或按用例规则跳过。
- 采用渐进式换芯，四期迁移；每期旧 CLI/离线流程可运行、可独立回滚，不做一次性重写。
- 本蓝图只搭底座并迁移 XHS；抖音真实接入后续单独立案。

拍板清单（均给出推荐默认）：

1. **迁移方式**：推荐渐进迁移并保留旧入口兼容壳；一次性重写更快看到新目录，但回归与数据风险明显更高。
2. **多平台导出**：推荐单平台 XHS 继续保留旧文件契约，多平台运行按 `<run>/<platform>/` 分区并增加根 `manifest.json`；直接给现有 CSV 加 `platform` 列更简单，但会破坏现有消费者。
3. **数据库切换**：推荐新建平台化 v2 表、迁移 XHS 数据后切读写，旧表保留到 Gate 2 后再另案清理；原表原地改主键更省空间，但回滚风险更高。

## 1. 整体目标 / 非目标

### 1.1 整体目标

1. 采用“Clean Architecture 依赖规则 + 按用例组织 + 平台插件化”的模块化单体，形成可机械检查的边界。
2. 建立平台限定身份：内容、创作者、评论均以 `platform + external_id` 唯一标识，杜绝跨平台 ID 冲突。
3. 用细粒度端口表达搜索、创作者列表、内容详情、评论等能力，消除宽泛基类、`hasattr()` 和隐式 `NotImplementedError` 协议。
4. 平台插件显式注册配置 schema、能力与工厂；运行时不再用平台名条件分支组装。
5. 把搜索、追踪、研究报告重构为应用用例；CLI 只负责参数解析和调用，不再访问用例私有函数。
6. 将 XHS 的 MediaCrawler 命令、进程、原始文件读取、解析翻译和媒体归档拆成可独立测试的组件。
7. 将 MySQL、CSV/Markdown 导出、本地 Web feed 迁到平台化模型，同时给既有 XHS 用户路径提供明确兼容层。
8. 以第二平台 contract fixture 证明扩展点成立；该假平台只在测试中运行，不包含抖音真实访问实现。

### 1.2 非目标

- 不实现抖音真实搜索、登录、解析、下载或反爬逻辑。
- 不改账号排名、专业度、时间衰减、典型内容选择等业务公式。
- 不替换 MediaCrawler、MySQL、Typer、Pydantic 或静态 Web 技术栈。
- 不引入微服务、消息队列、依赖注入框架、动态安装第三方插件或远程任务系统。
- 不做代理池、验证码破解、自动互动或提高采集并发。
- 不为了“以后可能用”预留发布订阅、通用工作流 DSL、跨进程插件协议。
- 不在同一期同时保留两套业务算法；兼容层只做模型/入口/输出投影，不复制核心规则。

## 2. 模块边界与依赖方向

### 2.1 现状事实与问题边界

- `src/pipelines/runtime.py:49-81` 用 provider 条件分支直接构造 MediaCrawler/fixture，新增平台必改组装根。
- `src/pipelines/config.py:60-113` 把 MediaCrawler 专属配置与通用研究配置置于同一模型，平台配置不能自治。
- `src/core/ports.py:13-26` 只声明三个方法；`src/pipelines/run_research.py` 另通过 `search_many`、`list_creator_notes`、`fetch_note_details` 的运行时探测调用未入契约能力。
- `src/pipelines/run_research.py:20` 直接导入 `src.adapters.parsers.normalize_creator_ref`，平台引用解析泄漏进应用编排。
- `src/adapters/mediacrawler_adapter.py:65-885` 同时承担命令构建、进程生命周期、日志解析、JSONL 读回、领域翻译、媒体复制和多种采集流程。
- `src/models.py:6-121` 的实体只以裸 `note_id/account_id` 标识，没有平台维度；`src/adapters/mysql_store.py` 的表主键同样按裸外部 ID 建立。
- `web/feed.py:128-134` 直接查询具体 MySQL 表，展示层与存储 schema 紧耦合。

这些事实说明：现状保留“依赖倒置”的思想价值，但不保留现有目录与大接口作为目标结构。

### 2.3 强制依赖方向

```text
entrypoints ─┬─► application ─► domain
             ├─► platforms   ─► application ports + domain
             └─► infrastructure ─► application ports + domain

web ─► infrastructure 的只读查询接口/稳定 read model
domain ─X─► application/platforms/infrastructure/entrypoints
application ─X─► platforms/infrastructure/entrypoints
platforms/<A> ─X─► platforms/<B>
```

通过 AST/import 架构测试守护上述规则，不新增重量级依赖。`domain` 使用标准库 dataclass/enum；Pydantic 只用于入口配置与边界校验，避免领域模型绑死序列化框架。

### 2.4 模块所有权

- `domain`：统一语言、身份和值规则；不知道 XHS、抖音、MediaCrawler、CSV、MySQL。
- `application`：定义“做什么”和失败语义；不知道外部工具怎样采集、数据怎样落盘。
- `platforms/<name>`：拥有该平台引用、配置、原始字段和到统一领域的翻译；平台差异不得进入 domain 条件分支。
- `infrastructure`：拥有数据库 schema、文件格式、日志和媒体存放；不得包含平台业务解析。
- `entrypoints`：唯一组装根；可依赖所有外层模块，但只做解析、选择、注入和调用。

## 3. 接口契约草案

### 3.1 平台限定身份与领域模型

```python
class Platform(StrEnum):
    XHS = "xhs"
    # DOUYIN 在真实插件立案时加入；测试第二平台使用独立 TestPlatform 值/夹具

@dataclass(frozen=True, slots=True)
class EntityId:
    platform: Platform
    external_id: str

@dataclass(frozen=True, slots=True)
class Content:
    id: EntityId
    creator_id: EntityId
    title: str
    body: str
    url: str
    published_at: datetime | None
    collected_at: datetime
    metrics: Engagement
    media: tuple[MediaRef, ...]
    source_queries: tuple[str, ...] = ()
```

约束：

- `Content.id.platform == Content.creator_id.platform`，构造时校验。
- 领域时间统一为带时区 `datetime`；未知发布时间为 `None`，不再用空字符串表达。
- 平台专属字段不塞进通用 `dict metadata` 逃逸口；只有被两个业务用例消费的字段才进入 domain，其余保留在 raw artifact 或平台 read model。
- 旧 `Note/Account/...` 在兼容层与新模型双向投影；新业务规则只消费新模型，避免长期双核心。

### 3.2 细粒度采集能力

```python
class SearchProvider(Protocol):
    def search(self, request: SearchRequest) -> CollectionResult: ...

class CreatorFeedProvider(Protocol):
    def list_creator_content(self, request: CreatorFeedRequest) -> CollectionResult: ...

class ContentDetailProvider(Protocol):
    def fetch_details(self, request: DetailRequest) -> CollectionResult: ...

class CommentProvider(Protocol):
    def fetch_comments(self, request: CommentRequest) -> CollectionResult: ...

@dataclass(frozen=True)
class PlatformServices:
    platform: Platform
    search: SearchProvider | None = None
    creator_feed: CreatorFeedProvider | None = None
    content_detail: ContentDetailProvider | None = None
    comments: CommentProvider | None = None
```

- 用例在开始时验证所需能力；缺少必需能力抛 `CapabilityUnavailable(platform, capability)`，可选能力由用例显式定义“跳过并记录”，不得靠 `hasattr()`。
- `CollectionResult` 包含领域实体、raw artifact 引用和结构化 `CollectionFailure(code, subject, message, retryable)`；部分成功可表达，配置错误/插件缺失必须 fail fast。
- live collector 不再因路径不存在静默切 fixture；fixture 是显式 collector 选择。

### 3.3 插件注册

```python
class PlatformPlugin(Protocol):
    name: str
    config_model: type[BaseModel]
    def build(self, config: BaseModel, runtime: RuntimeDeps) -> PlatformServices: ...

registry.register(XhsPlugin())
services = registry.build(platform_name, platform_config, runtime_deps)
```

- 注册发生在 `entrypoints.container` 的显式 bootstrap 列表；registry 本身不 import XHS。
- 重名注册、未知插件、配置 schema 不匹配均启动即报错。
- 本需求不做 setuptools entry points 或运行时扫描；本地显式注册更可读、可测试。

### 3.4 配置契约与旧配置兼容

目标配置版本：

```yaml
schema_version: 2
platforms:
  xhs:
    enabled: true
    collector: mediacrawler       # 或 fixture；无静默降级
    mediacrawler:
      root: ../MediaCrawler
      login_type: qrcode
      # ...仅 XHS/MediaCrawler 参数

jobs:
  search:
    platforms: [xhs]
    pages: 1
    limit: 20
    window_days: 30
  track:
    targets: [{platform: xhs, external_id: "..."}]

storage: {enabled: true, database: xhs_recon}
output: {root: data/exports}
```

- `schema_version` 必填；v1 旧配置由 `LegacyConfigAdapter` 一次性翻译为 v2，并发弃用警告。
- `configs/sample.yaml`、现有主题配置和 `run.sh` 在本需求内仍可运行。
- 新文档只教授 v2；兼容适配器在期4评估保留，删除需另行需求和用户 Gate。

### 3.5 用例接口

```python
class SearchResearch:
    def execute(self, command: SearchCommand) -> ResearchRun: ...

class TrackCreators:
    def execute(self, command: TrackCommand) -> ResearchRun: ...

class RunResearch:
    def execute(self, command: ResearchCommand) -> ResearchRun: ...
```

- `ResearchRun` 是内存结果与运行元数据，不返回文件路径 dict；写库/导出由注入端口完成并返回 `RunReceipt`。
- `research` 组合 search/track，不导入或调用其私有函数。
- CLI 命令名 `search/track/research`、`run.sh` 入口保持；旧模块路径作为薄 shim 转调新入口。

### 3.6 存储与导出兼容

数据库推荐采用新 v2 表：`platform_creators`、`platform_contents`、`platform_comments`、`platform_creator_profiles`，唯一键均包含 `platform`。迁移脚本：

1. 建 v2 表；
2. 将现有表数据以 `platform='xhs'` 幂等复制；
3. 对账行数、孤儿引用与抽样内容；
4. 切换新 repository 与 Web read model；
5. 旧表只读保留，不在本需求删除。

导出：

- 单平台 XHS：根目录继续生成现有 CSV/Markdown 列与文件名，作为 compatibility projection。
- 多平台：根 `manifest.json` 记录 schema/run/platforms/errors；每个平台写 `<platform>/accounts.csv`、`contents.csv` 等，不混用裸 ID。
- Web feed 从稳定 `FeedQuery`/read model 取数；payload 增加 `platform`，XHS 视觉主题不是领域约束。

## 5. 风险与未决项

### 5.1 主要风险

- **模型迁移造成双核心**：迁移期只允许“旧模型 → 新模型”边界适配；新用例和新规则不得反向依赖旧模型。每期 checklist 检查新增旧模型引用。
- **数据库迁移损伤本地历史数据**：采用新表 + 幂等复制 + 对账 + 旧表保留；任何删除旧表操作不在本需求范围。
- **兼容投影掩盖平台字段**：旧 CSV 只允许在单 XHS 运行生成；多平台必须走分区 v2 输出，禁止把跨平台裸 ID 混进旧文件。
- **插件层变成新的万能层**：平台包只做采集与翻译；排名、时间窗、watchlist 合成留在 domain/application，架构测试守护。
- **MediaCrawler 拆分引入行为漂移**：期2 先用 characterization tests 锁定命令、超时、读回、错误与媒体路径，再逐组件搬迁。
- **范围过大**：四期每期单独 Gate；若任一期超过标准级规模，先继续拆期，不扩大当期范围。

### 5.2 Gate 1 前待用户确认

若用户只回复“按推荐方案”，即视为接受摘要中的三个推荐默认：渐进迁移、分区多平台导出、新建 v2 表迁移。任何一项改选都需要在蓝图原文登记 Gate 决策并相应调整分期。

### 5.3 禁止事项

- 禁止在 domain/application 出现 `xhs`、`douyin`、MediaCrawler、pymysql、Typer 依赖。
- 禁止用 `hasattr()`、捕获 `AttributeError` 或默认 `NotImplementedError` 代替能力契约。
- 禁止 live provider 配置错误时静默切到 fixture。
- 禁止代理池、验证码破解、自动点赞/评论/私信、并发放大。
- 禁止实施期顺手加入抖音真实代码或未被用例消费的泛化接口。

## 蓝图验证备忘

各期实施后回填：

- 平台限定身份是否覆盖存储、导出、媒体与 Web，不再存在裸 ID 冲突点；
- 细粒度能力是否足以表达 XHS 实际路径，是否出现新的运行时探测；
- 第二平台 contract fixture 是否确实只新增平台包、注册和配置；
- 兼容 shim 是否保持单向、是否有外部调用仍要求延长保留；
- 是否需要出蓝图 v2（改变目标、跨期契约或分期时必须重审）。
