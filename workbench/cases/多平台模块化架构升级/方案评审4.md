# 多平台模块化架构升级 方案评审 #4

**评审对象**：`开发蓝图v3.md` → `开发蓝图v4.md` 差量

## 结论

**verdict：需修改。** v3 评审的删旧/回滚矛盾和用户能力/Web 断链已关闭；原“期 2 过大”已大幅改善，但原 B 批次仍整体落在第 3 期，尚非单一风险域。recovery bundle 已有格式、幂等导入和图片 sha256 验收，但 `since_watermark` 缺少数据库级变更序列支撑，且未规定导入前全量摘要校验/原子失败。剩余 2 个阻塞项关闭前不得进入 Gate 1。

## 逐章差量核查

### 给用户的摘要

摘要已将新系统顺序改为“离线核心→真实 XHS→新数据/消费面→切换→清理”，旧系统保留到切换验收后，与用户的重写决策一致。

- **Q0**：摘要声称“每期只有一个风险域”能否被第 3 期的任一子系统独立失败所证伪？MySQL 迁移可绿而 Web 失败，或 recovery 绿而媒体索引失败，这些是不同回滚单元，见阻塞 1。

### §1 整体目标 / 非目标

目标已纳入 backfill、bundle、静态 Web 入口、recovery 工具和延后删旧；非目标明确不做 new→legacy 回填，但保证新数据以后可回修复后新系统。

- **Q1**：保留 `run.sh` 命令名是否违反用户“不兼容代码/配置/导出”的决策？§3.4 明示这是新契约下的使用体验，旧 Python API/YAML/输出/payload 仍不兼容，**不矛盾**。

### §2 模块边界与依赖方向

新包零旧 import 禁线未放宽；旧数据迁移和 recovery 由新 infrastructure 以 SQL/新契约处理，无需运行时 import 旧 store。

- **Q2**：第 5 期删除旧模块后，新系统的迁移/恢复工具是否会因 import 旧 DTO/store 失效？§2.3 禁线覆盖整个 `src/recon`，且§2.1 规定迁移为新 schema/media 索引的独立处理，**可由架构测试机械证伪，无方案矛盾**。

### §3 接口、数据与用户能力契约

命令处置表已覆盖当前 `run.sh` 的 demo/search/track/track-all/real/browser/backfill/web/bundle，Web 也从用户命令闭环到新 read model 和静态站产物。recovery 的恢复目标被限定为修复后新库，诚实不承诺旧 UI 可见。

- **Q3a**：全部 `run.sh` 用户能力是否都有新处置？亲核现有 `run.sh` case 分支后，九个用户命令全部命中 §3.4，**质疑已关闭**。
- **Q3b**：`export-recovery --since cutover_watermark` 能否保证选中切换后对“既有复合键”的更新？v4 只定义了迁移元数据水位，未要求每表有单调变更序列/`updated_at`/run ledger，也未定义边界包含性，因此新增记录或许可选，旧键更新不可保证，见阻塞 2。
- **Q3c**：损坏的 JSONL 或图片是否会在被导入一部分后才被发现？蓝图只要求 manifest/schema 不匹配 fail fast，没有要求导入前校验每个 JSONL/媒体摘要与数量，也没有原子事务/临时目录切换语义，见阻塞 2。

### §4 分期表

第 2 期现在只承担 live XHS，第 4 期只承担生产切换/观察，第 5 期只承担切换 Gate 2 后清理，三者均有独立 Gate 与回滚。第 1 期作为离线核心闭环也可在不动旧系统的前提下整体删除回退。

- **Q4a**：第 4 期 Gate 2 前旧入口与业务实现是否仍在？交付列明示“旧代码保留”，第 5 期又被上一期 Gate 2 硬阻断，**原删除/回滚矛盾已关闭**。
- **Q4b**：第 3 期是否一个单一回滚单元？它同时交付持久化、历史迁移、媒体索引、两种消费面和 recovery；回滚列只写“删候选库/新站点”，无法对应 output/bundle/recovery 代码，**非单一风险域**。

### §5 风险、禁止事项与决策记录

v4 已正确登记用户没有要求固定两期，拆为五期不改变旁路重写、无兼容壳、保留数据/图片的目标。清理期的独立 commit 也保留了代码级反悔能力。

- **Q5**：五期分段是否重新引入对旧 API/YAML/导出的兼容层？没有；旧系统只是切换前的独立生产基线，新包禁止 import 旧包，第 5 期删除旧业务实现，**忠于用户 Gate**。

### 蓝图验证备忘

备忘已加入候选迁移/recovery、全命令、清理顺序和数据保留重审触发点，但还不能机械捕获 watermark 漏更新或第 3 期内多风险域。

- **Q6**：一个切换后只变更互动数的已有 Content 未进 recovery bundle 时，蓝图现有验证项会失败吗？不一定；往返测试只能验它导出的集合，不能证明集合本身完整。

## 阻塞项

1. **第 3 期仍非单一风险域**：同期包含 MySQL schema/repository、legacy 迁移/对账、媒体索引、新 output/bundle、FeedQuery/Web 和 recovery export/import。这些可独立失败、独立验收，回滚列也只覆盖“候选库/新站点”，没有覆盖 output/bundle/recovery 代码。需至少拆分“数据底座与迁移/recovery”和“消费面 Web/bundle”，为各自定义 Gate 与回滚。
2. **recovery 增量选取与完整性失败语义不足**：`--since cutover_watermark` 没有对应到新表的单调 `change_seq`/run ledger/等价机制，无法证明已有键更新不漏；manifest 虽记 sha256，却没有要求导入前验证全部 JSONL/图片的数量与摘要，也没有 DB 事务+媒体临时目录的原子或可重试失败语义。需补充可证伪的变更水位、preflight 全量校验、部分失败处理和已有键更新+新媒体的边界测试。
