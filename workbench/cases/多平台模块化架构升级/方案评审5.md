# 多平台模块化架构升级 方案评审 #5

**评审对象**：`开发蓝图v4.md` → `开发蓝图v5.md` 差量

## 结论

**verdict：通过。** v4 评审的两个阻塞项已关闭：数据底座、迁移/recovery 与消费面已拆为独立 Gate；recovery 已用同事务 change-log、`(from_seq,to_seq]`、全量 preflight、内容寻址媒体、单 DB 事务与故障注入形成可证伪闭环。七期均有独立验收和回滚，用户命令、Web 链路、生产切换与删旧顺序未退化，未发现新阻塞。

## 逐章差量核查

### 给用户的摘要

摘要已精确对应七个小 Gate，并把“每次写入有变更序列”和“恢复无半导入”作为用户可见承诺。

- **Q0**：摘要的七个 Gate 是否可在不动生产旧系统的情况下逐期验收？第 1～5 期都明确使用新包/隔离目录/候选库，第 6 期才切生产，**可证伪且无断链**。

### §1 整体目标 / 非目标

目标新增事务变更日志、内容寻址图片和可重放 recovery；非目标仍禁止旧 API/YAML/输出/写接口兼容、双写旧表和 new→legacy 回填。

- **Q1**：为保数据新增 change-log 和 recovery 是否变相恢复了旧 store 兼容？不是；它们只属于新 schema，且 recovery 只能导入修复后新系统，**仍忠于用户“重写、不兼容代码、保数据/图片”的 Gate**。

### §2 模块边界与依赖方向

迁移/recovery 从 MySQL persistence 中拆到外层 `infrastructure/migration`，直接读旧 SQL schema，不 import 旧 store/DTO；新 domain/application 禁线未放宽。

- **Q2**：第 7 期删除 `src.models/core/adapters/pipelines` 后，legacy 迁移器是否仍可执行？§2.3 规定迁移器直接按 SQL schema 读取且新包零旧 import，**架构测试可机械证明**。

### §3 接口、数据与用户能力契约

v5 保留开放身份、细粒度能力、新配置/输出、全命令白名单和 Web 链，并新增可实施的 change-log/recovery 契约。

- **Q3a**：已有键只刷新互动数时，是否会因业务 `collected_at` 不变而漏出 recovery？不会；repository 每次 upsert 与 change-log 同事务，实体 `updated_seq` 取 change `seq`，且契约测试显式覆盖“已有键只改互动数”，**原漏更新阻塞已关闭**。
- **Q3b**：恢复区间是否会重复/遗漏恰在水位的 change？边界固定为 `(from_seq,to_seq]`，并要求测试 seq 恰等于 from/to，**可机械证伪**。
- **Q3c**：损坏 JSONL/图片或 DB 中途异常是否会留下部分业务行？所有 count/hash/唯一性/引用在写库前校验，媒体先按 hash 原子复制，业务行与 `recovery_imports` 在单个 MySQL 事务；失败只可留可回收 blob，**不留半导入业务行**。
- **Q3d**：同一 `run_id` 的重试如何与同一运行中的合法多次更新区分？蓝图已用同 run 重试契约测试限定结果，且生产 recovery 水位在停止写入后捕获，因此同 run 的变更可聚合到同一 change seq，但实施方案必须明确 `ON DUPLICATE KEY` 时复用已有 seq，不得将合法二次 upsert 当成错误。该要求可由已列的同 run 重试/已有键测试共同验证，**不新增蓝图阻塞**。

### §4 分期表

第 1 期只做离线核心；第 2 期只做 live XHS；第 3 期只做新持久化；第 4 期只做迁移/恢复；第 5 期只做输出/消费；第 6 期切生产/观察；第 7 期在前期 Gate 2 后删旧收口。

- **Q4**：任一期失败时是否需要同时回滚两个已通过的风险域？第 1/2 期删新包子域，第 3 期删候选库，第 4 期删候选数据/工具，第 5 期删消费适配器/产物，第 6 期恢复旧入口，第 7 期回退独立清理 commit；**每期均有单独回滚单元**。

### §5 风险、禁止事项与决策

风险与禁止项已跟七期对齐：第 3 期先钉事务水位，第 4 期才做数据，第 5 期失败不拖数据底座，第 6 期 Gate 2 前禁止删旧，第 7 期清理可独立回退。

- **Q5**：七期并存时是否需要 legacy shim 才能保持旧生产可用？不需要；新旧是两个隔离 composition root，旧生产只在第 6 期切换，新包始终禁止 import 旧包，**分期没有恢复兼容层**。

### 蓝图验证备忘

备忘已覆盖本轮两个阻塞的机械检查：每次写入日志、已有键更新、故障零半导入、Web/bundle 独立验收和删旧顺序。

- **Q6**：一个已有 Content 只刷新互动数后，损坏其 recovery 图片，验证备忘能否同时捕获漏更新和媒体损坏？§3.6/3.7 的边界测试+损坏注入以及备忘的对应项可直接失败，**可证伪**。

## 新增阻塞检查

未发现新阻塞。实施细化时须保持两个已有契约：（1）change-log 唯一键冲突必须复用既有 seq，而不是让合法同 run 重试/再更新失败；（2）recovery import 的实体 upsert 仍须经 repository 在同事务记录目标库本地 change seq，不得绕开 §3.6/§5.2 禁线。两项均可在第 3/4 期方案与已要求的契约/故障测试中机械验证，不要求改变蓝图目标或分期。

## 建议栏（不立案）

1. 第 3 期详细方案将 change-log 唯一键冲突的 SQL 语义写成明示契约，并增加“同 run 同实体合法更新两次”用例。
2. 第 4 期详细方案明示 recovery import 产生目标库本地 change seq，避免后续 recovery 水位无法追踪已恢复记录。
