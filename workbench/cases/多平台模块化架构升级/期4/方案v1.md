# 多平台模块化架构升级 期4 方案 v1

## 给用户的摘要

期4负责旧数据迁移和消费面切换，但分两步执行：先建立只增不删、默认 dry-run 的幂等导入
与对账；确认字段和数量无损后，再让 Web/bundle/旧命令读取新模型。真实数据库写入和入口
切换不在未验证时自动发生。

## 1. 起点

Git 起点 `07eda553812bfea08c24d881fd6430e51ce09992`，承接期1–3工作树。

## 2. 现状与约束盘点

旧库包含 accounts/notes/comments/creator_profiles；新库候选表与旧表同库共存。旧 Web 依赖
作者头像/IP、评论图片/子评论数，故迁移前必须补齐新 schema，不能只迁最小采集字段。

## 3. 目标 / 非目标

目标：无损新 schema；幂等 legacy import；保留 first/last/fetched/comment_key/media path；
dry-run 对账；新库查询投影；保持 bundle 四文件和 Web payload。

非目标：不删除旧表/图片，不双写，不自动执行真实导入，不在对账前切 `run.sh`。

## 4. 接口与数据设计

`LegacyImporter` 同库读取旧四表，按新唯一键 upsert。`source_keywords` 拆成
`content_keywords`；`image_paths` 拆 `media_assets` 但不搬文件；评论 `comment_key` 原样保留。
命令默认 dry-run，只有显式 `--apply` 才写候选表。导入不生成 run id 或迁移台账。

## 5. 实现步骤

1. 补齐 Content/Comment 与候选表的 Web 必需字段。
2. 实现旧四表读取、转换、幂等写入和计数报告。
3. 增默认 dry-run 的 `migrate-legacy` CLI。
4. 实现新库 feed/export 查询投影与精确契约回归。
5. 对账后再决定入口切换；旧代码仍保留。

## 6. 测试策略

离线行 fixture 验证时间状态、关键词拆分、comment key、媒体路径、dry-run 零写入和显式
apply；Web/bundle 使用当前精确回归。真实 MySQL 仅在用户允许后执行。

## 7. 验收标准

期4 checklist 全绿；迁移 dry-run 数量可解释；Web/bundle 精确契约不变；旧表无删除。

## 8. 风险与禁止事项

禁止默认写库、删除旧表、搬图片、重置 fetched 时间、生成运行台账或未对账切入口。

## 9. 回滚口径

切换前删除 importer/新投影即可；候选表即使已导入也只停用，不自动 drop；旧系统保持可用。

## Gate 决策结果

- 用户确认功能独立，并要求暂不执行真实采集测试。
- 期4真实导入和正式切换继续受显式验证 Gate 约束。
