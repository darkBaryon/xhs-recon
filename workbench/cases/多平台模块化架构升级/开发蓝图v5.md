# 多平台模块化架构升级 开发蓝图 v5

## 给用户的摘要

在 `src/recon/` 旁路重写多平台模块化单体：保留历史数据、图片、已验证采集行为与测试资产；不保留旧 Python API、YAML、CSV/Web payload 或业务实现兼容。

- 七个小 Gate：离线核心 → XHS 实采 → 新数据库 → 数据迁移与恢复 → 输出/Web → 生产切换 → 删旧收口。
- 旧系统直到生产切换验收后仍完整可用；删旧是最后一个独立阶段。
- 新库每次写入都产生单调 `change_seq`，观察期 recovery 可覆盖新增和已有键更新。
- Recovery 导入先全量验 hash/count，再准备内容寻址媒体，最后单事务写库；失败可安全重试，不产生半导入业务行。

拍板项：**无**。用户已确认“旁路重写、代码不兼容、数据与图片保留”。

## 1. 整体目标 / 非目标

### 1.1 目标

1. 新建独立 `src/recon/`，新 domain/application 对旧业务包零运行时依赖。
2. 建立开放 `PlatformId + external_id` 身份、统一领域模型、细粒度能力与显式插件注册。
3. 把 search、track、research、backfill-media 做成应用用例；CLI、MySQL、输出、Web 是外层适配器。
4. 以同样本表征测试迁移聚合、排名、时间窗、典型内容选择，保持算法语义。
5. 拆迁 MediaCrawler 已验证的命令、超时、读回、评论、进度和媒体行为。
6. 建平台化 MySQL schema、事务变更日志、旧库单向迁移、图片内容寻址存储和可重放 recovery。
7. 建 manifest 平台分区输出、bundle、FeedQuery 与静态 Web 正式入口。
8. 完成可回退生产切换后，删除旧业务实现并补齐新平台接入文档。
9. 用 `test_social` contract fixture 证明新增平台不修改 domain/application。

### 1.2 非目标

- 不实现真实抖音采集。
- 不兼容旧 Python API、YAML、CSV/Markdown、bundle、Web payload 或旧数据库写接口。
- 不让新代码调用旧 pipeline/store/exporter，不双写旧表，不做 new→legacy 回填。
- Recovery 只保证回修复后的新系统；回退期间旧 UI 不展示观察期新数据。
- 不修改业务算法缺陷，不删除只读旧表/备份。
- 不引入微服务、消息队列、DI 框架、动态插件发现、代理池、验证码破解、自动互动或并发放大。

### 3.6 MySQL 事务变更序列

新实体表使用平台复合键，并含 `created_seq/updated_seq`。另建：

```sql
recon_change_log(
  seq BIGINT AUTO_INCREMENT PRIMARY KEY,
  entity_type VARCHAR(32), platform VARCHAR(32), external_id VARCHAR(128),
  operation VARCHAR(16), run_id VARCHAR(64), changed_at DATETIME(6),
  UNIQUE(run_id, entity_type, platform, external_id, operation)
)
```

- repository 每个实体 upsert 与对应 change-log 插入在**同一数据库事务**；包括仅互动数/档案字段更新。
- 实体 `updated_seq` 取该次 change `seq`；删除若未来出现也必须写 tombstone，本需求不主动删除实体。
- `capture_watermark()` 返回当前已提交最大 seq；边界固定为 `(from_seq, to_seq]`。
- 同一 run 重试由 change-log 唯一键与实体复合键保证幂等；事务失败不得留下实体已改而日志未写的状态。
- 契约测试覆盖：新键、已有键只改互动数、多实体事务回滚、同 run 重试、边界 seq 恰等于 from/to。

### 3.7 Recovery bundle 与原子导入

```text
data/recovery/<bundle_id>/
├── manifest.json  # schema, bundle_id, from_seq, to_seq, counts, 每文件 sha256
├── creators.jsonl
├── contents.jsonl
├── comments.jsonl
├── creator_profiles.jsonl
└── media/blobs/<sha256>
```

**导出**：在一致性快照内先捕获 `to_seq`，按 `(from_seq,to_seq]` change-log 得到受影响复合键，再读取每键最新实体；媒体由 `media_assets(updated_seq, sha256, ...)` 同一水位选取。写临时目录，计算全部 count/hash 后原子 rename 为 bundle。

**导入 preflight（写库前全部完成）**：校验 manifest/schema/bundle_id；逐 JSONL 解析并核 counts/hash/复合键唯一性/引用完整性；逐媒体核 sha256；任何错误零业务写入。

**导入提交**：

1. 将媒体按 sha256 原子复制到内容寻址 blob 库；已存在同 hash 跳过，失败时数据库尚未写，重试安全；孤立 blob 可由维护命令回收。
2. 单个 MySQL 事务中按复合键 upsert 全部实体/媒体映射，并写 `recovery_imports(bundle_id, manifest_sha256, status='complete')`。
3. 同 bundle+同 manifest 再导入直接 no-op；同 bundle id 不同 hash fail fast。
4. DB 事务失败全部业务行回滚，已复制 blob 无害且可重试；禁止逐文件部分提交。

验收：包含“已有 Content 只更新互动数 + 新 Content + 新图片”的范围，导出后向空候选库导入两次；实体摘要、引用、from/to、媒体 hash 全一致；分别损坏 JSONL/图片和注入 DB 中途异常，验证零业务半导入。

## 5. 风险、禁止事项与决策

### 5.1 风险控制

- 重写遗漏：表征测试先行，无证据旧分支不迁移。
- 双核心：新包零旧 import；旧系统只作切换前生产基线。
- 真实采集：第2期隔离目录/库 smoke。
- 数据损伤：第3期先钉事务水位，第4期才迁移/恢复；旧库与图片不删。
- Recovery 漏更新：change_seq 覆盖每次 upsert，固定半开区间并测已有键更新。
- 半导入：全量 preflight、内容寻址媒体、单 DB 事务、故障注入。
- 消费面失败不拖数据库：第5期独立 Gate。
- 删旧过早：第7期硬依赖第6期 Gate2。

### 5.2 禁止事项

- 禁止新包 import 旧业务包或增加旧 API/YAML/CSV shim。
- 禁止绕开 repository 写实体，避免 change-log 漏记。
- 禁止未通过全量 preflight 就导入 recovery，或逐文件提交业务数据。
- 禁止第6期 Gate2 前删旧实现。
- 禁止 live→fixture 静默降级、运行时能力探测、平台条件渗入核心。
- 禁止顺带实现抖音、修算法、铺投机接口或扩大采集行为。

### 5.3 Gate 决策结果

- **来源**：2026-07-16 用户确认项目小、个人自用、不需代码兼容，并同意按重写方案推进。
- **落实**：旁路重写；保留历史数据/图片和验证资产；不保旧 API/YAML/输出；用独立小 Gate 建新系统、迁移、切换、删旧。
- **说明**：更多分期只降低实施/回滚风险，不恢复兼容层，不改变用户选择的新架构终态。

#### 2026-07-16 Gate 收缩决策

- **来源**：用户确认长期重复任务应继续由 YAML 配置驱动，无需为个人工具保存完整运行参数、配置快照、计数和 recovery 台账；关键数据归属只需正确区分搜索关键词。
- **落实**：保留 `domain/application/platforms/infrastructure/entrypoints` 轻量六边形架构；取消事务 change-log、recovery bundle、完整 run ledger 和七期切换主线；以 `account/search/watchlist` 三个配置驱动用例组织实施。
- **处置**：本 v5 改为「需修改」，不得按此实施；替代方案见开发蓝图 v6。

## 蓝图验证备忘

逐期回填：新核心是否零旧依赖；XHS live 是否保持表征；所有写入是否同事务记录 change_seq；迁移/recovery 是否覆盖已有键更新且故障零半导入；Web/bundle 是否独立可验；用户命令是否切换前齐全；旧代码是否只在切换 Gate2 后删除；新平台是否只改插件/注册/配置。改变数据保留、change-log、命令白名单或切换顺序须出 v6。
