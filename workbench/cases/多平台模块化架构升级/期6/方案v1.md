# 多平台模块化架构升级 期6 方案 v1

## 给用户的摘要

现状是 account 批量采集、watchlist 却逐账号创建浏览器会话，证明按 application 定义
Collector 会让相同平台能力产生不同实现。改为三种能力型 Port：搜索、创作者主页列表、内容
详情；account/watchlist 共用后两种。拍板：用户确认按此实施，无待拍板项。

## 1. 起点

起点为期5已实施工作区；旧实现证据显示 `_creator_two_phase` 会一次提交整批账号，新
`SyncWatchlist` 则逐账号调用 `list_creator_notes([account_id])`。

## 2. 现状与约束盘点

- MediaCrawler adapter 已支持账号列表和帖子列表批量参数，可直接复用，不改采集内核。
- application 应决定到期、已知内容和评论刷新语义，不得决定页面、子进程或会话数量。
- 继续保持单并发、YAML、数据库、输出和用户配置不变；不增加采集范围或真实平台行为。

## 3. 目标 / 非目标

目标：只保留 `SearchCollector`、`CreatorFeedCollector`、`ContentDetailCollector`；恢复
watchlist 每批一次列表、一次详情；按账号独立保存和标记成功。

非目标：不实现抖音、不调整排名/刷新规则、不增加通用执行框架或 DI 框架。

## 4. 接口与数据设计

`CreatorFeedCollectionRequest` 接受账号元组和每账号上限，返回统一 `CreatorFeedResult`；
`ContentDetailCollectionRequest` 接受内容引用元组和评论开关，返回完整内容结果。account 组合
两者做全量，watchlist 在两者之间完成数据库 diff，backfill 只依赖内容详情。

## 5. 实现步骤

1. 收敛 collection Port 与 registry。
2. 把 XHS 账号列表和详情翻译改为共享能力实现。
3. 迁移 account/watchlist/backfill/container/progress。
4. 删除 XHS watchlist 专用采集器，补批量和部分失败测试。
5. 更新 README、平台指南和机械守护。

## 6. 测试策略

- 两个账号同批时断言 adapter 列表调用一次、详情调用一次。
- 断言批量结果仍拆为两个账号保存，空账号失败且不影响成功账号。
- account、backfill、search-only、进度、fixture CLI 和全量离线回归。

## 7. 验收标准 + 检查清单 YAML

同目录 `checklist.yaml` 为机械事实源；三种 Port 存在，旧按 app Collector 清零，批量行为、
全测、lint、格式、架构守护、工作台和用户配置哈希均通过。

## 8. 风险与禁止事项

禁止 application 按账号调用采集器；禁止失败账号误标完成；禁止修改并发、代理、验证码、
自动互动或用户日常配置。

## 9. 回滚口径

本期只改 Python 边界和测试，不迁移数据；可整体回退本期文件恢复期5实现。

## 实施记录

- 三种能力型 Port、registry、共享 XHS creator feed/detail 已落地。
- account/watchlist/backfill 已改为依赖注入共享能力；XHS watchlist 专用采集器已删除。
- 批量会话、结果拆分、失败隔离、评论开关和 search-only 契约测试已覆盖。

## 实施偏差记录

无。

## 蓝图验证备忘

本次修正进一步落实蓝图“平台能力组合”目标：application 只表达业务意图；具体平台决定
采集执行方式。抖音仍可只注册 SearchCollector，不需要空实现其他能力。

## Gate 决策结果

- 来源：用户确认采集应为通用能力并通过依赖注入共享，随后明确“就这么做”。
- 落实：按能力而非 application 定义 Port，保留三个最小采集接口。
