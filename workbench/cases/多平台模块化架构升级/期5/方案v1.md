# 多平台模块化架构升级 期5 方案 v1

## 给用户的摘要

原项目代码并不差。期5不按目录删除旧代码，而是先区分“已验证、应继续复用的基础设施”和
“已被新用例取代的业务编排”。MediaCrawler、fixture、解析器、配置、日志和进度实现迁入或
保留为新架构适配器；只在新入口不再依赖后，删除重复的旧 search/track/research 核心。

## 1. 当前事实

- `run.sh` 九个原命令已经切换到 `src.recon`，真实 watchlist 观察通过。
- 新入口复用 `src.adapters` 中的成熟采集实现；配置、日志和进度已迁入新入口。
- 期初盘点的 `src/models.py + core + adapters + pipelines + web` 约 4479 行，其中
  MediaCrawler、解析器、配置、日志和进度属于可复用技术资产，不应因架构升级重写。
- 旧数据库表与媒体继续保留，不属于本期清理对象。

## 2. 保留、迁移、删除边界

### 保留并迁移归位

- `mediacrawler_adapter.py`：真实采集、限速、超时、图片提升、日志与批次隔离。
- `fixture_adapter.py`、`parsers.py`：离线契约和平台 payload 翻译。
- `config.py` 的现有 YAML 字段与资产引用规则。
- `logging_setup.py`、`progress.py`：日志文件、TTY 进度与非 TTY 可观测性。
- Web 静态页面模板和已经验证的 XHS 展示投影。

迁移优先采用移动或薄包装，不重写成熟行为；每移动一项先保持原测试，再让新测试接管。

### 最终可删除

- 旧 `src.pipelines.cli/run_research` 的 search/track/research 编排。
- 已由 `src/recon/domain/application` 取代的 aggregator/ranker/watchlist/selector 重复实现。
- 只验证旧入口、且新契约已有等价覆盖的测试。
- 不再被任何入口引用的旧 MySQL store 与旧导出编排。

### 禁止删除

- 旧 MySQL 数据表、`data/media`、历史导出和 raw 证据。
- 尚被新入口或维护脚本引用的模块。
- 仅因目录名带 `old/legacy` 就删除、但没有引用证明和替代测试的代码。

## 3. 实施顺序

1. 建立旧模块引用清单和机械守护，确认新入口实际依赖。
2. 将配置、日志、进度收口到新 composition root，保留当前 YAML/日志行为。
3. 将 Fixture/MediaCrawler/解析器作为 XHS 基础设施归位，保留旧测试作为特征测试。
4. 让 `src/recon` 不再 import 旧业务核心。
5. 按文件删除已无引用的旧编排和重复算法，每批执行全量回归。
6. 更新 README 和抖音 search-only 接入说明；旧数据只读保留。

## 4. 验收标准

- 日常 `run.sh` 命令、YAML、日志、进度、Web 和 bundle 行为保持。
- MediaCrawler 连续列表/详情批次隔离测试持续通过。
- `src/recon` 对已删除旧业务模块零 import；架构测试机械阻止反向依赖。
- fixture demo、全量测试、真实历史 Web smoke 和工作台预检全绿。
- 抖音 search-only 插件只需实现搜索能力，不修改 account/watchlist/domain。

## 5. 回滚

基础设施迁移与旧业务删除分批进行；删除批次只在所有消费者清零后发生。任一批失败就恢复
该批文件和 import，数据库与媒体无需回滚。期5不修改真实业务数据。

## 实施记录

- 已完成引用盘点并拍板保留/删除边界。
- 已通过真实运行发现并修复 MediaCrawler 同一运行内批次目录污染，证明保留成熟适配器并
  增强边界比整体重写更合适。
- 已将新入口的配置模型、资产引用解析、时间源、日志和 TTY 进度归位到
  `src/recon/entrypoints`；新入口不再 import 旧 `src.pipelines`。
- 已为 XHS 插件建立窄 adapter contract，移除平台层对旧 `src.core.ports/src.models` 的
  类型依赖；新增 platforms/infrastructure 跨层与旧核心禁线测试。
- 当前 `src/recon` 仅在 composition root 直接装配成熟 `src.adapters`，领域、应用、平台和
  基础设施层均不依赖旧业务核心。
- 已补平台接入指南和可执行的抖音 search-only 契约演练：完整运行搜索用例，同时证明无需
  account/watchlist/detail 假实现。
- 已补领域策略专项测试，并在消费者清零后删除旧 `src.pipelines`、重复 core 算法、旧
  MySQL store、临时迁移脚本及只覆盖旧入口的测试。
- 成熟 MediaCrawler、fixture、parser、兼容 DTO 与 adapter 端口继续保留；本期没有删除旧
  数据表、历史媒体、raw 或导出证据。
- README、增量存储现状和平台接入文档已同步；全量离线回归与架构守护通过。

## 实施偏差记录

- 蓝图目录示意把 MediaCrawler 全部放在 `src/recon/platforms/xhs/mediacrawler`。实施中确认旧
  adapter 已覆盖子进程参数、超时、日志、图片和多批次行为，整体移动只会扩大风险，因此
  保留 `src/adapters/{mediacrawler_adapter,fixture_adapter,parsers}.py`，由新 entrypoints
  组装、XHS contract 隔离；新 domain/application/platforms/infrastructure 均不依赖旧业务
  核心。此偏差不保留旧 pipeline，也不改变平台能力接口。
- 旧测试没有按文件名全留；只保留仍验证采集技术资产的特征测试。仅验证已删除入口和重复
  算法的测试由 `tests/recon` 的用例、领域策略、架构和平台契约测试接管。

## 蓝图验证备忘

- YAML 仍是 account/search/watchlist 的长期输入；`run.sh` 只选择功能。
- 旧命令已切到新入口，真实 watchlist 观察通过；用户运行配置哈希保持不变。
- 同帖跨关键词仍由 `content_keywords` 区分，未引入 run ledger、配置快照或指标历史。
- 抖音 search-only 契约演练无需 account/watchlist/detail 假实现，也未修改核心。
- 旧业务 import 清零，MediaCrawler 批次隔离、全量测试、lint、工作台视图和期5 preflight
  均通过；旧数据库、历史媒体与 raw 证据未删除。
