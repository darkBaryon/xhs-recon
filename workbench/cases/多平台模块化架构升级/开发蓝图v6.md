# 多平台模块化架构升级 开发蓝图 v6

## 给用户的摘要

把当前项目改造为**配置驱动的轻量六边形模块化单体**：保留已验证的 XHS 采集、解析、媒体与算法行为，重写稳定的应用入口、平台边界和关键词归属。

- 用户长期编辑 `configs/<主题>/run.yaml`，`run.sh` 只选择 `account/search/watchlist` 等功能；CLI 参数仅作临时覆盖。
- 不建 run ledger、配置快照、change-log、recovery bundle、事件总线或通用工作流。
- 帖子只保存一份，通过 `content_keywords` 区分不同关键词的搜索归属。
- 四期按用例纵向交付：账号分析 → 搜索 → watchlist → 消费面切换收口。

拍板项：**无**。用户已确认上述使用方式和数据收缩。

## 1. 整体目标 / 非目标

### 1.1 目标

1. 建立 `domain/application/platforms/infrastructure/entrypoints` 的单向依赖边界，保持单进程、单仓库、单部署。
2. 将竞品账号分析、关键词搜索、watchlist 最新帖作为三个一等应用用例。
3. 延续当前 `CONFIG=configs/<主题>/run.yaml ./run.sh <功能>` 的重复使用习惯，不要求每次重写账号或关键词参数。
4. 使用 `(platform, external_id)` 标识账号与内容，使新增抖音搜索时无需修改核心用例。
5. 以独立关系保存关键词与内容的多对多归属，去除对帖子 `source_keywords` JSON 合并的依赖。
6. 选择性迁移现有成熟能力：行为和测试必须保留，零散 pipeline 与临时脚本由新用例取代。

### 1.2 非目标

- 不实现真实抖音采集；本次只保证平台边界允许将来增加。
- 不引入微服务、消息队列、DI 框架、动态插件发现、事件溯源或工作流引擎。
- 不为每个类建接口；仅平台采集、持久化和输出等真实边界定义 Port。
- 不保存每次运行的参数、配置快照、成功/失败数量或独立归属台账。
- 不默认保存历次互动指标快照；本期只保存帖子最新值和 `updated_at`。
- 不要求小红书与抖音能力对称；平台可显式声明只支持部分能力。

## 2. 模块边界与依赖方向

```text
src/recon/
├── domain/
│   ├── {identity,content,research}.py
│   └── policies/{aggregate,ranking,time_window,selection}.py
├── application/
│   ├── ports/{collection,repository,output}.py
│   ├── account/use_case.py
│   ├── search/use_case.py
│   ├── watchlist/use_case.py
│   └── backfill/use_case.py
├── platforms/
│   ├── {contract,registry}.py
│   └── xhs/
│       ├── {collector,config,references,translators}.py
│       └── mediacrawler/{commands,process,reader,gateway}.py
├── infrastructure/
│   ├── persistence/mysql/{schema,repository}.py
│   ├── output/{files,bundle}.py
│   ├── query/feed.py
│   ├── media/library.py
│   └── logging/setup.py
└── entrypoints/{config,container,cli}.py
```

```text
entrypoints ─┬─► application ─► domain
             ├─► platforms ───► application ports + domain
             └─► infrastructure ─► application ports + domain
```

- `domain` 不知道 XHS、抖音、YAML、Typer 或 MySQL。
- `application` 只表达用例流程与所需能力，不直接调 MediaCrawler 或 SQL。
- `platforms/xhs` 拥有 XHS 原始数据转换和采集行为；平台判断不得渗入用例。
- `infrastructure` 保存 MySQL、文件、媒体、Web 查询和日志实现。
- `entrypoints` 组装配置、平台、repository 与输出，并向 `run.sh` 提供稳定命令。

## 3. 接口与数据契约

### 3.1 配置是长期任务定义

```yaml
platform: xhs

account_analysis:
  accounts:
    - id: competitor_a
      url: https://example.invalid/account-a
  fetch_comments: false
  download_media: true

search:
  keywords: [留学辅导, 美国留学, 英国留学]
  sort: [hot, latest]
  max_items: 200

watchlist:
  accounts: [account_a, account_b]
  incremental: true
```

实际字段尽量延续现有 `configs/<主题>/run.yaml`；仅为新账号分析能力增加必要配置，不做无业务价值的全量改名。

```bash
./run.sh account
./run.sh search
./run.sh watchlist
CONFIG=configs/留学辅导/run.yaml ./run.sh search
```

### 3.2 核心能力

- `AccountAnalysis`：按配置读取一个或多个竞品账号，全量或增量获取帖子与当前互动数，默认不采评论正文，产出账号汇总和帖子排行。
- `KeywordSearch`：按配置中主关键词、扩展词、热度/时序、时间窗和数量运行现有搜索行为，并保存实际命中关键词。
- `Watchlist`：按配置读取账号列表，沿用轮转、增量、批间暂停和单采集实例锁，个别账号失败不丢失其他已获数据。

平台契约只声明用例真正需要的能力，允许平台不对称：例如未来抖音第一版可仅注册 search，选择 account 时必须给出明确的不支持错误。

### 3.3 最小数据模型

```text
creators
contents
comments
media_assets
content_keywords
```

- `creators/contents/comments/media_assets` 保存当前业务事实，平台实体以 `(platform, external_id)` 唯一。
- `contents` 保存最新互动数与 `updated_at`，重复采集执行 upsert。
- `content_keywords(platform, content_id, keyword, collected_at)` 保存内容与实际搜索词的多对多关系，唯一键为 `(platform, content_id, keyword)`。
- 同一帖子被多个关键词命中时，帖子不复制，只增加关键词关系。
- 账号分析通过 `contents.creator_id` 归属；watchlist 是配置，不另建业务归属表。
- 本蓝图不建 `collection_runs`、`account_run_contents`、`watchlist_run_contents`、`content_metric_snapshots` 和 `change_log`。

### 3.4 保留与重写边界

**保留行为并迁移实现**：MediaCrawler 命令、超时、读回、评论、进度和媒体行为；扩词、聚合、排名、时间窗、watchlist 轮转；Web、bundle 和 backfill 的有效部分。先用现有测试锁定行为，再迁入新边界。

**重写**：临时脚本与零散 pipeline 编排、账号分析正式入口、平台契约、composition root、关键词归属持久化。

## 4. 分期表

| 期 | 交付内容 | 验收口径 | 依赖 |
|---|---|---|---|
| **期1 账号分析纵向闭环** | 建最小目录和端口；从当前 YAML 读竞品账号；复用 XHS 采集；保存账号/帖子最新值；新增 `./run.sh account`；输出账号汇总与帖子排行 | fixture 全链路；真实单账号 smoke；默认无评论；重复运行幂等；不要求 CLI 重写账号 | 无 |
| **期2 搜索迁移** | 迁移搜索用例与现有算法；增 `content_keywords`；保留扩词、hot/latest、时间窗和配置入口 | 同帖跨关键词不复制；每个实际搜索词可查；与现有 fixture 稳定业务投影一致 | 期1 |
| **期3 watchlist 迁移** | 迁移 track/track-all 的增量、轮转、暂停、锁和部分失败行为；统一为 watchlist 用例 | 现有 watchlist 配置可迁移；只取新帖；单账号失败不抛弃已完成结果 | 期1 |
| **期4 消费面与收口** | 迁移 backfill/Web/bundle、补账号与搜索查询；切换剩余 `run.sh`；删除已取代的 pipeline 和临时脚本；加平台契约演练 | 日常命令只读配置即可运行；Web/导出可按关键词筛选；测试平台只实现 search 而核心不改 | 期2、期3 |

每期都必须交付可重复运行的用户功能，不设纯目录、纯接口或纯迁移工具期。

## 5. 风险与未决项

### 5.1 风险控制

- **架构先行拖慢交付**：期1必须用真实账号 smoke 证明入口，不验收空骨架。
- **重写丢失采集细节**：先以当前专项测试和 fixture 锁定行为，再按职责迁移，不凭印象重写。
- **新旧数据并存**：对旧表和媒体只做备份与单向读取；新 schema 验证前不删原数据。
- **过度抽象**：新 Port 必须同时有用例与适配器；无第二个变体且不是真实外部边界时，不拆接口。
- **配置破坏**：延续现有字段语义，提供必要的一次性迁移说明；不修改用户当前未提交的配置。

### 5.2 禁止事项

- 禁止将账号、关键词设计为日常必填 CLI 位置参数。
- 禁止建立 run ledger、事务 change-log、recovery bundle 或指标历史快照，除非后续有独立需求和真实用例。
- 禁止为了目录整齐重写已验证的 MediaCrawler 行为。
- 禁止将平台 `if/else` 放入 domain/application，或要求所有平台实现完全相同能力。
- 禁止在一期内重写全部项目；每期都保持可回退的旧入口直到对应新用例验收。

### 5.3 Gate 决策结果

- **架构**：保留轻量六边形分层，不改为纯 `jobs/` 垂直目录。
- **入口**：长期任务配置优先；`run.sh` 选功能；CLI 只临时覆盖。
- **数据**：只增业务必需的关键词归属，不建运行审计体系。
- **改造策略**：保留采集内核和已验证行为，选择性重写应用层、平台边界和数据归属。

## 蓝图验证备忘

逐期回填：用户是否只编辑 YAML 即可重复运行；账号分析是否默认无评论并产出排行；同帖跨关键词是否只存一份且可查归属；watchlist 是否保留增量/轮转/锁行为；平台是否可声明部分能力；是否出现无用例支撑的 Port 或运行台账。
