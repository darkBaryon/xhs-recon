# 多平台模块化架构升级 开发蓝图 v3

## 给用户的摘要

依据 Gate 决策，放弃围绕旧代码做兼容式重构，改为在 `src/recon/` **旁路重写新架构**：先用 fixture 跑通完整新系统，再接入真实 XHS、迁移历史数据并一次切换。

- 保留：历史 MySQL 数据、持久图片、MediaCrawler 已验证行为、解析规则、业务算法和高价值 fixtures/tests。
- 不保留：旧 Python 导入、旧配置格式、旧 CLI 内部实现、旧 CSV 列级兼容、双表双写和长期兼容壳。
- 新系统从第一天使用开放平台身份、细粒度能力端口和平台插件；未来抖音只新增平台包与配置。
- 旧系统在切换前保持原样可运行；切换成功后删除旧业务实现，旧数据库只读封存。

拍板项：**无**。用户已于 2026-07-16 确认“代码不兼容、数据保留”的推荐重写路线。

## 1. 整体目标 / 非目标

### 1.1 目标

1. 在全新 `src/recon/` 包中建立模块化单体，不让新核心依赖旧 `src.models/core/adapters/pipelines`。
2. 采用开放的 `PlatformId + external_id` 身份、平台无关领域模型和细粒度采集能力。
3. 将搜索、追踪、完整研究实现为应用用例；CLI、配置、存储、导出均为外层适配器。
4. 将经过测试的排名、聚合、时间窗、典型内容选择等规则迁移到新 domain，并用同输入/同输出表征测试验证。
5. 将 MediaCrawler 的已验证行为拆入 XHS 插件，不重新探索已知的命令、超时、读回、评论和媒体规则。
6. 建立新的平台化 MySQL schema，将旧库 XHS 历史数据单向、幂等迁入；持久图片保留并重建引用。
7. 新建平台化文件输出与 Web read model；切换后只有新系统写入。
8. 用 `test_social` fixture 插件证明新增平台不修改 domain/application。

### 1.2 非目标

- 不实现真实抖音采集。
- 不保持旧 Python API、旧 YAML、旧导出文件或旧 Web payload 的兼容。
- 不在新代码中调用旧 pipeline、旧 store 或旧 exporter 作为过渡承载。
- 不对旧表双写，不实现 v2→legacy 反向同步；旧库仅作为迁移源和封存备份。
- 不改变既有业务算法的语义；若表征测试发现旧算法缺陷，单独立案，不在迁移中顺修。
- 不引入微服务、消息队列、DI 框架、动态插件发现、代理池、验证码破解或自动互动。
- 不重写 MediaCrawler 上游项目。

## 2. 模块边界与依赖方向

### 2.1 保留资产与淘汰边界

| 类型 | 处置 | 依据 |
|---|---|---|
| fixtures 与离线 E2E 样本 | 复制/转换为新 contract fixtures | 它们锁定当前真实字段和主要行为 |
| 聚合、排名、时间窗、选典型规则 | 按新模型迁移，以表征测试对照 | 平台无关且已有专项测试 |
| MediaCrawler 命令、进程、JSONL 读回、媒体规则 | 拆分迁移，以现有测试和 mock 输出锁定 | 885 行大类职责混杂，但外部经验昂贵 |
| MySQL 历史数据、`data/media/xhs` | 单向迁移/保留 | 用户数据有价值 |
| 旧配置、CLI 私有接口、导出 schema、Web payload | 不兼容重建 | 个人自用，无外部兼容成本 |
| `src.models/core/adapters/pipelines` 业务实现 | 切换后删除 | 防止双核心长期共存 |

### 2.3 依赖铁律

```text
entrypoints ─┬─► application ─► domain
             ├─► platforms ───► application ports + domain
             └─► infrastructure ─► application ports + domain

domain ─X─► pydantic/typer/yaml/pymysql/platforms/旧 src
application ─X─► platforms/infrastructure/entrypoints/旧 src
platforms/<A> ─X─► platforms/<B>
new src/recon ─X─► src.models/core/adapters/pipelines
```

期1开始即用 AST/import 测试守护。标准库 dataclass/enum/Protocol 承载核心；Pydantic 仅用于入口配置和 raw 边界。

### 2.4 “重用”规则

- 复用的是**已验证行为和测试资产**，不是让新核心 import 旧模块。
- 允许从旧纯函数迁移实现，但落入新模型签名并通过新测试；不得留跨包代理调用。
- MediaCrawler 逻辑按职责移动后，旧适配器在切换前仍独立工作；新旧不共享可变运行状态。
- 任何无法由现有测试解释的旧分支先不迁移；确有真实用例再补契约。

## 3. 接口与数据契约

### 3.1 开放平台身份

```python
@dataclass(frozen=True, slots=True)
class PlatformId:
    value: str  # ^[a-z][a-z0-9_-]{1,31}$

@dataclass(frozen=True, slots=True)
class EntityId:
    platform: PlatformId
    external_id: str
```

`XHS = PlatformId("xhs")` 定义在 XHS 平台包；测试插件直接构造 `PlatformId("test_social")`。domain 不维护平台枚举。

核心实体统一为 `Content/Creator/Comment/CreatorProfile`；时间用带时区 `datetime`，未知发布时间为 `None`；互动量封装为 `Engagement`。平台私有字段只留 raw artifact/read model。

### 3.2 采集能力与插件

```python
class SearchProvider(Protocol):
    def search(self, request: SearchRequest) -> CollectionResult: ...

class CreatorFeedProvider(Protocol):
    def list_creator_content(self, request: CreatorFeedRequest) -> CollectionResult: ...

class ContentDetailProvider(Protocol):
    def fetch_details(self, request: DetailRequest) -> CollectionResult: ...

class CommentProvider(Protocol):
    def fetch_comments(self, request: CommentRequest) -> CollectionResult: ...

class PlatformPlugin(Protocol):
    name: str
    config_model: type[BaseModel]
    def build(self, config: BaseModel, deps: RuntimeDeps) -> PlatformServices: ...
```

`PlatformServices` 用显式可选字段声明能力；用例启动即验证，禁止 `hasattr()`/默认 `NotImplementedError`。`CollectionResult` 表达实体、raw artifact 和结构化部分失败；配置错误与 live 根目录不可用 fail fast，绝不静默切 fixture。

### 3.3 新配置与 CLI

```yaml
schema_version: 1
platforms:
  xhs:
    collector: fixture             # 或 mediacrawler
    fixture: {root: tests/fixtures}
jobs:
  search: {platforms: [xhs], keywords: [留学辅导], pages: 1, limit: 20}
  track: {targets: [{platform: xhs, external_id: "..."}]}
storage: {driver: memory}           # 真实切换后 mysql
output: {root: data/exports}
```

正式入口：`python -m src.recon.entrypoints.cli search|track|research --config <path>`；`run.sh` 在切换时直接改到新入口。旧 YAML 不解析，提供一份新 sample 和当前主题的新配置，由用户切换后只维护新格式。

### 3.4 用例、存储与输出

```python
SearchResearch.execute(SearchCommand) -> RunReceipt
TrackCreators.execute(TrackCommand) -> RunReceipt
RunResearch.execute(ResearchCommand) -> RunReceipt
```

用例只依赖 `ResearchRepository/ResearchOutput/Clock`。期1提供 `InMemoryResearchRepository` 和新版文件输出，使 fixture 可完整跑通；期2提供 MySQL 实现。

新文件契约从 v1 起自带根 `manifest.json`，数据按 `<run>/<platform>/` 分区，使用通用命名 `creators.csv/contents.csv/comments.csv`。不生成旧 `notes.csv/accounts.csv` 兼容投影。Web 只消费 `FeedQuery` read model，payload 每个实体带 platform。

### 3.5 新数据库与单向迁移

新表：`platform_creators`、`platform_contents`、`platform_comments`、`platform_creator_profiles`，主/唯一键包含 platform；迁移元数据表记录 schema version、迁移批次与源库水位。

切换协议：

1. 旧系统停止运行，记录旧库水位并做数据库备份；
2. 幂等执行 legacy→new，所有旧记录标 `platform='xhs'`；
3. 对账逐表行数、唯一键、孤儿引用、确定性内容摘要和图片文件存在性；
4. 新系统用迁移库跑 fixture + 最小真实 XHS smoke，再生成 Web/导出抽查；
5. 全绿后修改 `run.sh`/日常配置到新入口，旧表和旧代码进入只读封存；
6. 观察期内若失败，停止新系统并恢复备份/旧入口；观察期内不并行运行新旧写入。

由于这是个人单机工具，观察期内新系统采集的数据可在回退前单独导出为迁移包，再恢复旧系统；不为此建设长期双写或反向同步。Gate 2 后旧代码删除，旧表继续只读保留。

## 4. 分期表

| 期 | 交付内容 | 验收口径 | 回滚点 |
|---|---|---|---|
| **期1：新核心离线闭环** | 全新 domain/application/ports/registry；XHS fixture 与 `test_social` 插件；新配置/CLI；内存 repository；新版文件输出；迁移纯业务规则；架构/契约/表征/E2E 测试 | 新入口以 fixture 完成 search/track/research；旧新算法同样本投影一致；`test_social` 不改核心即可注册；新包对旧 src 零 import；旧系统仍全绿 | 直接删除 `src/recon` 与新测试/配置，旧系统未切换 |
| **期2：真实 XHS、数据迁移与切换** | 拆迁 MediaCrawler；MySQL repository/schema；单向迁移与对账；媒体库引用；Web FeedQuery；真实 smoke；切 `run.sh`；删除旧业务实现与旧测试，保留必要迁移工具/旧表 | mock 命令行为与旧实现一致；临时库重复迁移幂等且摘要一致；真实最小采集成功；Web/新导出可用；切换后全量 lint/test/demo/workbench 绿；旧 src 无业务实现残留 | Gate 2 前可停新系统并恢复备份/旧入口；新数据先导出迁移包，不承诺自动反向同步 |

期2实施按三个可独立评审批次进行：A. XHS live 插件；B. MySQL/媒体/Web 与迁移；C. 切换和旧代码删除。任一批阻塞，不进入下一批。两期各自走标准级 Gate；若期2详细方案证明单期超出可评审尺度，必须在 Gate 前拆期，不能实施中悄悄扩范围。

## 5. 风险、禁止事项与 Gate 决策

### 5.1 风险控制

- **重写遗漏隐含行为**：现有测试、fixture 和命令构造先形成表征基线；无证据旧分支不迁移。
- **双核心漂移**：新包禁止 import 旧业务包；旧系统只在切换前作为独立基线运行，不共同演进。
- **期2过大**：按 A/B/C 批次评审；详细方案若仍超标，Gate 前拆分。
- **历史数据损伤**：备份、只停写迁移、幂等批次、摘要对账、旧表只读保留。
- **新输出不兼容**：这是用户明确接受的代码/契约不兼容；新 Web 与文档同批交付。
- **真实采集不稳定**：fixture/mocked contract 是 CI Gate，真实 smoke 只验证环境链路，不以网络波动否定架构正确性。

### 5.2 禁止事项

- 禁止新包 import 旧 `src.models/core/adapters/pipelines`。
- 禁止为旧 API/YAML/CSV 增加兼容 shim。
- 禁止无备份、停写水位和对账就迁移/切换。
- 禁止 `hasattr()` 能力探测、live→fixture 静默降级和平台条件分支渗入核心。
- 禁止顺带实现真实抖音、修业务算法或增加无用扩展接口。
- 禁止代理池、验证码破解、自动互动或并发放大。

### 5.3 Gate 决策结果

- **来源**：2026-07-16 用户确认项目小、个人自用、不需顾虑代码兼容，并回复“可以的，按你说的来”。
- **已决策**：旁路重写新架构；复用验证过的行为/测试资产；保留历史数据与图片；不保留旧代码、配置、导出契约；先离线闭环再真实接入和切换。
- **待决策**：无。若实施前改变“数据保留”或“两期切换”口径，须出 v4 重审。

### 5.4 v2→v3 差量

| v2 | v3 |
|---|---|
| 四期渐进迁移 | 两期旁路重写与一次切换 |
| legacy config/store/output/entrypoint bridges | 全部删除，不建立兼容层 |
| 新旧数据正反同步回滚 | 旧→新单向迁移；备份+停写+观察期回退 |
| 单 XHS 保持旧输出 | 直接启用新 manifest + 平台分区输出 |
| 期4再清旧实现 | 期2切换批次删除旧业务实现 |

## 蓝图验证备忘

期1验证新核心是否真的对旧包零依赖、第二平台是否只新增插件；期2验证迁移是否幂等无损、MediaCrawler 行为是否由表征测试保持、切换后是否只剩一个业务核心。任何跨期契约、数据保留或切换策略变化必须出 v4。
