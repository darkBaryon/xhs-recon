# 多平台模块化架构升级 方案评审 #6

**评审对象**：`workbench/cases/多平台模块化架构升级/开发蓝图v6.md`

## 结论

**verdict：需修改。** v6 的轻量六边形方向与 YAML 长期任务入口可行，但蓝图 Gate 仍有四个阻塞项：

1. **兼容处置未闭合**：当前 YAML 与 v6 示例字段不同，当前 `run.sh` 九个命令也没有逐项保留/替代/删除及迁移验收；这不满足需求在蓝图 Gate 前拍板 CLI、配置和导出兼容策略的要求。
2. **历史数据与图片不可证明可用**：现库主键是裸 `account_id/note_id`，v6 改为平台复合身份和新表，却没有 schema 映射、单向迁移、对账、媒体路径处置与 Web/导出读回验收。
3. **公共契约不足以进入分期实施**：缺 Port 签名、平台注册/能力声明结构、不支持/部分失败语义、repository 原子性和配置 schema；也未把架构测试、契约测试与新平台接入文档放入明确分期验收。
4. **分期回滚点缺失**：四期表无回滚列，第 4 期同时切换入口、迁移消费面并删旧 pipeline/脚本，没有“切换验收后再独立删旧”的可回退 Gate。

## 逐章核查

### 给用户的摘要

- **可证伪质疑 Q0**：“用户长期编辑现有 `configs/<主题>/run.yaml`”是否意味着现有文件可不改或按已列映射迁移后运行？
- **验证**：亲读当前配置，实际是 `provider`、`keywords_file`、`watchlist_file`、`search.pages/limit/sort/window_days/batch_size`、`creator.*`；v6 示例却是 `platform`、`account_analysis.accounts`、`search.keywords/sort[]/max_items`、`watchlist.accounts/incremental`。蓝图只说“尽量延续”和“必要的一次性迁移说明”，没有字段映射或验收，**质疑成立，属阻塞项 1**。

### §1 整体目标 / 非目标

- **可证伪质疑 Q1**：“选择性迁移现有成熟能力”是否覆盖需求指定的全部用户可见主路径？
- **验证**：`run.sh` 当前公开 `demo/search/track/track-all/real/browser/backfill/web/bundle` 九个命令；v6 明示新命令只有 `account/search/watchlist`，§3.4/第 4 期只另提到 backfill/Web/bundle，没有逐项处置 `demo/real/browser`，也没有说明 `track/track-all` 改名是兼容还是破坏性迁移。**无法证明目标完整，属阻塞项 1**。

### §2 模块边界与依赖方向

- **可证伪质疑 Q2**：新 `domain/application` 反向 import 平台、MySQL、Typer 或旧 `src` 时，蓝图指定的哪个机械检查会失败？
- **验证**：v6 有依赖图和文字禁线，但分期表和验证备忘都没有指定 AST/import 架构测试何时交付、检查哪些边；当前 `tests/test_imports.py` 仅 import 包，不检查依赖方向。**该禁线尚不可机械证伪，属阻塞项 3**。

### §3 接口与数据契约

- **可证伪质疑 Q3a**：仅实现 search 的第二平台被调用 account/watchlist 时，系统在组装期还是运行中失败，返回什么结构化错误？
- **验证**：v6 只说“平台契约只声明用例需要的能力”和“明确的不支持错误”，没有 Port 签名、capability/registry 数据结构、失败时点和错误类型，也没有契约测试表。当前代码确实同时存在 `NotImplementedError`、`hasattr/getattr` 探测和 MediaCrawler 不可用时降级 fixture 的不同失败语义。**新契约不足以排除继续能力探测，属阻塞项 3**。
- **可证伪质疑 Q3b**：旧库中的帖子、账号、评论和图片在新 schema 下是否能被新 Web/导出查到，且数量、关联和图片可用性不退化？
- **验证**：现有 MySQL DDL 以裸 `account_id`、`note_id` 主键，帖子内联 `source_keywords` JSON，Web 直读 `notes/comments/creator_profiles` 并依赖 `image_paths` 现有路径；v6 改为 `(platform, external_id)` 与 `content_keywords`，但没有旧→新映射、外键/事务边界、媒体路径映射、对账或候选库消费验收。`content_keywords(platform, content_id, ...)` 中 `content_id` 是外部 ID 还是代理键也未定义。**历史数据可用性无法证伪，属阻塞项 2**。
- **可证伪质疑 Q3c**：运行现有用户 YAML 时，配置校验能否拒绝未知平台、错误的 sort 类型和冲突资产来源？
- **验证**：当前 `RunConfig` 默认忽略未知键，`search.sort` 是标量字符串，而 v6 示例为列表；现有资产参照解析有“双源冲突即失败”的 11 项专项回归，v6 没有配置 schema/兼容表和对应契约测试。**属阻塞项 1 和 3**。

### §4 分期表

- **可证伪质疑 Q4**：第 4 期切换后发现 Web 无法读旧数据时，能否在不恢复同期已删 pipeline/脚本的前提下回到旧入口？
- **验证**：四期表只有交付、验收、依赖，无回滚点；第 4 期同时做 backfill/Web/bundle、查询、剩余 `run.sh` 切换、删 pipeline/脚本和平台演练。§5.2 只要求“对应新用例验收前”保留旧入口，没有切换观察 Gate 与独立删旧回退单元。**回滚闭环不成立，属阻塞项 4**。

### §5 风险与未决项

- **可证伪质疑 Q5**：“对旧表和媒体只做备份与单向读取”能否防止新系统切换后的历史数据不可见？
- **验证**：该风险项没有指定谁单向读、读入何种模型、新旧数据如何合并去重、图片路径如何保留，也没有 count/hash/引用/Web 可见性验收。需求处置历史又明记用户 Gate 是“历史数据与图片保留”，**当前措施不可验，属阻塞项 2**。

### 蓝图验证备忘

- **可证伪质疑 Q6**：备忘的六项全部通过时，是否必然满足需求中“历史数据可用、现有 CLI/导出消费有兼容或迁移层、架构测试、新平台最短接入文档”？
- **验证**：备忘只覆盖 YAML 重复运行、account 输出、关键词归属、watchlist 行为、部分能力和过度 Port；即使全通，上述四项需求仍可全部缺失。**备忘不充分，对应阻塞项 1–4**。

## 可证伪质疑

1. **[阻塞]** 现有 `run.yaml` 和九个 `run.sh` 命令能否通过明确的保留/替代/删除表与迁移测试？当前不能。
2. **[阻塞]** 裸 ID 旧库和媒体能否经迁移/读取映射在新复合身份模型中通过数量、关联、图片与 Web/导出对账？当前无方案可验。
3. **[阻塞]** 一个 search-only 假平台能否在无 `hasattr/getattr/NotImplementedError` 能力探测的前提下注册，并对 account 请求 fail fast？当前缺可执行契约。
4. **[阻塞]** 第 4 期切换失败能否不依赖同期删旧代码的逆向恢复？当前不能从蓝图证明。
5. **[非阻塞基线]** 当前全量质量门是否已绿？`pytest` 207 项全通，但 `ruff check src tests web` 因 `tests/test_web_feed.py` 三个 E501 失败，因而当前基线并非全绿。

## 建议栏（不立案）

1. v7 增加“现有用户契约处置表”：逐项列出九个 `run.sh` 命令、当前 YAML 字段/资产引用、CSV/bundle/Web 消费的保留或迁移及测试。
2. 为新数据底座增加单向迁移与对账 Gate：明确旧表→新表、`source_keywords`→`content_keywords`、裸 ID→XHS 复合键、媒体路径，并在候选库验收 Web/导出。
3. 把稳定公共契约写到蓝图：Port 签名、capability/registry、结构化部分失败、repository 事务边界、配置 schema 和契约测试矩阵。
4. 将“生产入口/消费面切换与观察”和“删旧收口”拆成独立 Gate，每期列出具体回滚产物。
5. 在实施前修复或明确基线豁免 `tests/test_web_feed.py` 的三个 E501，以免后续无法将“全量 ruff 通过”用作变更 Gate。
