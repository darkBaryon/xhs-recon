# 多平台模块化架构升级 开发蓝图 v4

## 给用户的摘要

按用户确认的路线，在 `src/recon/` 旁路重写新架构：保留历史数据、图片、已验证采集行为和测试资产，不保留旧代码/API/YAML/CSV 兼容。

- 新系统依次经过：离线核心 → 真实 XHS 插件 → 新库/输出/Web → 生产切换 → 旧代码清理。
- 每期只有一个风险域、一个 Gate 和一个明确回滚点；旧系统直到生产切换验收完成都不删除。
- 新平台使用开放 `PlatformId` 和显式能力端口；未来抖音主要新增平台包、注册和配置。
- 切换观察期的新数据可导成带校验和的 recovery bundle，并可幂等重放回修复后的新库。

拍板项：**无**。用户已确认“代码不兼容、数据与图片保留、按重写方案推进”。

## 1. 整体目标 / 非目标

### 1.1 目标

1. 在 `src/recon/` 建立独立模块化单体，新核心对旧业务包零运行时依赖。
2. 建立开放的平台限定身份、统一领域模型、细粒度采集能力和显式平台注册。
3. 将 search、track、research、backfill-media 实现为应用用例；CLI/Web/输出/存储为外层适配器。
4. 用表征测试迁移聚合、排名、时间窗、典型内容选择等业务规则，不在重写中改变语义。
5. 拆迁 MediaCrawler 已验证的命令、超时、读回、评论、进度和媒体行为到 XHS 插件。
6. 建平台化 MySQL schema，将旧库和 653 张持久图片单向、幂等迁移并对账。
7. 建新版平台分区输出、bundle、FeedQuery 和静态 Web 生成入口。
8. 在完成可回退切换后删除旧业务实现，只留下新架构、迁移/恢复工具和只读旧表。
9. 用 `test_social` contract fixture 证明新增平台不修改 domain/application。

### 1.2 非目标

- 不实现真实抖音采集。
- 不保持旧 Python API、旧 YAML、旧 CSV/Markdown、旧 bundle 或旧 Web payload 兼容。
- 不让新代码调用旧 pipeline/store/exporter 作为兼容承载。
- 不双写旧表，不实现 new→legacy 数据回填；回退时新数据只保证可恢复到修复后的新系统。
- 不修改业务算法语义；发现旧缺陷单独立案。
- 不删除只读旧数据库表或备份。
- 不引入微服务、消息队列、DI 框架、动态插件发现、代理池、验证码破解、自动互动或并发放大。

## 2. 模块边界与依赖方向

### 2.1 资产处置

| 资产 | 处置 |
|---|---|
| fixtures、mock 输出、专项测试 | 转换为新 contract/characterization tests |
| 聚合/排名/时间窗/选典型 | 迁移到新 domain，以同样本投影对照 |
| MediaCrawler 行为 | 按职责拆迁，不重新探索外部经验 |
| MySQL 历史数据、`data/media/xhs` | 单向迁入新 schema/media 索引，旧资产保留 |
| CLI 用户能力 | 以新契约重写，见 §3.4；不保持内部/API 兼容 |
| 旧 YAML/导出/Web payload | 直接替换为新版本 |
| `src.models/core/adapters/pipelines` | 第5期删除业务实现 |

### 2.3 依赖铁律

```text
entrypoints ─┬─► application ─► domain
             ├─► platforms ───► application ports + domain
             └─► infrastructure ─► application ports + domain

domain/application ─X─► 旧 src、平台、基础设施、CLI/DB 框架
platforms/<A> ─X─► platforms/<B>
src/recon ─X─► src.models/core/adapters/pipelines
```

标准库 dataclass/Protocol 承载核心；Pydantic 仅在配置和 raw 边界。AST/import 测试从第1期起守护。

### 2.4 重用规则

- 复用行为和测试，不建立对旧模块的运行时 import。
- 旧纯函数可迁移实现，但必须换成新模型签名并由表征测试证明等价。
- 新旧系统在切换前是两个独立 composition root，不共享运行时对象或写入目录。
- 无测试、无真实用例解释的旧分支不迁移；不得以“也许以后用”为由复制。

## 3. 接口、数据与用户能力契约

### 3.1 身份和领域模型

```python
@dataclass(frozen=True, slots=True)
class PlatformId:
    value: str  # ^[a-z][a-z0-9_-]{1,31}$

@dataclass(frozen=True, slots=True)
class EntityId:
    platform: PlatformId
    external_id: str
```

`XHS` 常量属于 XHS 插件；`test_social` 直接构造合法值，domain 无平台枚举。核心实体为 `Content/Creator/Comment/CreatorProfile`，时间统一带时区，未知为 `None`；平台私有字段只进 raw artifact/read model。

### 3.2 采集能力与失败

```python
SearchProvider.search(SearchRequest) -> CollectionResult
CreatorFeedProvider.list_creator_content(CreatorFeedRequest) -> CollectionResult
ContentDetailProvider.fetch_details(DetailRequest) -> CollectionResult
CommentProvider.fetch_comments(CommentRequest) -> CollectionResult
```

`PlatformServices` 显式持有可用能力；用例启动验证，禁止 `hasattr()`/默认 `NotImplementedError`。`CollectionResult` 表达实体、raw artifact 与结构化部分失败。未知插件、配置错误、live 根不可用均 fail fast，不静默切 fixture。

### 3.3 新配置、存储与输出

```yaml
schema_version: 1
platforms:
  xhs:
    collector: fixture
    fixture: {root: tests/fixtures}
jobs:
  search: {platforms: [xhs], keywords: [留学辅导], pages: 1, limit: 20}
  track: {targets: [{platform: xhs, external_id: "..."}]}
storage: {driver: memory}
output: {root: data/exports}
```

用例只依赖 `ResearchRepository/ResearchOutput/Clock`。第1期用内存 repository，第3期切 MySQL。文件输出从 v1 起使用根 `manifest.json` + `<run>/<platform>/creators.csv|contents.csv|comments.csv`，不生成旧文件投影。

新表为 `platform_creators/contents/comments/creator_profiles`，唯一键含 platform；迁移元数据记录 schema、批次和源水位。

### 3.4 用户可见命令处置

| 当前能力 | 新系统处置 | 正式入口/验收 |
|---|---|---|
| `demo` | 迁移 | `./run.sh demo` → 新 fixture research |
| `search` | 迁移 | 新 `cli search` |
| `track` | 迁移 | 新 `cli track` |
| `track-all` / `track --loop` | 迁移 | 新 `cli track --loop`，保持单实例锁和批间暂停能力 |
| `real` | 迁移 | 新 `cli research` |
| `browser` | 保留 shell 能力 | `run.sh browser` 继续只启动/检查 CDP，不属于 Python 领域层 |
| `backfill` / `backfill-media` | 迁移 | 新 `cli backfill-media`，只补详情/图片、不重抓评论 |
| `web` | 迁移 | 新 `cli web`：`FeedQuery` → `data/site/` 静态站 → 输出 `web: <index>` |
| `bundle` | 迁移 | 新 `cli bundle`：按新 manifest 打研究快照 zip |
| 旧 Python 模块入口 | 删除 | 文档只列新 `src.recon.entrypoints.cli` |
| `scripts/backfill_store.py` | 由 legacy migration 替代后删除 | 迁移集成测试覆盖 |
| 旧集成脚本 | 改为新 tests/tools 或删除 | 第5期清单逐项验收 |

`run.sh` 的用户命令名保留是使用体验选择，不代表旧 Python API/配置/输出兼容；参数如与新模型冲突可在各期方案中明确调整。

### 3.5 Web 正式调用链

```text
./run.sh web
  → src.recon.entrypoints.cli web
  → FeedQuery（只读新 repository/read model）
  → StaticSiteOutput.build(payload, data/site)
  → stdout: web: data/site/index.html
  → run.sh 可选 open
```

第3期在候选新库上验收该完整链路；静态前端允许重用视觉资产，但 payload 与查询代码按新平台模型重写。

### 3.6 单向迁移、切换与 recovery bundle

legacy→new：停旧写入、备份、记录水位、幂等复制 XHS 数据、对账行数/唯一键/孤儿/摘要/图片存在性。旧表不删除。

观察期 recovery bundle 契约：

```text
data/recovery/<bundle_id>/
├── manifest.json       # schema_version, source_db, since_watermark, exported_at, counts, sha256
├── creators.jsonl
├── contents.jsonl
├── comments.jsonl
├── creator_profiles.jsonl
└── media/              # 本期新增图片，按 platform/content_id；manifest 逐文件 sha256
```

- `export-recovery --since <cutover_watermark>` 在回退前从新库导出切换后新增/更新实体和新增媒体。
- `import-recovery <bundle>` 只导入**修复后的新 schema**，按平台复合键幂等 upsert；不承诺旧系统可消费。
- 验收：导出→空候选库导入两次→行数/内容摘要/媒体 sha256 均一致；manifest/schema 不匹配 fail fast。
- 回退期间这些数据只离线保存，在旧 UI 不可见；这是“不做 new→legacy 兼容”的明确代价。

## 4. 分期表

| 期 | 单一风险域与交付 | 验收口径 | 回滚点 |
|---|---|---|---|
| **期1：新核心离线闭环** | domain/application/ports/registry；XHS fixture + `test_social`；新配置/CLI；内存 repository；新文件输出；迁移纯规则 | fixture 跑通 search/track/research；旧新算法投影一致；第二平台只改插件/注册/配置；新包零旧 import；旧系统仍绿 | 删除新包/测试/配置即可，旧系统未变 |
| **期2：真实 XHS 插件** | MediaCrawler commands/process/reader/translators/gateway；progress；search/creator/detail/comments/backfill 能力；mock/characterization + 最小真实 smoke（不切生产） | 命令逐 flag、超时、部分失败、JSONL、评论、图片、进度与旧行为对照；新 CLI 在隔离 raw/media 目录跑最小真实链路 | 删除 XHS live 实现；旧生产入口/库未变 |
| **期3：新数据与消费面** | MySQL schema/repository；legacy→new 迁移与对账；媒体索引；新 output/bundle；FeedQuery/Web 正式入口；recovery export/import；候选库全链路 | 临时库重复迁移幂等；图片/摘要对账；同裸 ID 跨平台不冲突；web/bundle 链路可用；recovery 往返两次一致 | 删除候选库/新站点，旧生产仍未切 |
| **期4：生产切换与观察** | 停写+备份+最终迁移；真实 smoke；修改 `run.sh`/日常新配置；完成一次受控日常运行；观察与回退演练；旧代码保留 | 所有用户命令处置表逐项验收；新数据 recovery bundle 可重放；用户 Gate 2 后才认定新入口为生产 | 回退：停新→导 recovery→恢复备份/旧 `run.sh`；旧代码仍在 |
| **期5：旧实现清理与文档** | 仅在期4 Gate 2 后删除旧业务实现/旧测试/替代脚本；更新 README/接入指南/架构守护；旧 DB 只读保留 | `src` 无双核心；旧符号/入口清单归零；新平台演练通过；lint/test/demo/workbench/site 全绿 | 清理独立 commit 可回退；不影响已经运行的新库/新入口 |

每期独立走标准级完整流程，上一期未 Gate 2 不开下一期。第4期未验收绝不进入删除旧代码的第5期。

## 5. 风险、禁止事项与决策记录

### 5.1 风险控制

- **重写遗漏**：先锁表征测试；无证据旧分支不迁移。
- **双核心漂移**：新包零旧 import；旧系统在第4期前仅作独立生产基线。
- **真实采集风险**：第2期在隔离目录 smoke，不碰生产库/媒体。
- **历史数据风险**：第3期候选迁移，第4期停写最终迁移；备份、摘要、旧表共同兜底。
- **观察期新数据**：recovery bundle 保证修复后可回新系统，明确不回填旧系统。
- **删旧过早**：第5期依赖第4期 Gate 2，清理独立 commit。
- **用户能力遗漏**：§3.4 为白名单；删除/新增必须经偏差裁决。

### 5.2 禁止事项

- 禁止新包 import 旧业务包或为旧 API/YAML/CSV 建 shim。
- 禁止 live→fixture 静默降级、`hasattr()` 能力探测、平台条件渗入核心。
- 禁止第4期 Gate 2 前删除旧业务实现。
- 禁止无备份、停写水位、对账和 recovery 演练就切换。
- 禁止顺带实现抖音、修算法、增加投机接口或扩大采集行为。

### 5.3 Gate 决策结果

- **来源**：2026-07-16 用户确认项目小、个人自用、不需代码兼容，并同意按重写方案推进。
- **落实**：旁路重写；保留历史数据/图片和验证资产；不保留旧代码/API/YAML/输出契约；经独立 Gate 分期后切换与清理。
- **说明**：用户没有要求固定“两期”；v3 评审已证明原期2跨多个风险域，v4 按用户目标拆成五个可回退小期，不改变重写方向。

### 5.4 v3 评审关闭表

| # | 阻塞项 | v4 修订 |
|---|---|---|
| 1 | 期2横跨 A/B/C 且无独立 Gate | 升格为第2～5期，每期独立交付/Gate/回滚 |
| 2 | Gate2前删除旧实现与回滚矛盾 | 第4期只切换且保留旧代码；第4期 Gate2 后第5期才清理 |
| 3 | 观察期新数据/图片不可恢复 | 定义 recovery bundle、export/import、幂等与 sha256 往返验收 |
| 4 | Web入口和用户能力边界缺失 | §3.4 逐项处置全部 run.sh 能力；§3.5 定义 Web 完整调用链 |

## 蓝图验证备忘

逐期回填：新核心是否零旧依赖；XHS live 行为是否由表征保持；候选迁移/recovery 是否幂等无损；全部用户命令是否在切换前可验；旧代码是否只在切换 Gate 2 后删除；新增平台演练是否只触及平台包、注册和配置。改变数据保留、命令白名单或切换/清理顺序须出 v5。
