# 多平台模块化架构升级 期2 方案 v1

## 给用户的摘要

期2把关键词搜索做成与 account 平级的独立应用接口。它只共享实体和基础设施，
不调用 account、不读 account 输出，也不更新账号采集状态。关键词仍从 YAML 读取；
同一帖子只保存一次，通过 `content_keywords` 区分由哪些实际搜索词发现。

拍板项：无。独立功能接口与暂不执行真实采集测试由用户本轮确认。

## 1. 起点

固定 Git 起点 `07eda553812bfea08c24d881fd6430e51ce09992`，承接期1未提交工作树；
用户配置 `configs/留学辅导/run.yaml` 不修改，`data/` 不清理。

## 2. 现状与约束盘点

- 旧 `_search_stage` 已有关键词扩展、分页、时间窗、账号排名和 fixture/MediaCrawler 行为。
- 旧搜索把关键词塞进实体 JSON 字段，无法稳定表达同帖跨词归属。
- 新 account 用例已证明 composition root、平台 wrapper、repository/output Port 可复用。
- search 必须独立：不能调用 account 用例，不能更新 `creator_fetched_at/comments_fetched_at`。

## 3. 目标 / 非目标

目标：新增独立 SearchCollector/SearchContents/SearchRepository/SearchOutput；保留 YAML
关键词、扩展词、分页、limit、sort、时间窗与排名；增加 `content_keywords`；提供独立
`search/latest` 输出与 fixture 闭环。

非目标：不跑真实搜索；不迁 watchlist/Web/bundle；不切换现有 `run.sh search`；
不保存 run id、配置快照、搜索排名或搜索时间。

## 4. 接口与数据设计

```python
class SearchCollector(Protocol):
    def collect_search(self, request: SearchCollectionRequest) -> SearchCollectionResult: ...

class SearchRepository(Protocol):
    def save_search(self, result: SearchCollectionResult) -> None: ...
```

`SearchContents.execute`：扩词 → 每个实际关键词独立采集 → 时间窗 → 同事务保存实体与
关键词关系 → 跨词去重分析 → 独立输出。表唯一键为
`content_keywords(platform, content_external_id, keyword)`。搜索写 creator 时
`creator_fetched_at=NULL`，重复实体也不得覆盖已有账号采集状态。

## 5. 实现步骤

1. 新增搜索领域结果、扩词/时间窗/排名策略及 Port。
2. 新增 XHS SearchCollector 与显式 registry 能力。
3. repository 增 `content_keywords` 和关键词事务。
4. 新增独立文件输出、config/container/CLI。
5. 墁跨词归属、search-only 平台、fixture E2E 和状态隔离测试。

## 6. 测试策略

- 同帖跨两个词：实体一份，关系两条，排名 keyword_count=2。
- search-only 平台可搜索，account 在采集前 fail fast。
- MySQL 同事务写内容与关键词；search 不写/清空 `creator_fetched_at`。
- YAML fixture 生成三个 search 专用文件；真实采集按用户要求暂不执行。

## 7. 验收标准

以同目录 `checklist.yaml` 为唯一机械事实源；全量 lint/test、架构禁线、用户配置哈希、
fixture E2E 和关键词事务全部通过。

## 8. 风险与禁止事项

禁止 search 调 account/watchlist；禁止运行台账；禁止静默 fixture 降级；禁止本期切换
旧 `run.sh search`，确保旧日常入口仍可回退。

## 9. 回滚口径

删除 search 新用例、XHS search wrapper、search output 和 `content_keywords` 候选表即可；
旧 `run.sh search` 未切换。

## Gate 决策结果

- 用户确认功能应是类似后端独立接口，互不耦合。
- 用户确认本期暂不测试真实采集，继续离线实施。

## 实施记录

1. 新增独立 `SearchContents` 用例、SearchCollector/SearchRepository/SearchOutput Port，
   以及关键词扩展、时间窗和账号排名策略。
2. 新增 XHS search wrapper 和 registry 搜索能力；按 `search.batch_size` 分批，
   MediaCrawler 可复用多词会话，fixture 保持逐页旧行为。
3. MySQL 新增 `content_keywords`，每个实际关键词的实体与关系同事务保存；search
   插入 creator 时不写 `creator_fetched_at`，重复时也不清空已有状态。
4. 新增 YAML search config、composition root、CLI 和 `search/<timestamp>-search` 独立输出。
5. 新增跨词归属、search-only 平台、功能互不 import、MySQL 状态隔离和 fixture E2E。
6. 补齐搜索阶段日志：开始参数、批次、逐关键词采集/过滤/归属、去重汇总和输出；E2E 同时验证控制台与文件日志。
7. 在 entrypoint 以装饰器接入 TTY 搜索进度条，复用 MediaCrawler 的关键词、页和详情事件；非 TTY 保持稳定阶段日志。

验证：`ruff check src tests web`、`pytest -q`、`bash -n run.sh`、`git diff --check`
均通过；真实搜索按用户要求未执行。

## 实施偏差记录

- 本期提供 `python -m src.recon.entrypoints.cli search --config ...` 独立接口，但未切换
  `run.sh search`；这是方案明确的回退边界，待消费面迁移后统一切换。
- XHS wrapper 对旧 adapter 的 `search_many` 做过渡性优化探测；平台对外能力仍由显式
  registry 注册，不要求未来 search-only 平台伪造 account 接口。

## 蓝图验证备忘

- account/search 应用用例互不 import；共享 repository 只复用实体写入，不共享业务状态。
- 同帖跨关键词在分析中一份、数据库实体一份，关键词关系可分别查询。
- `content_keywords` 未加入 run id、抓取时间、排名或配置快照。
- 进度显示只在 composition root 包装 collector，account/search 应用接口仍不依赖 Rich 或彼此实现。
