# 多平台模块化架构升级 开发蓝图 v8

## 给用户的摘要

把项目改造为**配置驱动的轻量六边形模块化单体**：保留已验证的 XHS 采集、解析、媒体和算法行为，重写应用入口、平台边界和关键词归属。

- 用户继续编辑 `configs/<主题>/run.yaml`；`run.sh` 只选功能，CLI 只作临时覆盖。
- account、search、watchlist 是三个一等用例；现有九个命令保留，新增 account/watchlist。
- 帖子只存一份，`content_keywords` 只记帖子与实际搜索词，不记 run id、排名或时间。
- 不建 run ledger、配置快照、change-log、recovery bundle、指标快照、事件总线或工作流引擎。
- 五期纵向交付：account → search → watchlist → 历史数据/消费面切换 → 独立删旧。

拍板项：**无**。架构、配置优先和数据收缩已由用户 Gate 确认。

## 1. 整体目标 / 非目标

### 1.1 目标

1. 建立 `domain/application/platforms/infrastructure/entrypoints` 单向依赖的单进程、单仓库应用。
2. 让账号分析、搜索、watchlist 只需维护 YAML 就能重复运行，不再写临时脚本。
3. 实体以 `(platform, external_id)` 唯一；平台可只实现 search 等部分能力。
4. 保留现有 YAML 资产、日常命令、bundle 文件契约、历史数据和图片可用性。
5. 以幂等单向导入迁移旧数据与增量状态，候选新库对账后才切换。

### 1.2 非目标

- 不实现真实抖音，不要求平台能力对称。
- 不引入微服务、队列、DI 框架、动态插件、通用 Unit of Work 或工作流引擎。
- 不保存运行参数/配置/计数/归属或历次指标，不做双写、双向迁移或 recovery。
- 不为每个类抽接口；Port 只位于平台采集、持久化和输出边界。

## 2. 模块边界与依赖方向

```text
src/recon/
├── domain/
│   ├── {identity,content,research}.py
│   └── policies/{aggregate,ranking,time_window,selection}.py
├── application/
│   ├── ports/{collection,repository,output}.py
│   ├── account/use_case.py
│   ├── search/use_case.py
│   ├── watchlist/use_case.py
│   └── backfill/use_case.py
├── platforms/
│   ├── {contract,registry}.py
│   └── xhs/
│       ├── {collector,config,references,translators}.py
│       └── mediacrawler/{commands,process,reader,gateway}.py
├── infrastructure/
│   ├── persistence/mysql/{schema,repository,legacy_import}.py
│   ├── output/{files,bundle}.py
│   ├── query/feed.py
│   ├── media/library.py
│   └── logging/setup.py
└── entrypoints/{config,container,cli}.py

tests/recon/{architecture,contract,characterization,unit,e2e}/
```

```text
entrypoints ─┬─► application ─► domain
             ├─► platforms ───► application ports + domain
             └─► infrastructure ─► application ports + domain
```

期1建 AST/import 测试机械拦截：`domain` 不反向依赖任何外层；`application` 不依赖 platforms/infrastructure/entrypoints；`platforms` 不依赖 infrastructure/entrypoints/旧 `src`，不同平台不互引；`infrastructure` 不依赖 platforms/entrypoints/旧 `src`。`entrypoints` 是唯一 composition root。

## 3. 稳定契约

### 3.1 命令与配置

| 命令 | 处置 |
|---|---|
| `demo/search/track/track-all/real/browser/backfill/web/bundle` | 命令名与当前用法保留；`track/track-all` 调新 watchlist，`real` 组合新 search+watchlist |
| `account` | 新增，从 YAML 读竞品账号 |
| `watchlist` | 新增为 `track` 同用例的直接别名 |

保留当前 `provider/mediacrawler/keywords_file/search/ranking/watchlist_file/watchlist/creator/selection/comments/logging/export/store` 字段、标量 `search.sort` 和外置资产互斥规则。只新增：

```yaml
account_analysis_file: configs/<主题>/accounts.yaml
account_analysis:
  incremental: false
  max_notes: null
  fetch_comments: false
```

默认全量、全部帖子、不采评论正文，媒体沿用 `mediacrawler.download_images`。`CONFIG=...` 和 CLI `--config` 继续支持；账号/关键词不是日常必填位置参数。

启动时 fail fast：未知字段/平台、非法 sort、资产双源冲突、`provider=mediacrawler` 但目录不可用。最后一项是唯一刻意改变的旧配置语义：旧版会静默降级 fixture，新版为避免真实任务混入假数据而报错；离线必须显式 `provider: fixture`。

### 3.2 可组合平台能力

```python
class SearchCollector(Protocol):
    def search(self, request: SearchRequest) -> CollectionResult: ...

class CreatorFeedCollector(Protocol):
    def collect_account(self, request: AccountRequest) -> CollectionResult: ...

class DetailCollector(Protocol):
    def fetch_details(self, request: DetailRequest) -> CollectionResult: ...

class CommentCollector(Protocol):
    def fetch_comments(self, request: CommentRequest) -> CollectionResult: ...

@dataclass(frozen=True)
class PlatformServices:
    platform_id: str
    search: SearchCollector | None = None
    creator_feed: CreatorFeedCollector | None = None
    details: DetailCollector | None = None
    comments: CommentCollector | None = None
```

registry 是显式 `platform_id -> PlatformServices` 映射。用例启动前校验 handler，缺失则在采集前抛 `UnsupportedPlatformCapability`。不用 `hasattr/getattr/NotImplementedError` 探测。search-only 契约测试不实现 creator handler，验证 search 成功、account 无副作用失败。

`CollectionResult` 包含已获实体和结构化子项错误；单账号/帖子失败可保留其他结果，配置、缺能力、持久化错误整体失败。

### 3.3 Repository 与部分成功

```python
class ResearchRepository(Protocol):
    def save_account_result(self, result: AccountResult) -> None: ...
    def save_search_result(self, keyword: str, result: SearchResult) -> None: ...
    def read_watchlist_state(self, creators: Sequence[EntityId]) -> WatchlistState: ...
    def save_watchlist_creator(self, result: CreatorCollectionResult) -> None: ...
    def mark_comments_fetched(self, contents: Sequence[EntityId], at: datetime) -> None: ...
```

account/watchlist 每个账号的实体 upsert 与 `creator_fetched_at` 独立事务提交，失败不标记成功；评论只在对应内容采集成功后标记 `comments_fetched_at`。search 每个关键词的实体 upsert 与关键词关系同事务。不建通用 Unit of Work 或运行台账。

### 3.4 最小数据契约

```text
creators(platform, external_id, ..., first_seen_at, last_seen_at, creator_fetched_at)
contents(platform, external_id, creator_platform, creator_external_id,
         ..., first_seen_at, last_seen_at, comments_fetched_at, updated_at)
comments(platform, content_external_id, comment_key, external_id?, ...)
creator_profiles(platform, creator_external_id, ...)
media_assets(platform, content_external_id, path, ...)
content_keywords(platform, content_external_id, keyword)
```

creator/content 以 `(platform, external_id)` 唯一；comment 以 `(platform, content_external_id, comment_key)` 唯一，`comment_key = external_id or sha256(body)`。`content_keywords` 唯一键为 `(platform, content_external_id, keyword)`，不保存其他运行数据。upsert 保留 `first_seen_at`、更新 `last_seen_at/updated_at`；增量标记只在采集成功后更新。watchlist 是配置，不建归属表。

### 3.5 历史导入

| 旧数据 | 新数据 |
|---|---|
| `accounts.account_id/first_seen_at/last_seen_at/creator_fetched_at` | `creators(platform='xhs', external_id=account_id, 同名状态)` |
| `notes.note_id/account_id/first_seen_at/last_seen_at/comments_fetched_at/...` | `contents(platform='xhs', external_id=note_id, creator_*=xhs/account_id, 同名状态)` |
| `notes.source_keywords` JSON | 拆成去重 `content_keywords(xhs, note_id, keyword)` |
| `comments(note_id, comment_key, comment_id, ...)` | `comments(xhs, note_id, comment_key, external_id=comment_id, ...)`，`comment_key` 原样迁移 |
| `creator_profiles(account_id, ...)` | `creator_profiles(xhs, account_id, ...)` |
| `notes.image_paths` 与原文件 | 拆媒体映射，不搬文件，保留原路径 |

`legacy_import` 直读旧 SQL schema，按新键 upsert，可重复执行；旧表/图片不删。对账实体数、引用、关键词数、comment key、媒体存在性、first/last/fetched_at；迁移后以同一时钟输入对比新旧 watchlist/评论 due 集合，必须一致。

### 3.6 输出兼容

`bundle` 继续输出当前 `research.json/accounts.json/notes.jsonl/README.md` 四文件和字段；新模型由 output adapter 投影回 XHS 旧契约，现有 `test_web_bundle.py` 作精确回归。bundle 仍是一次导出目录的研究快照，不是数据库 run ledger。Web/导出先保持当前 XHS 投影，只增按 keyword 筛选；多平台输出留到真实第二平台需求。

## 5. 风险、禁止事项与决策

风险控制：用当前 fixture/专项测试/真实 smoke 锁定采集；当前 YAML 作配置契约；切换前候选库对账并跑完 Web/导出/bundle；Port 必须同时有用例、适配器和契约测试。

禁止：日常必填账号/关键词 CLI 参数；run ledger/change-log/recovery/指标快照；为目录整齐重写 MediaCrawler；`hasattr/getattr/NotImplementedError` 能力探测；live 静默降 fixture；切换期删旧；新消费面验收前删旧数据/图片。

### 5.1 Gate 决策结果

- **架构**：轻量六边形分层，不改为纯 `jobs/`。
- **入口**：YAML 长期配置优先，`run.sh` 选功能，CLI 只临时覆盖。
- **数据**：只增关键词归属，不建运行审计。
- **改造**：保留采集内核，选择性重写应用层、平台边界和数据归属。
