# 多平台模块化架构升级 方案评审 #3

**评审对象**：`开发蓝图v2.md` Gate 决策 → `开发蓝图v3.md` 差量

## 结论

**verdict：需修改。** v3 已正确响应用户“个人自用、旁路重写、保留数据/图片、不保留代码/配置/导出兼容”的 Gate 决策，新包零旧 import 与复用已验证行为也能自洽。但期 2 规模、删除时点与回滚窗口、观察期新数据保全及新 Web/用户入口存在 4 个阻塞级缺口。

## v2 Gate 与需求核对

- v2 Gate 原文确认的是：项目小、个人自用、不承担代码兼容成本、按重写方案推进。
- v3 已明确不保留旧 Python API、YAML、CSV/Web payload 契约和旧 store/output/entrypoint bridge，与 Gate 方向一致。
- 需求仍要求迁移 XHS 搜索、追踪、增量存储、导出和 Web 消费，以及迁移文档和新平台接入文档；“不兼容”不等于未明示授权就删除现有用户能力。

## 逐章差量核查

### 给用户的摘要

保留与不保留边界已改为旁路重写口径，但“切换成功后删旧实现”没有把“成功”锚定在观察期/Gate 2 后。

- **Q0**：旧业务实现被删除时，是否已越过最后可回滚 Gate？§3.5 说 Gate 2 后删除，§4 却把删除作为期 2 Gate 2 前的验收项，答案不唯一，见阻塞 2。

### §1 整体目标 / 非目标

新核心、XHS live、新库、图片引用、新输出和 Web 的目标齐全；旧 API/YAML/文件/payload 不兼容也有明文。

- **Q1**：“不保持旧 CLI 内部实现”是否同时授权删除 `web/backfill-media/bundle/browser/track --loop` 等用户能力？Gate 只明确不保留代码、配置和导出兼容，需求反而明示要迁移 Web；当前不能推导出这些功能可被默认删除，见阻塞 4。

### §2 模块边界与依赖方向

“复用测试/行为，不复用运行时依赖”已明确化；新 `src/recon` 禁止 import 旧包，而 `tests/recon/characterization` 可在测试边界对照新旧输入/输出，两者不矛盾。

- **Q2**：保留旧算法和 MediaCrawler 行为是否要求新包运行时 import 旧实现？§2.4 明确要求迁移到新签名、不留跨包代理；现有算法与 MediaCrawler 均有可转换的测试资产，**质疑已关闭**。

### §3 接口与数据契约

新平台身份、能力、配置、用例、输出、新表和旧→新迁移方向自洽。但观察期的“迁移包”没有成为可回放的数据契约。

- **Q3a**：新包能否在不 import 旧代码的情况下读旧库？`migrate_legacy.py` 可按旧 schema 直接 SQL 读取，而旧 DDL 在 `src/adapters/mysql_store.py` 中明确，**方案可实施**。
- **Q3b**：观察期回退前导出的“迁移包”能否在修复后重放回新库，并保留同期新图片？蓝图未定义包 schema、媒体引用、导入器或验收方式，当前只是一份不保证可恢复的备份声明，见阻塞 3。
- **Q3c**：新输出与 Web 在切换前是否都有消费入口？新文件输出在期 1 已可 fixture 验收，Web/FeedQuery 在期 2B 后可基于新库验收，顺序本身不断链；但正式 CLI 未给 Web 调用路径，见阻塞 4。

### §4 分期表

期 1 在旧系统不变的前提下建立新离线闭环，回滚清晰。期 2 同时包含 live 采集、新 MySQL、历史迁移、媒体库、Web、真实 smoke、入口切换与旧核心删除，已经是多个独立风险域。

- **Q4a**：“A/B/C 独立评审批次”是否等价于每批都有独立验收、回滚点和 Gate？蓝图仍规定两期各走一次标准级 Gate，A/B/C 无独立交付口径和回滚点，答案为否，见阻塞 1。
- **Q4b**：期 2 Gate 2 前是否存在可执行的旧入口回滚？同一行又要求 Gate 2 验收时“旧 src 无业务实现残留”，与回滚列直接矛盾，见阻塞 2。

### §5 风险、禁止事项与 Gate 决策

风险表已识别重写漏行为、双核心、历史数据损伤和真实采集波动，但对期 2 过大只做了延后判断，而非在蓝图阶段收敛。

- **Q5a**：现在是否已有足够事实判定期 2 超出单期尺度？亲核现有实现：MediaCrawler 适配器 885 行、MySQL store 462 行、Web feed 143 行，且另含新 schema/迁移/媒体/切换/删除；答案为是，无需等详细方案再判定，见阻塞 1。
- **Q5b**：v3 声称“两期切换”是用户已决策事项吗？v2 Gate 原文只确认重写与不承担代码兼容成本，没有将“必须两期”写成用户决策；因此拆分期 2 不违反 Gate。

### 蓝图验证备忘

备忘已覆盖零旧 import、MediaCrawler 表征、迁移幂等和单核心，但没有跟踪观察期新数据可重放性与用户功能处置。

- **Q6**：如果回退后的新数据包无法重放，或 Web 没有新入口，现有备忘项是否会失败？不会，因为它们未被列入；需在修订版补入机械验收。

## 阻塞项

1. **期 2 超出可独立实施/评审的尺度**：A（live XHS）、B（MySQL/迁移/媒体/Web）、C（切换/清理）有不同依赖、验收与回滚风险，但当前只有一个期 Gate。现有代码规模已足以证伪“可等详细方案再判断”。需把 A/B/C 升格为有独立交付、验收、回滚点和 Gate 的分期，或给出同等严格的子期流程。
2. **删除旧实现与 Gate 2 前回滚相互矛盾**：§3.5 说 Gate 2 后删旧代码，§4 却把“旧 src 无业务实现残留”作为期 2 Gate 2 前验收条件，同时声称 Gate 2 前可切回旧入口。需将删除移到观察期与 Gate 2 后的独立清理期，或定义可验证的旧版恢复介质和操作。
3. **观察期新数据/图片只“可导出”，不保证可恢复**：迁移包没有 schema、媒体归档、导入/重放器和完整性验收。这与用户“数据/图片保留”决策不足以闭环。需定义可幂等重放到修复后新库的观察期快照包，或明确请用户接受这段数据只离线保留、不保证回库。
4. **新 Web 与用户功能/不兼容边界未闭合**：§3.3 正式 CLI 只有 `search|track|research`，而现有 CLI/`run.sh` 还有 `web`、`backfill-media`、`bundle`、`browser`、`track --loop` 等用户能力。新 Web 只定义 `FeedQuery` read model，未定义从 CLI/静态站生成到产物的调用入口；需求又明确要迁移 Web 消费。需逐项声明这些能力是以新契约迁移还是用户已授权删除，并给 Web 一条切换前可验收的新入口。

## 建议栏（不立案）

无；上述问题均属本蓝图的实施、回滚、数据保留和验收闭环，不是范围外新需求。
