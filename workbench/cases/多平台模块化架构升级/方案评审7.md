# 多平台模块化架构升级 方案评审 #7

**评审对象**：`workbench/cases/多平台模块化架构升级/开发蓝图v7.md`

## 结论

**verdict：需修改。** v7 已将评审 #6 的第 4 项“生产切换与删旧混在同期”实质闭合，也为其余三项增加了正确的章节；但四项阻塞尚未全部闭合，仍有三类阻塞：

1. **兼容契约自相矛盾且导出无迁移契约**：蓝图同时承诺“已有字段语义不变”和“live 失败不得降级 fixture，fixture 只能显式选择”，而当前 `provider=mediacrawler` 在目录不存在时明确降级 fixture；`bundle` 又明示换新 schema，却无旧→新文件/字段映射、版本或下游迁移层。
2. **公共 Port 与架构拦截仍不可执行**：`PlatformCollector` 强制每个平台同时提供 `search/collect_account`，search-only 平台若要满足 Protocol 只能提供蓝图又禁止的空实现/抛错方法；Repository/Output 也没有 watchlist 写入、增量标记或部分失败提交契约。架构测试列出的禁线还漏了 `platforms -> infrastructure/entrypoints/旧 src`。
3. **历史导入尚不足以证明现有数据与增量行为可用**：评论的新 `external_id` 未映射当前 `comment_key` 的 ID/正文哈希降级规则，`creator_fetched_at/comments_fetched_at/first_* / last_*` 等现行轮转、评论增量和首末次语义没有映射，`creator_profiles` 更仍留到期4方案再拍板。

## 逐章核查

### 给用户的摘要

- **可证伪质疑 Q0**：“保留现有 XHS 行为”是否与“fixture 只能由配置显式选择”同时成立？
- **验证**：当前 `configs/留学辅导/run.yaml` 显式写 `provider: mediacrawler`，并在 `mediacrawler_dir` 注释标明“不存在则降级 fixture”；`runtime.build_adapter` 在该目录不存在时也实际返回 `FixtureAdapter`。v7 §3.2 说已有字段语义不变，§3.3/§5.2 却禁止该行为。**同一份现有 YAML 的预期结果不唯一，属阻塞项 1。**

### §1 整体目标 / 非目标

- **可证伪质疑 Q1**：不保存运行归属时，新 `bundle` 能否保持当前“一次运行研究快照”的输出口径？
- **验证**：当前 `web/bundle.py` 明确从单次 `run_dir` 生成 `research.json/accounts.json/notes.jsonl/README.md`，测试锁定了文件集和字段。v7 一方面不建 run 归属，另一方面说 `bundle` 输出新 schema 研究快照，但未说候选库如何选出“这一次”，也未给现有消费者的字段迁移契约。**目标口径尚不可验，属阻塞项 1。**

### §2 模块边界与依赖方向

- **可证伪质疑 Q2**：`platforms/xhs` 直接 import `infrastructure` 或旧 `src.adapters` 时，期1架构测试是否必然失败？
- **验证**：依赖图规定 `platforms -> application ports + domain`，但蓝图列出的机械规则只禁止 domain/application 反向依赖及平台互引，没有禁止平台包依赖 infrastructure/entrypoints/旧 `src`。**按蓝图实现的测试可全绿而依赖图已被破坏，属阻塞项 2。**

### §3 稳定契约

- **可证伪质疑 Q3a**：当前 bundle 的下游程序能否不改代码消费期4新 bundle，或仅按蓝图列明的迁移表修改？
- **验证**：§3.1 只说“输出新 schema”，期4只验 `bundle 可见/smoke`；没有文件集、字段、版本、兼容适配器或迁移说明。现测试确切要求四个文件及 `account_id/note_id/side/in_window` 等字段，“能生成”不等于“消费兼容”。**属阻塞项 1。**
- **可证伪质疑 Q3b**：search-only 平台能否在不定义 account 方法、不抛 `NotImplementedError` 的前提下满足 `PlatformCollector` Protocol？
- **验证**：不能。蓝图 Protocol 本身同时要求 `search` 和 `collect_account`，与“平台可只声明部分能力”冲突；capability 只能阻止调用，不能让结构类型不再要求该方法。**属阻塞项 2。**
- **可证伪质疑 Q3c**：watchlist 单账号失败后，哪个已列 Port 能原子保存成功子项并标记 `creator_fetched_at/comments_fetched_at`？
- **验证**：当前 `Store` 端口分别有 upsert 与 `mark_*_fetched`，当前 watchlist 编排依赖这些标记轮转。v7 `ResearchRepository` 只有 `save_account_result/save_search_result/read_watchlist_state`，没有 watchlist 提交、增量标记或子项成功的事务边界，而它只为 search 规定事务。**属阻塞项 2。**
- **可证伪质疑 Q3d**：旧评论迁移两次是否会得到同一复合身份，并且不重抓已抓评论？
- **验证**：当前评论唯一性是 `(note_id, comment_key)`，`comment_key = comment_id or sha256(body)`；新 schema 写成 `comments(platform, external_id, ...)`，映射表只说“继续使用现有稳定降级键”，未定义 `external_id` 是 `comment_key`、原 comment ID 还是带 content 的复合值。同表也未映射 `comments_fetched_at/creator_fetched_at`。**幂等与增量行为无法由蓝图推导，属阻塞项 3。**

### §4 分期表

- **可证伪质疑 Q4**：期4切换后发现历史 Web 不可用时，是否可不恢复已删代码直接回到旧入口？
- **验证**：可以。v7 把生产切换放在期4并明确保留旧 pipeline/脚本，把删旧单独放到期5且要求独立提交；期4回滚栏又明确恢复旧 `run.sh` 和旧库。**评审 #6 阻塞项 4 已闭合。**
- **新的非阻塞质疑**：期4观察运行产生的新数据在恢复旧库时是否允许丢失？蓝图没有声明回滚 RPO；可在分期方案明确观察期禁止性写入或允许的丢失窗口。

### §5 风险、禁止事项与决策

- **可证伪质疑 Q5**：以当前用户 YAML 做契约 fixture 时，是应断言降级 fixture 成功，还是断言启动失败？
- **验证**：两种结果都可从蓝图逐字推出：§5.1 要以当前 YAML 锁定行为，§3.2 要已有字段语义不变，而§5.2 又禁止当前降级语义。**风险措施无法写成唯一预期的契约测试，属阻塞项 1。**
- **关于评审 #6 关闭表**：“旧库/图片无迁移对账”已部分闭合（已有表级映射、幂等导入、对账和候选库读回），但评论身份、抓取状态与 `creator_profiles` 仍未闭合；Port/config/测试契约也只部分闭合。

### 蓝图验证备忘

- **可证伪质疑 Q6**：备忘中九项全部通过时，是否必然证明旧 bundle 消费者兼容、评论/增量状态完整迁移和全部架构禁线有机械拦截？
- **验证**：不必然。备忘只问“配置/命令按处置表延续”和“旧数据/图片在消费面可见”，“可见”可在下游格式已破坏、轮转时间已清空或平台层反向依赖时仍然通过。**备忘需补输出契约投影、迁移后增量判据和完整架构边的可执行检查。**

## 可证伪质疑

1. **[阻塞]** 当前 `provider=mediacrawler` 且目录不存在的 YAML，v7 契约到底要求 fixture 降级还是 fail fast？必须二选一并给现有配置迁移/测试。
2. **[阻塞]** 当前 bundle 程序化消费者能否通过旧格式兼容层，或一张完整的文件/字段迁移表转向新 schema？当前只有生成 smoke。
3. **[阻塞]** search-only 假平台能否不定义 account 能力便满足 Protocol，且 watchlist 能否通过明确的 Repository 契约保存成功子项与增量标记？当前不能。
4. **[阻塞]** 架构测试能否拦住 `platforms -> infrastructure/entrypoints/旧 src`？当前蓝图未列这三条禁线。
5. **[阻塞]** 评论降级键、creator/comment 抓取时间和首末次时间能否在新 schema 下幂等对账，且迁移后 watchlist/评论不会被当作“从未抓取”？当前无映射与验收。
6. **[已关闭 #6-4]** 期4切换失败能否不恢复已删代码回退？能，因删旧已独立到期5。

## 建议栏（不立案）

1. 对 `provider=mediacrawler` 目录缺失明确选一：若改为 fail fast，将它列为制定好的配置迁移，而不是“字段语义不变”。
2. 为现有 CSV/bundle 消费增加稳定投影或完整的 old→new 文件/字段/版本表，并用当前 `test_web_bundle.py` 的精确契约作兼容回归，不只做 smoke。
3. 将平台能力分为可组合的小 Protocol/能力 handler，或使 registry 映射 capability 到可空 handler；同时补齐 watchlist 持久化与部分成功事务语义。
4. 在旧数据表补上 `comment_key`、评论原 ID、`creator_fetched_at/comments_fetched_at/first_* / last_*`、profile 的唯一新表映射，并将“迁移后增量选择不退化”纳入对账。
5. 保留期4/期5的分拆；在期4方案内补允许的回滚 RPO 即可，不需因此重新合期。
