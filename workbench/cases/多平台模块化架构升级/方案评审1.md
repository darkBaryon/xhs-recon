# 多平台模块化架构升级 方案评审 #1

**评审对象**：`cases/多平台模块化架构升级/开发蓝图.md`

## 结论

**verdict：需修改。** 蓝图的总体分层方向可行，但平台扩展契约、模型迁移方向、数据库回滚及分期依赖存在 4 个自相矛盾或断链的阻塞项，修订前不得进入 Gate 1。

## 逐章核查

### 给用户的摘要

渐进迁移、兼容旧入口、多平台分区导出与 v2 表的拍板项表达清楚。

- **Q0**：“新增平台不修改领域核心”能否由新增一个非 XHS 平台而不改 `domain` 证明？按 §3.1 现有 `Platform` 定义不能，见阻塞 1。

### §1 整体目标 / 非目标

目标与非目标已把“搭底座并迁移 XHS”和“不实现真实抖音”分开，范围基本收敛。

- **Q1**：第二平台 contract fixture 能否与未来真实平台走同一套领域身份契约？§3.1 规定测试平台使用独立 `TestPlatform` 值/夹具，尚不能证明真实扩展时 domain 不变。

### §2 模块边界与依赖方向

Clean Architecture 依赖禁线、模块所有权与唯一组装根均有可机械检查的表达。

- **Q2**：`PlatformPlugin.config_model: type[BaseModel]` 是否让 Pydantic 进入 domain？亲核目标目录与依赖图后确认，该契约位于外层 `platforms`，不违反 domain 禁止 Pydantic 的约束；**已关闭，不阻塞**。

### §3 接口契约草案

细粒度能力端口、显式注册、配置版本和组合用例的主体方向一致，但核心契约有两处冲突。

- **Q3a**：新增真实平台时是否可以不改 `domain.identity.Platform`？§3.1 的 `StrEnum` 只含 `XHS`，且注释明示真实插件立案时再加 `DOUYIN`，答案为否，见阻塞 1。
- **Q3b**：旧模型能否从新模型反向投影？§3.1 明示“双向投影”，而 §5.1 只允许“旧模型 → 新模型”，当前无唯一答案，见阻塞 2。

### §4 分期表

四期递进的总体顺序合理，但期 2 交付无法仅依据当期表述独立落地，期 3 回滚口径也不保数据完整。

- **Q4a**：期 2 切换新用例后，`ResearchRun` 如何经 `ResearchOutput` 产生兼容旧导出的 `RunReceipt`？§3.5 已定义这组依赖，但期 2 未列出其实现，而输出/存储实现被放到期 3，见阻塞 4。
- **Q4b**：期 3 切到仅写 v2 表后，“切回旧 repository”能否读到切换后新增数据？蓝图没有 v2→旧表回填或回滚期双写契约，答案为否，见阻塞 3。

### §5 风险与未决项

风险表覆盖双核心、数据损伤、兼容投影和行为漂移，但没有消解前述迁移与回滚矛盾。

- **Q5**：“新表 + 幂等复制 + 旧表保留”是否足以保证切换后回滚不丢数？它只保证切换前的旧→v2 迁移，不保证切换后 v2 新写入可被旧 repository 读取，故该风险尚未关闭。

## 阻塞项

1. **平台扩展与 domain 不变矛盾**：§3.1 `Platform(StrEnum)` 只含 `XHS`，并明示真实抖音插件立案时修改枚举，与摘要“新增平台不修改领域核心”矛盾。需重新确定可扩展的平台身份类型或改写承诺与契约测试。
2. **模型迁移方向自相矛盾**：§3.1 允许旧/新模型双向投影，§5.1 却只允许旧→新。需指定唯一依赖方向，并说明旧 CLI/导出兼容如何在不反向依赖旧核心的前提下实现。
3. **期 3 回滚会丢失切换后数据**：切到仅写 v2 表后，旧表不包含新写入；直接切回旧 repository/read model 会不可见这些数据。需补充可验证的回滚数据策略及对账口径。
4. **期 2 用例切换依赖断链**：§3.5 定义 `ResearchRun`/`ResearchOutput`/`RunReceipt`，期 2 要切换新用例且保持旧导出，但分期表到期 3 才实现输出/存储。需明确期 2 必需的端口实现、兼容投影与验收路径，或调整切换时点。

## 建议栏（不立案）

无新增范围外建议；本轮只要求修正蓝图内部契约、分期与回滚口径。
