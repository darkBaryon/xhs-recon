# 多平台模块化架构升级 期7 方案 v1

## 给用户的摘要

真实搜索即使每词只保留少量结果，旧实现仍会对每个候选追加详情请求，详情失败还可能退回
HTML，请求量远高于配置中的结果数。现将搜索定义为轻量发现能力：只保存搜索列表卡片；账号
分析、watchlist 等需要正文的功能继续通过内容详情 Port 独立完成。

## 1. 目标 / 非目标

目标：消除搜索的逐帖详情请求放大；验证码、IP 限流或数据风险信号出现后立即停止整场搜索；
保留关键词归属、标题、作者与列表可见互动数。

非目标：不绕过验证码、不伪装设备或人类行为、不更换代理或账号、不承诺平台永不触发风控。

## 2. 接口与职责

- application 仍只依赖 `SearchCollector`，不感知浏览器与列表/详情页面。
- XHS adapter 将 search 固定映射为 `LIST_ONLY`；详情由 `ContentDetailCollector` 承担。
- MediaCrawler 的 XHS search 在列表模式直接落 `search_notelist`，不调用详情、评论和媒体接口。
- 列表卡片携带 `source_keyword`，数据库仍能区分不同关键词的搜索归属。

## 3. 风险熔断

- CAPTCHA 与 IPBlock 不自动重试。
- MediaCrawler 首次数据请求错误终止当前搜索会话，不继续下一个关键词。
- adapter 将风险日志转成显式失败；application 发现该失败后不执行后续 batch。
- 风控只在日志可见时自动识别；浏览器单独显示而接口无异常时无法可靠自动判断。

## 4. 验收

- 搜索命令含 `--list_only yes`。
- 列表卡片可解析为帖子、账号和关键词关系。
- 风险信号使部分结果作废，并阻止后续关键词批次。
- 全量离线测试通过；不执行真实 XHS 验证。

## 5. Gate 决策结果

- 来源：用户报告真实搜索已触发风控，要求该问题必须解决。
- 落实：用降低请求和硬熔断修复，不使用任何规避平台安全控制的手段。
