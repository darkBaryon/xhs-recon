# 多平台模块化架构升级 开发蓝图 v2

## 给用户的摘要

把当前围绕 XHS 生长的项目升级为**多平台模块化单体**：领域和用例只认统一内容模型与细粒度能力端口，小红书/未来抖音各自在独立平台包中完成采集和翻译，MySQL、导出、Web 属于外层基础设施。

- Before：新增平台需修改全局配置、运行时分支、主编排、模型、存储和导出，部分能力靠 `hasattr()` 猜。
- After：平台名是开放值对象；新增平台主要新增 `platforms/<name>/`、显式注册并通过契约测试，不修改领域核心。
- 四期渐进迁移；旧 CLI、配置、MySQL 和导出经外层兼容桥继续工作，每期可验证、可回滚。
- 本蓝图只建设底座并迁移 XHS；抖音真实接入另行立案。

推荐拍板：渐进迁移；单 XHS 保持旧导出、多平台按平台分区；新建 v2 表并通过停写+双向迁移脚本提供可核验回滚。若用户回复“按推荐方案”，视为接受三项默认。

## 1. 整体目标 / 非目标

### 1.1 目标

1. 采用“Clean Architecture 依赖规则 + 按用例组织 + 平台插件化”的模块化单体。
2. 内容、创作者、评论使用开放的 `platform + external_id` 唯一身份，跨平台同 ID 不冲突。
3. 以搜索、创作者列表、内容详情、评论等细粒度端口取代宽泛 `ResearchAdapter`、`hasattr()` 与隐式 `NotImplementedError`。
4. 平台插件自治其配置 schema、引用解析、采集和 raw→domain 翻译；组装根不写平台条件分支。
5. 搜索、追踪、完整研究成为独立应用用例；CLI 只解析参数并调用公开用例。
6. 拆分 MediaCrawler 大适配器，使命令、进程、读回、翻译和媒体归档可独立测试。
7. 迁移 MySQL、文件导出和 Web read model，并保持现有 XHS 用户入口与数据可恢复。
8. 用任意字符串标识的第二平台 contract fixture 证明扩展时 domain 零修改。

### 1.2 非目标

- 不实现抖音真实搜索、登录、解析、下载或反爬。
- 不改变排名、专业度、时间衰减、典型内容选择等业务公式。
- 不替换 MediaCrawler、MySQL、Typer、Pydantic 或静态 Web 技术栈。
- 不引入微服务、消息队列、DI 框架、动态第三方插件发现或工作流 DSL。
- 不做代理池、验证码破解、自动互动或并发放大。
- 不删除旧数据库表；清理历史表另案处理。
- 不保留两套业务算法；兼容桥只翻译配置、数据和输出，不承载业务规则。

## 2. 模块边界与依赖方向

### 2.1 现状依据

- `src/pipelines/runtime.py:49-81` 按 provider 条件分支构造具体适配器。
- `src/pipelines/config.py:60-113` 混放平台专属和通用配置。
- `src/core/ports.py:13-26` 未声明编排实际探测调用的全部能力。
- `src/pipelines/run_research.py:20` 从 XHS parser 导入引用归一化逻辑。
- `src/adapters/mediacrawler_adapter.py:65-885` 混合命令、进程、读回、翻译、媒体和多用例流程。
- `src/models.py:6-121` 与 MySQL 表均以裸 `note_id/account_id` 标识。
- `web/feed.py:128-134` 直接依赖具体表结构。

因此保留的是依赖倒置原则，不把现有目录或大接口当作目标架构。

### 2.3 依赖规则

```text
entrypoints ─┬─► application ─► domain
             ├─► platforms ───► application ports + domain
             └─► infrastructure ─► application ports + domain

compatibility bridges ─► new contracts + legacy modules
new domain/application ─X─► compatibility/legacy/platform/infrastructure
platforms/<A> ─X─► platforms/<B>
```

- `domain` 仅标准库 dataclass/enum/正则；不依赖 Pydantic、Typer、YAML、MySQL 或平台包。
- Pydantic `config_model` 位于外层 plugin/entrypoint 边界，符合上述规则。
- 兼容桥可以双向转换**数据流**，但代码依赖单向指向新契约和旧模块；新 domain/application 永不 import 旧类型。这修正 v1 把“数据投影方向”和“代码依赖方向”混为一谈的问题。
- AST/import 测试机械守护禁线，不新增架构框架。

### 2.4 所有权

- `domain`：跨平台统一语言、身份、业务规则。
- `application`：用例、所需能力、事务顺序与失败语义。
- `platforms/<name>`：平台配置、引用、采集、raw 翻译。
- `infrastructure`：数据库、文件、read model、日志、媒体和兼容桥。
- `entrypoints`：唯一组装根，只负责解析、注册、注入、调用。

## 3. 接口契约草案

### 3.1 开放平台身份

```python
_PLATFORM_RE = re.compile(r"^[a-z][a-z0-9_-]{1,31}$")

@dataclass(frozen=True, slots=True)
class PlatformId:
    value: str

    def __post_init__(self) -> None:
        if not _PLATFORM_RE.fullmatch(self.value):
            raise ValueError("invalid platform id")

@dataclass(frozen=True, slots=True)
class EntityId:
    platform: PlatformId
    external_id: str

XHS = PlatformId("xhs")              # 定义在 platforms/xhs，不在 domain 枚举
TEST_SOCIAL = PlatformId("test_social")
```

新增 `douyin` 只需平台包构造 `PlatformId("douyin")`；domain 无需修改。registry 在启动时拒绝重复名称。

统一 `Content` 使用 `EntityId`，并校验内容与创作者 platform 相同；时间为带时区 `datetime`，未知发布时间用 `None`。平台专属字段保留在 raw artifact/read model，只有跨用例消费的稳定字段才进入 domain。

### 3.2 能力端口与失败

```python
class SearchProvider(Protocol):
    def search(self, request: SearchRequest) -> CollectionResult: ...

class CreatorFeedProvider(Protocol):
    def list_creator_content(self, request: CreatorFeedRequest) -> CollectionResult: ...

class ContentDetailProvider(Protocol):
    def fetch_details(self, request: DetailRequest) -> CollectionResult: ...

class CommentProvider(Protocol):
    def fetch_comments(self, request: CommentRequest) -> CollectionResult: ...

@dataclass(frozen=True)
class PlatformServices:
    platform: PlatformId
    search: SearchProvider | None = None
    creator_feed: CreatorFeedProvider | None = None
    content_detail: ContentDetailProvider | None = None
    comments: CommentProvider | None = None
```

- 用例启动时显式校验所需能力；缺必需能力抛 `CapabilityUnavailable`，可选能力按用例规则跳过并记录。
- `CollectionResult` 统一携带实体、raw artifact 与 `CollectionFailure(code, subject, message, retryable)`，允许部分成功。
- 配置错误、未知插件、live collector 根目录不可用均 fail fast；fixture 必须显式选择，不允许静默降级。

### 3.3 插件与配置

```python
class PlatformPlugin(Protocol):
    name: str
    config_model: type[BaseModel]
    def build(self, config: BaseModel, runtime: RuntimeDeps) -> PlatformServices: ...
```

`entrypoints.container` 显式注册 `XhsPlugin()`；registry 本身不 import XHS，也不做运行时扫描。

v2 配置：

```yaml
schema_version: 2
platforms:
  xhs:
    enabled: true
    collector: mediacrawler       # 或 fixture
    mediacrawler: {root: ../MediaCrawler, login_type: qrcode}
jobs:
  search: {platforms: [xhs], pages: 1, limit: 20, window_days: 30}
  track:
    targets: [{platform: xhs, external_id: "..."}]
storage: {enabled: true, database: xhs_recon}
output: {root: data/exports}
```

旧配置经 `LegacyConfigAdapter` 转为 v2 命令/config DTO 并提示弃用；现有 `configs/sample.yaml`、主题配置和 `run.sh` 本需求内继续运行。新文档只教授 v2。删除旧配置支持须另案 Gate。

### 3.4 应用、repository 与 output

```python
class SearchResearch:
    def execute(self, command: SearchCommand) -> RunReceipt: ...

class TrackCreators:
    def execute(self, command: TrackCommand) -> RunReceipt: ...

class RunResearch:
    def execute(self, command: ResearchCommand) -> RunReceipt: ...
```

用例内部产生 `ResearchRun`，依次调用注入的 `ResearchRepository` 与 `ResearchOutput`，最终返回包含运行 ID、持久化统计、产物路径和失败摘要的 `RunReceipt`。

**期2 必备兼容实现明确如下：**

- `LegacyXhsRepositoryBridge`：实现新 repository port，调用现有 `Store/MySQLStore`；允许在桥内进行新/旧 DTO 双向数据转换，但新 application/domain 不 import 旧类型。
- `LegacyXhsFileOutput`：直接把 canonical model 投影为现有 XHS CSV/Markdown schema，不把旧模型带回应用层。
- `LegacyEntrypointShim`：保持 `src.pipelines.cli`、`run.sh` 和旧 Python 主入口，单向转调新 CLI/use cases。

期2 切新用例时以上三项同批落地，因此数据库/导出完整 v2 化虽在期3，旧行为仍有实现承接。

### 3.5 v2 存储、导出与回滚

v2 表：`platform_creators`、`platform_contents`、`platform_comments`、`platform_creator_profiles`，唯一键含 `platform`。

切换协议：

1. 停止采集写入并记录水位 `cutover_run_id`；
2. 幂等执行 legacy→v2（现有数据标 `xhs`）；
3. 对账每表行数、唯一键、孤儿引用和确定性内容摘要；
4. shadow-read 比较旧/v2 repository 的 XHS read model；
5. 通过后切新 repository 并恢复写入。

回滚协议：

1. 再次停写并记录 `rollback_run_id`；
2. 对 `platform=xhs` 执行幂等 v2→legacy 反向同步，覆盖切换后的新增/更新；
3. 重跑行数、唯一键、孤儿引用和内容摘要对账；
4. 对账全绿才切回旧 repository/read model；失败则保持 v2、不允许带数据缺口回退。

本需求没有真实非 XHS 写入，反向同步只需覆盖 XHS 全字段；未来首个真实第二平台上线前，旧 repository 回滚路径自然失效，届时由该平台需求重新定义版本回滚支持窗。

导出：单 XHS 根目录保持旧文件名/列；多平台根写 `manifest.json`，数据写 `<run>/<platform>/...`。Web 通过 `FeedQuery` 读稳定 read model，payload 带 platform，不再直接查询业务表。

## 5. 风险与禁止事项

### 5.1 风险控制

- **双核心**：兼容桥可转换数据，但业务规则只存在于新 domain；架构测试禁止新 application/domain import legacy。
- **行为漂移**：期2 搬迁前锁定命令、超时、错误、读回、媒体和导出 characterization tests。
- **历史数据**：新表、停写水位、正反迁移、摘要对账和旧表保留共同兜底。
- **旧契约拖延**：shim 有清单和测试；删除需外部调用盘点，不能以“清理”为由破坏入口。
- **过度插件化**：只做进程内显式注册，不做动态发现；没有真实用例的接口不加入。
- **单期膨胀**：任一期超过标准级尺度则继续拆期，不扩大当期范围。

### 5.2 禁止事项

- domain/application 禁止出现 XHS、抖音、MediaCrawler、pymysql、Typer 依赖。
- 禁止用 `hasattr()`、`AttributeError` 或默认 `NotImplementedError` 代替能力契约。
- 禁止 live 配置错误时静默切 fixture。
- 禁止无停写水位与对账就切数据库或回滚。
- 禁止代理池、验证码破解、自动互动、并发放大。
- 禁止顺手实现真实抖音或未被用例消费的泛化接口。

## 蓝图验证备忘

各期回填：开放 `PlatformId` 是否保持 domain 零修改；能力端口是否覆盖 XHS 且无运行时探测；兼容桥是否未承载业务规则；正反迁移是否实测无缺口；新平台演练是否只触及平台包、注册和配置。改变目标、跨期契约或回滚协议时必须出蓝图 v3 重审。

## Gate 决策结果

- **来源**：2026-07-16 用户在 Gate 1 明确项目规模小、个人自用、不需要承担代码兼容成本，并确认“按重写方案”推进。
- **影响**：本版渐进兼容、旧入口 shim、旧配置适配、旧 repository/output bridge 与正反双向回滚不再成立；按变更协议停止进入实施，另出开发蓝图 v3。
