# 多平台模块化架构升级 开发蓝图 v7

## 给用户的摘要

把项目改造为**配置驱动的轻量六边形模块化单体**：保留现有 XHS 采集、解析、媒体和算法行为，选择性重写应用入口、平台边界和关键词归属。

- 用户继续长期编辑 `configs/<主题>/run.yaml`；`run.sh` 只选功能，CLI 参数仅作临时覆盖。
- 新增配置驱动的竞品账号分析；搜索与 watchlist 迁移已有行为，不重造采集内核。
- 帖子只存一份，`content_keywords` 只记录帖子与实际搜索词的关系。
- 不建 run ledger、配置快照、change-log、recovery bundle、指标快照、事件总线或工作流引擎。
- 五期纵向交付：账号分析 → 搜索 → watchlist → 历史数据/消费面切换 → 删旧收口。第五期只为让第四期可独立回退。

拍板项：**无**。使用方式、数据收缩和架构方向已由用户 Gate 确认。

## 1. 整体目标 / 非目标

### 1.1 目标

1. 建立 `domain/application/platforms/infrastructure/entrypoints` 单向依赖，保持单进程、单仓库、单部署。
2. 将 account、search、watchlist 建成三个稳定用例，终结“每次临时写脚本”的用法。
3. 继续使用 `CONFIG=configs/<主题>/run.yaml ./run.sh <功能>`，保留现有配置资产和日常命令。
4. 实体以 `(platform, external_id)` 唯一，平台可只声明自己支持的能力，为未来抖音 search 留出真实接入点。
5. 将内联 `source_keywords` JSON 迁为可查询的多对多关系，同帖跨关键词不复制。
6. 历史数据与图片单向迁移后仍可由 Web/导出读取；迁移前不删旧表和媒体。

### 1.2 非目标

- 不实现真实抖音采集，不要求平台能力对称。
- 不引入微服务、消息队列、DI 框架、动态插件发现或通用工作流。
- 不为每个类建接口；只在平台采集、持久化和输出边界定义 Port。
- 不保存每次运行参数、配置快照、计数、运行归属或历次指标。
- 不做双向迁移、双写、change-log 或 recovery 系统；只做幂等单向导入和对账。

## 2. 模块边界与依赖方向

```text
src/recon/
├── domain/
│   ├── {identity,content,research}.py
│   └── policies/{aggregate,ranking,time_window,selection}.py
├── application/
│   ├── ports/{collection,repository,output}.py
│   ├── account/use_case.py
│   ├── search/use_case.py
│   ├── watchlist/use_case.py
│   └── backfill/use_case.py
├── platforms/
│   ├── {contract,registry}.py
│   └── xhs/
│       ├── {collector,config,references,translators}.py
│       └── mediacrawler/{commands,process,reader,gateway}.py
├── infrastructure/
│   ├── persistence/mysql/{schema,repository,legacy_import}.py
│   ├── output/{files,bundle}.py
│   ├── query/feed.py
│   ├── media/library.py
│   └── logging/setup.py
└── entrypoints/{config,container,cli}.py

tests/recon/{architecture,contract,characterization,unit,e2e}/
```

```text
entrypoints ─┬─► application ─► domain
             ├─► platforms ───► application ports + domain
             └─► infrastructure ─► application ports + domain
```

`tests/recon/architecture` 从期1开始机械拦截：`domain` 不得 import application/platforms/infrastructure/entrypoints/旧 `src`；`application` 不得 import platforms/infrastructure/entrypoints/旧 `src`；不同 `platforms/<id>` 不得互相 import。

## 3. 稳定契约

### 3.1 现有用户入口处置

| 现有命令 | 处置 |
|---|---|
| `demo` | 保留，切换到新 fixture research |
| `search` | 保留命令与配置用法，期2切换新 search |
| `track` | 保留，作为新 watchlist 用例的旧名别名 |
| `track-all` | 保留，等价于 watchlist loop，保留暂停和单例锁 |
| `real` | 保留，组合新 search + watchlist，不建新业务层 |
| `browser` | 保留当前 CDP 启动/检查 |
| `backfill` | 保留补详情/图片、不重采评论语义 |
| `web` | 保留命令，切换到新 FeedQuery |
| `bundle` | 保留命令，输出新 schema 的研究快照；不是 recovery bundle |
| `account` | 新增，从 YAML 读竞品账号 |
| `watchlist` | 新增更直接的命令名；与 `track` 调用同一用例 |

### 3.2 配置兼容契约

以当前 `RunConfig` 和 YAML 为基线，保留 `provider/mediacrawler/keywords_file/search/ranking/watchlist_file/watchlist/creator/selection/comments/logging/export/store`。

- `keywords_file` 与已有 inline keywords/synonyms 互斥规则保留。
- `watchlist_file` 与 `watchlist.manual` 互斥规则保留。
- `search.sort` 继续使用现有标量字符串和 MediaCrawler 取值，不改成列表；需跑 hot/latest 时使用两份主题配置或临时覆盖。
- 未知顶层字段、未知平台、非法 sort 和双资产源在启动时 fail fast；已有字段语义不变。
- 只新增 `account_analysis_file` 和 `account_analysis`：前者指向长期竞品账号资产，后者仅包含全量/增量、最大帖数、评论和媒体选项。默认全量、不采评论正文，媒体沿用 `mediacrawler.download_images`。
- CLI 必须可显式指定 `--config`，`run.sh` 继续支持 `CONFIG`；账号/关键词不设为日常必填位置参数。

### 3.3 平台与应用 Port

```python
@dataclass(frozen=True)
class PlatformCapabilities:
    search: bool
    creator_feed: bool
    content_detail: bool
    comments: bool

class PlatformCollector(Protocol):
    platform_id: str
    capabilities: PlatformCapabilities
    def search(self, request: SearchRequest) -> CollectionResult: ...
    def collect_account(self, request: AccountRequest) -> CollectionResult: ...

class ResearchRepository(Protocol):
    def save_account_result(self, result: AccountResult) -> None: ...
    def save_search_result(self, keyword: str, result: SearchResult) -> None: ...
    def read_watchlist_state(self, creator_ids: Sequence[EntityId]) -> WatchlistState: ...

class ResearchOutput(Protocol):
    def write_account(self, result: AccountAnalysisResult) -> None: ...
    def write_search(self, result: SearchAnalysisResult) -> None: ...
```

- registry 使用显式 `platform_id -> PlatformCollector` 映射，启动用例前校验所需 capability；未注册平台或缺能力抛结构化 `UnsupportedPlatformCapability`，不进入采集。
- 不使用 `hasattr/getattr/NotImplementedError` 做能力探测，不把 live 错误静默降级成 fixture。fixture 只能由配置显式选择。
- `CollectionResult` 包含已获实体和结构化子项错误；单账号/单帖失败可保留其他结果，配置错误、平台缺能力和持久化失败整体失败。
- `save_search_result` 在一个 MySQL 事务内 upsert 实体并插入对应关键词关系；不建通用 Unit of Work 框架。

### 3.4 最小数据契约

```text
creators(platform, external_id, ...)
contents(platform, external_id, creator_platform, creator_external_id, ..., updated_at)
comments(platform, external_id, content_platform, content_external_id, ...)
media_assets(platform, content_external_id, path, ...)
content_keywords(platform, content_external_id, keyword)
```

- 实体表保存当前最新业务事实，使用复合唯一键；内容重采只更新最新指标和 `updated_at`。
- `content_keywords` 唯一键为 `(platform, content_external_id, keyword)`，只表达“此帖由此词搜到”，不保存 run id、排名、配置或运行时间。
- 账号分析通过 creator 外键归属；watchlist 是配置，不另建归属表。
- 不建 `collection_runs/account_run_contents/watchlist_run_contents/content_metric_snapshots/change_log`。

### 3.5 旧数据和图片单向导入

`legacy_import` 直接按旧 SQL schema 读取，不 import 旧 Python store/DTO：

| 旧数据 | 新数据 |
|---|---|
| `accounts.account_id` | `creators(platform='xhs', external_id=account_id)` |
| `notes.note_id/account_id` | `contents(platform='xhs', external_id=note_id, creator_*=xhs/account_id)` |
| `notes.source_keywords` JSON | 拆成去重的 `content_keywords(xhs, note_id, keyword)` |
| `comments` | 带 `platform='xhs'` 迁移，继续使用现有稳定降级键 |
| `creator_profiles` | 合并到 creator 详情或新等价表，期4方案拍板 |
| `notes.image_paths` 与现有文件 | 迁媒体映射但不搬动原文件，保留可读路径 |

- 导入使用新复合键 upsert，可重复执行；旧表和图片不删。
- 对账项：各实体去重行数、creator-content/comment 引用完整性、关键词拆分数与去重结果、媒体文件存在性。
- 切换前必须用候选新库生成 Web/导出/bundle，抽查历史账号、关键词、帖子、评论和图片可读。

## 4. 分期表

| 期 | 交付内容 | 验收口径 | 回滚 |
|---|---|---|---|
| **期1 account 纵向闭环** | 最小 domain/ports/registry/config/container；XHS account 采集迁移；新 account 配置资产与命令；账号/帖子保存和分析输出；架构与 Port 契约测试 | fixture 全链路；真实单账号 smoke；默认无评论；重复运行幂等；账号只从 YAML 读 | 删新 account 入口和新表/候选数据；旧入口/库/配置未改 |
| **期2 search 迁移** | 迁现有 search 和算法；增 `content_keywords`；保留配置、扩词、sort、时间窗；增 search-only 测试平台 | 同帖跨词只一份且关系可查；新旧 fixture 稳定投影一致；search-only 平台调 account 在采集前 fail fast | `run.sh search` 仍指旧 pipeline；删新 search 用例与候选关系数据 |
| **期3 watchlist 迁移** | 迁 track/track-all 的增量、轮转、暂停、锁与部分失败；增 `watchlist` 别名 | 现有 watchlist YAML 可直接用；只取到期/新帖；单账号失败不丢已完成结果 | `track/track-all` 仍指旧 pipeline；删新 watchlist 用例 |
| **期4 迁移与生产切换** | 幂等旧库/媒体导入对账；迁 demo/real/backfill/Web/bundle；用候选库验收历史消费面；备份后切换 `run.sh`；保留旧代码 | 九个原命令+两个新命令逐项 smoke；配置兼容测试；历史 count/引用/关键词/图片对账；Web/导出/bundle 可见 | 恢复旧 `run.sh` 与旧库备份；旧 pipeline/脚本仍在 |
| **期5 删旧收口** | 仅在期4 Gate 2 与观察运行通过后删已取代 pipeline/临时脚本，补新平台最短接入文档 | 旧用户命令仍可用；旧业务 import 清零；全量测试/架构测试通过；按文档增 search-only 平台无需改核心 | 删旧使用独立提交，直接回退该提交；新系统与数据不变 |

每期都交付可重复运行的用户功能；不设纯目录、纯接口、纯恢复系统期。

## 5. 风险、禁止事项与决策

### 5.1 风险控制

- 重写丢采集细节：用当前 fixture、专项测试和真实 smoke 锁行为，再迁入新边界。
- 架构先行：期1验收可重复的账号分析，不验收空骨架。
- 配置破坏：以当前 YAML 为契约测试 fixture，用户未提交配置不得被改写。
- 历史数据不可见：期4切换前用候选库完成对账和全消费面读回。
- 过度抽象：新 Port 必须同时有用例、适配器和契约测试。

### 5.2 禁止事项

- 禁止将账号、关键词设计为日常必填 CLI 位置参数。
- 禁止新建 run ledger、change-log、recovery bundle 或历次指标快照，除非后续有独立需求。
- 禁止为目录整齐重写已验证的 MediaCrawler 行为，或让新平台修改 domain/application。
- 禁止用 `hasattr/getattr/NotImplementedError` 探测能力，或把 live 失败静默降级 fixture。
- 禁止在生产切换期删旧代码，或在新库消费面验收前删旧数据/图片。

### 5.3 Gate 决策结果

- **架构**：轻量六边形分层，不改为纯 `jobs/` 目录。
- **入口**：YAML 长期配置优先，`run.sh` 选功能，CLI 只临时覆盖。
- **数据**：只增关键词归属，不建运行审计体系。
- **改造**：保留采集内核和已验证行为，选择性重写应用层、平台边界和数据归属。

## 蓝图验证备忘

逐期回填：是否只编辑 YAML 就能重复运行；account 是否默认无评论并产出汇总/排行；同帖跨关键词是否只存一份且可查归属；watchlist 是否保留增量/轮转/锁；平台是否可声明部分能力；现有配置/命令是否按处置表延续；旧数据/图片是否在新消费面可见；生产切换是否能不恢复已删代码直接回退；是否出现无用例支撑的 Port 或运行台账。
