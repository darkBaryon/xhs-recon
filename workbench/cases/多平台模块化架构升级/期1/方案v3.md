# 多平台模块化架构升级 期1 方案 v3

## 给用户的摘要

期1直接交付可重复使用的竞品账号分析，不再搭纯离线骨架。

- Before：要采某个账号时必须临时写脚本或借用 track，且 creator 采集强制带评论。
- After：账号长期放在 YAML/资产文件，运行 `./run.sh account`；默认获取全部帖子、不采评论，保存最新数据并输出汇总/排行/时间分布。
- 新应用层使用轻量六边形边界；XHS 平台层包装现有已验证 adapter，不复制 MediaCrawler 实现。

拍板项：**无**。用户已确认配置驱动、默认无评论与轻量六边形架构。

## 1. 起点

`07eda553812bfea08c24d881fd6430e51ce09992`。用户未提交的 `configs/留学辅导/run.yaml` SHA-256 为 `3d6d883b523fd27e5acc5a95d39e5bb981d87453ef7ffe82c349ba325ae4859f`，本期不修改；未跟踪 `data/` 不清理。

## 2. 现状与复用盘点

- `MediaCrawlerAdapter.fetch_creator_notes` 已实现多账号单会话、详情、档案、评论、图片、进度、超时部分抢救和失败账号识别，可直接复用。
- 当前 creator 命令写死 `get_comment=yes/get_sub_comment=yes`，需添加默认兼容的 `with_comments` 开关。
- MediaCrawler 的 creator 分页以 `CRAWLER_MAX_NOTES_COUNT` 为上限并在 `has_more=false` 时自然结束；账号分析 `max_notes: null` 映射为大上限 `10000`，非无限循环。
- `normalize_creator_ref` 已支持 ID/主页 URL；fixture 已有 creator contents/profiles；旧 exporter 的 CSV 字段可作表征基线。
- 现有 MySQL 使用 `~/.my.cnf`；新 repository 沿用连接方式，但写新 `creators/contents/media_assets` 表，不改旧 `accounts/notes`。

## 3. 目标 / 非目标

### 3.1 目标

1. 建最小 `src/recon/domain/application/platforms/infrastructure/entrypoints` 边界及架构测试。
2. 扩展现有 YAML：`account_analysis_file` + `account_analysis`，账号条目形状复用 watchlist 引用格式。
3. 新增 `./run.sh account`，配置错误和 live 目录不可用时 fail fast。
4. 默认全量、无评论；支持最大帖数、评论和媒体显式配置。
5. 帖子/账号以 `(platform, external_id)` 幂等 upsert；不开店时仍可完成文件输出。
6. 输出 `account_summary.csv`、`account_contents.csv` 和 `account_report.md`，保留每次目录并维护 account 专用 `latest`。

### 3.2 非目标

- 不迁移 search/watchlist/Web/bundle/backfill，不修改现有日常入口语义。
- 不做旧数据导入，不建 `content_keywords`、run ledger、指标快照或 recovery。
- 不实现真实抖音或重写 XHS 原始 parser/crawler。

## 4. 接口与数据设计

```python
class AccountCollector(Protocol):
    def collect_accounts(self, request: AccountCollectionRequest) -> AccountCollectionResult: ...

class AccountRepository(Protocol):
    def save(self, result: AccountCollectionResult) -> None: ...

class AccountOutput(Protocol):
    def write(self, analysis: AccountAnalysis) -> dict[str, str]: ...
```

`AnalyzeAccounts.execute()` 是唯一用例编排：采集 → 按账号聚合 → 幂等保存 → 文件输出。部分失败保留已成功账号；配置/能力/持久化失败整体失败。

```text
creators(platform, external_id, nickname, profile...,
         first_seen_at, last_seen_at, creator_fetched_at, updated_at)
contents(platform, external_id, creator_platform, creator_external_id,
         title/body/url/metrics/media..., published_at,
         first_seen_at, last_seen_at, comments_fetched_at, updated_at)
content_comments(platform, content_external_id, comment_key, external_id, ...)
media_assets(platform, content_external_id, path)
```

creator/content 唯一键为 `(platform, external_id)`；comment 唯一键为
`(platform, content_external_id, comment_key)`；media 唯一键为
`(platform, content_external_id, path_hash)`。upsert 保留 `first_seen_at`，更新
`last_seen_at/updated_at`；重复运行不增重复行。

输出目录：`<export.out_dir>/account/<timestamp>-account/...`，
`<export.out_dir>/account/latest` 指向最新账号分析，不影响现有 search/track 的 `latest`。

## 5. 实现步骤

1. 增配置模型、账号资产解析与回归测试。
2. 建新 domain/ports/account use case 和纯聚合测试。
3. 建 XHS collector wrapper，给旧 adapter 增兼容的评论开关。
4. 建内存/MySQL repository 与 CSV/Markdown output。
5. 建 config/container/CLI，接入 `run.sh account`。
6. 加 fixture E2E、架构测试和全量回归。

## 6. 测试策略

- 配置：内联/文件账号正常注入；双源、缺文件、缺键失败；旧 YAML 仍可解析。
- 采集：fixture 转新领域模型；默认不返评论；限量可预测；MediaCrawler 命令的 yes/no flag 定点测试。
- 用例：聚合结果、互动排行、发布分布、部分失败和 repository/output 各一次。
- 持久化：用 fake connection 验证复合键 upsert/事务；真 MySQL 不进离线 CI。
- E2E：fixture 配置运行新 CLI，产出三个文件，无 comments；`run.sh` 路由存在。

## 7. 验收标准

- `./run.sh account` 仅从 YAML 读账号，未配时明确报错。
- fixture E2E 和全量测试通过；新架构依赖禁线通过。
- 旧 search/track/research 回归通过；用户配置哈希不变。
- 真实单账号 smoke 仅在已登录且用户配置账号后执行，不作离线 CI 前置。

## 8. 风险与禁止事项

- `10000` 是「遍历到 has_more=false」的安全上限，不是承诺绕过平台可见性限制。
- 禁止改用户当前 `run.yaml`、禁止真实采集测试无限扩大、禁止顺带实现 search/watchlist 新架构。
- 旧 adapter 仅允许增默认兼容参数，旧调用行为不变。

## 9. 回滚口径

删除 `src/recon`、`tests/recon`、新 account 配置样例和 `run.sh account` 路由，回退 adapter 的兼容开关；旧入口和旧表未切换。新表若已有数据仅停写，不自动 drop。

## 实施偏差记录

- 方案中的真实单账号 smoke 未执行：当前用户配置尚未提供 `account_analysis` 账号，且默认全量可产生大量实际请求；按方案其本就非离线 Gate 前置。
- 蓝图提到的 account 增量开关在配置模型中保留，但期1遇 `incremental: true` 会明确报「尚未实现」，不静默按全量执行。当前用户要求为全部帖子，不阻断本期主路径。
- 期1平台/组装层仍以 wrapper 复用旧 adapter/config/runtime；domain/application 已完全禁旧依赖。平台与 infrastructure 的旧依赖清零留在相应用例迁移期，避免为目录纯度复制成熟采集器。
