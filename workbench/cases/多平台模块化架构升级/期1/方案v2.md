## 4. 接口与数据设计

### 4.1 领域身份与内容

```python
@dataclass(frozen=True, slots=True)
class PlatformId:
    value: str  # ^[a-z][a-z0-9_-]{1,31}$

@dataclass(frozen=True, slots=True)
class EntityId:
    platform: PlatformId
    external_id: str  # strip 后非空

@dataclass(frozen=True, slots=True)
class Engagement:
    likes: int = 0; collects: int = 0; comments: int = 0; shares: int = 0
```

`Creator/Content/Comment/CreatorProfile` 使用 `EntityId`；跨实体引用必须同 platform；时间为带时区 `datetime|None`；集合为 tuple；互动不得为负。`CreatorRank/WatchTarget/TypicalContent/ResearchRun/RunReceipt` 放 `domain/research.py`。

`Content` 含标题、正文、标签、URL、互动、发布时间、采集时间、来源查询和媒体引用。`Comment` 含自身 ID、内容 ID、可空父 ID、正文、互动、创建/采集时间；不把平台专属字段塞入通用 metadata。

### 4.2 XHS 评论身份契约

翻译层统一规范：

```python
def comment_external_id(raw) -> str:
    if raw.comment_id.strip():
        return raw.comment_id.strip()
    canonical = "\0".join([
        raw.note_id.strip(),
        normalize_parent(raw.parent_comment_id),
        raw.user_id.strip(),
        str(raw.create_time or ""),
        raw.content.strip(),
    ])
    return "synthetic:" + sha256(canonical.encode("utf-8")).hexdigest()

normalize_parent(value): ""/"0"/None -> ""；否则 strip 后原值
```

- 非空 raw ID 原样作为 external ID；缺 ID 使用含 content 维度的稳定 SHA-256，只保存 hash，不把正文写入 ID。
- `parent_comment_id in {None,"","0"}` → `parent_id=None`；其余 → 同 platform 的 `EntityId`。
- fixture provider 按完整 `EntityId(platform, external_id)` 去重；两平台相同 raw ID 不冲突。
- 同 synthetic ID 且 canonical payload 相同视为重复；同 ID 但规范化 payload 不同则不静默覆盖，返回 `CollectionFailure(code="comment_identity_collision")` 并丢弃冲突行。
- 测试覆盖缺 ID 重复读取稳定、一级父为空、非零父引用、两平台同 raw ID 和强制 collision 分支。

### 4.3 采集能力与插件

`CollectionResult` 含 platform/operation、contents/creators/comments/profiles、结构化 failures 和 raw_artifacts。四个 Protocol 分别接受 `SearchRequest/CreatorFeedRequest/DetailRequest/CommentRequest`。

`PlatformServices` 显式持有可选能力；`require(capability)` 缺失抛 `CapabilityUnavailable`，禁止 `hasattr()`。`PlatformPlugin(name, config_model, build)` 经 `PlatformRegistry` 显式注册；重复、未知、name 不一致 fail fast。`test_social` 只在测试 composition root 注册。

### 4.4 副作用唯一所有者

```python
class SearchResearch:
    def execute(self, command: SearchCommand) -> ResearchRun: ...  # 不 save/write

class TrackCreators:
    def execute(self, command: TrackCommand) -> ResearchRun: ...  # 不 save/write

class RunResearch:
    def execute(self, command: ResearchCommand) -> ResearchRun:
        search_run = self.search.execute(command.search)
        track_run = self.track.execute(command.track)
        return merge_runs(search_run, track_run)                  # 不 save/write

class RunCoordinator:
    def execute(self, service: ResearchService[C], command: C) -> RunReceipt:
        run = service.execute(command)
        self.repository.save(run)                                 # 恰一次
        paths = self.output.write(run)                             # 恰一次
        return receipt(run, paths)
```

采集属于计算服务必要 I/O；repository/output 副作用只归 `RunCoordinator`。CLI 的 search/track/research 都选择相应 service 后经同一个 coordinator。测试以 spy 断言三命令分别 save/write 恰一次；组合用例可直接单测 merge 而零持久化/文件。

### 4.5 业务规则

- 聚合按 `EntityId` 去重并保序合并来源词。
- 排名保持 `10*内容数 + 5*词命中 + 0.01*(likes+collects+comments 平均值)`，shares 不入旧公式。
- 画像保持 `10*垂直度 + 1*窗内内容数`。
- 典型内容保持 `likes+2*collects+3*comments` 与可选半衰期。
- 时间窗缺时间剔除并计 missing，window<=0 原样。
- watchlist 保持 self→manual→auto、去重、self 不因 max_total 丢失。
- 表征只比 ID/顺序/分数/计数，float 用 `pytest.approx`，同分顺序由输入顺序保持。

### 4.6 Fixture 组合流程

新增 `tests/recon/fixtures/creator_comments.jsonl`，使用现有 creator 内容 ID（至少 `6a4661cd0000000017029d86`），包含：一级有 ID 评论、一级缺 ID 评论、二级评论三种。它只供新架构 contract，不修改旧 `tests/fixtures/comments.jsonl`。

`TrackCreators` 流程：creator-feed 得 contents/profiles → 以本次 content IDs 请求 detail（fixture 可原样返回已完整内容）→ 以同 IDs 请求 comments → CommentProvider **必须按 request IDs 过滤** → 合成 run。验收 target `601d...` 可得到关联评论；混入其他 note 的 fixture 行必须被过滤。

### 4.7 Repository、配置与输出

`ResearchRepository.save/known_content_ids/contents`；期1 `InMemoryResearchRepository` 按 `EntityId` 覆盖，不实现跨进程增量。`ResearchOutput.write` 返回路径 mapping；`Clock.now` 由 entrypoint 注入固定/系统实现。

`configs/recon/sample.yaml` 使用 schema_version=1、XHS fixture 四路径（comments 指向新关联 fixture）、search/track job、memory storage 和 `data/recon/exports`。通用 Pydantic config `extra=forbid`，平台段交 plugin config_model 二次校验。

CLI：`python -m src.recon.entrypoints.cli search|track|research --config configs/recon/sample.yaml`。

输出：`data/recon/exports/<run_id>/manifest.json` + `xhs/creators.csv|contents.csv|comments.csv|creator_ranks.csv|typical_contents.csv`，并更新 `latest`。manifest 的 counts/files 必须与 CSV 实际行数/路径一致；空集合也写只有表头的 CSV，保证文件契约稳定。

## 6. 测试策略

- 模型：非法平台/空 ID/跨平台引用/负互动/naive datetime 拒绝。
- 评论：有 ID、synthetic 稳定、`0→None`、二级父、跨平台同 ID、collision failure。
- registry/ports：重复/未知/能力缺失/name 不一致。
- policies：同样本投影比较聚合、排名/画像、窗口、watchlist、典型内容；shares 不入排名。
- fixture：四能力过滤正确；target `601d...` 评论非空且全部属于本次 creator 内容。
- use cases：三个 service 无 repo/output；coordinator 的 search/track/research 各 save/write 恰一次；部分失败留 receipt。
- contract：XHS 与 `test_social` 共用 search contract；同裸 ID 聚合为两实体。
- output：五 CSV 即使为空也存在，manifest counts/files 对账，latest 正确，不产生 `data/exports|media|raw`。
- CLI：三命令 0；坏 schema/platform/config 非 0。
- 架构/no-touch：新包零旧 import；核心零平台词；旧 tracked paths 对起点 diff 空；旧路径无未跟踪新增；用户配置 hash 不变。
- 回归：全量 pytest 绿。

盲区：不验证 live/DB/图片/Web/bundle；内存 repo 不证明跨运行增量；表征只覆盖已知行为向量。

## 7. 验收标准 + 检查清单 YAML

同目录 `checklist.yaml` 为唯一事实源：lint/format/全测、三 demo、架构禁线、no-touch、用户配置 hash、工作台视图均绿；manifest 与五个 XHS CSV 必存；关键接口和关联评论 fixture 落地。

- 期1虽模块多，但只含离线风险域；出现 live/MySQL/Web/run.sh 需求立即停并登记。

- 禁止用 mock 调用次数掩盖副作用接口；coordinator 是唯一 save/write 所有者。
- 禁止新包 import 旧函数；tests 表征边界可同时 import 新旧。
- 禁止万能 metadata、可变 domain 集合和原地修改实体。
- 禁止写旧 data 目录、修改旧业务/tests/web/run.sh/用户配置。
- 禁止网络、CDP、MySQL、代理池、验证码破解、自动互动和并发放大。
