# 多平台模块化架构升级 期1 方案评审 #2（差量评审）

**评审对象**：`期1/方案v2.md` + `期1/checklist.yaml`
**评审基准**：`期1/方案评审1.md` 的 4 个阻塞项；本轮只核 v1→v2 修订及其是否引入新矛盾。
**verdict：通过。** 4 个阻塞全部关闭；新增设计与蓝图 v5 的期 1 风险域、模块边界和回滚口径一致，未发现新阻塞。

## 四个阻塞差量对账

### 1. Comment 身份与父引用：已关闭

v2 §4.2 已把身份契约钉到 XHS translator：非空 raw ID 保留；缺 ID 对 `note_id / 规范化 parent / user_id / create_time / content` 的 NUL 分隔 canonical payload 做 SHA-256，并加 `synthetic:` 前缀。content 维度存在，重复读取稳定，正文不直接泄入 ID。

- `None / "" / "0"` 统一规范为无父引用，其余父 ID 才构造同平台 `EntityId`，不会再产生伪父 `EntityId("0")`。
- 去重键是完整 `EntityId(platform, external_id)`，两平台相同 raw ID 彼此隔离。
- synthetic ID 相同且 canonical payload 相同才按重复处理；ID 相同而 canonical payload 不同返回 `CollectionFailure(code="comment_identity_collision")` 并丢弃冲突行。实现时只需在翻译/fixture 批次内保留 `ID → canonical payload` 对照；强制 collision 可通过固定 digest seam/patch 触发，不依赖真实 SHA-256 碰撞。
- §6 已逐项覆盖 synthetic 稳定、`0→None`、二级父、跨平台同 ID 和 collision failure。

**Q1：规则是否足够确定，能区分“重复”与“碰撞”？** 是。身份输入、规范化、hash 算法、前缀、父映射、重复条件和冲突处置均已给定；没有把平台数据规则上移到 domain。阻塞 1 关闭。

### 2. Search / Track / Research 单次副作用：已关闭

v2 §4.4 将 `SearchResearch.execute`、`TrackCreators.execute` 和 `RunResearch.execute` 统一定义为返回 `ResearchRun` 的计算服务，不调用 repository/output；`RunResearch` 只组合两个内存 run。`RunCoordinator` 是唯一副作用所有者：先执行所选 service，再 `repository.save(run)` 一次、`output.write(run)` 一次，最后返回 receipt。三条 CLI 都经过同一个 coordinator。

**Q2：公开组合路径会不会再次保存或输出？** 不会。子服务和组合服务的签名都只返回 run，save/write 只出现在 coordinator；§6 又以三命令 spy 的“各恰一次”和组合服务“零持久化/文件”双向锁定。阻塞 2 关闭。

蓝图 v5 §3.3 对 `SearchResearch/TrackCreators/RunResearch` “只依赖 repository/output/clock”的表述是应用层依赖边界的粗粒度约束；v2 将 repository/output 收敛到同属 application orchestration 的 coordinator，是对单次副作用语义的职责细化，没有把依赖倒向外层，也未改变 fixture、内存 repository 或文件 output 的期 1 范围。

### 3. Track 关联评论 fixture 与过滤：已关闭

v2 §4.6 不再复用与 sample target 无交集的旧 `comments.jsonl`，而是新增 `tests/recon/fixtures/creator_comments.jsonl`，至少使用现有 creator content `6a4661cd0000000017029d86`，并包含一级有 ID、一级缺 ID、二级评论三类记录；旧 fixture 保持不动。

流程被明确为 creator-feed → 本次 content IDs 的 detail → 同一组 IDs 的 comments；CommentProvider 必须按 request IDs 过滤。验收同时要求 target `601d...` 得到非空关联评论，并用混入的其他 note 行证明过滤真实生效。

**Q3：非空评论验收是否仍可能靠“不过滤”假通过？** 不会。测试同时断言评论非空、全部属于本次 creator contents，且无关行被过滤；关联性和过滤性相互制约。阻塞 3 关闭。

### 4. no-touch、用户配置与完整文件契约：已关闭

checklist 已加入：

- 对固定起点 `07eda553812bfea08c24d881fd6430e51ce09992` 的旧 tracked 路径差量检查，覆盖 `src/models.py`、`src/core`、`src/adapters`、`src/pipelines`、`web`、`run.sh`、旧 `tests`，排除允许新增的 `tests/recon`；
- 同一旧路径集合的未跟踪文件检查，并显式过滤 `tests/recon/`；
- `configs/留学辅导/run.yaml` 的起草时 SHA-256 固定检查；
- `latest/xhs` 下 `creators.csv / contents.csv / comments.csv / creator_ranks.csv / typical_contents.csv` 五个必存产物；
- `creator_comments.jsonl`、`RunCoordinator` 和 `comment_identity_collision` 的机械存在性检查。

在当前脏工作树亲跑三条保护命令均为退出码 0。用户配置当前 hash 为 `3d6d883b523fd27e5acc5a95d39e5bb981d87453ef7ffe82c349ba325ae4859f`，与 checklist 相同；当前用户自己的 `configs/`、`data/`、`workbench/` 改动不会被旧路径命令误伤。Git 的排除 pathspec 与未跟踪结果过滤均将 `tests/recon` 留给本期新增。

**Q4：当前 dirty tree 上 no-touch 是否可执行且能区分用户既有改动与本期旧代码误改？** 是。它不要求全工作树干净，只比较指定旧路径与固定 commit，并独立用 hash 守住已修改的用户配置。阻塞 4 关闭。
