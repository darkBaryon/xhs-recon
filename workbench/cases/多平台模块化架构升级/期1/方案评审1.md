# 多平台模块化架构升级 期1 方案评审 #1

**评审对象**：`期1/方案v1.md` + `期1/checklist.yaml`

## 结论

**verdict：需修改。** 方案遵守蓝图 v5 的期 1 边界，仍是单一“新核心离线闭环”风险域，9 步顺序也可分 commit 评审。但 Comment 身份对真实缺 ID 数据无降级契约、`RunResearch` 与子用例的写入语义自相矛盾、sample track 的评论验收不可达，且 checklist 不能机械证明“旧系统零改”或新分区文件真落盘。共 4 个阻塞项。

## 逐章核查

### 给用户的摘要

摘要只承诺 `src/recon` 旁路新增、fixture 三命令闭环和旧系统不变，未夹带 live/MySQL/Web/run.sh。

- **Q0**：期 1 是否仍为单一离线风险域？§3.2 明确排除真实采集、DB、Web、run.sh 与删旧，§5 全部步骤只写新目录，**质疑已关闭**。

### §1 起点

`07eda553812bfea08c24d881fd6430e51ce09992` 与当前 HEAD 一致，`src/**/*.py` 亲核合计 4053 行；用户配置改动和未跟踪 `data/` 也确实存在。

- **Q1**：实施后能否机械证明旧业务代码/旧测试/用户配置没有变？当前 checklist 只跑回归，没有起点差量白名单或用户文件基线摘要，见阻塞 4。

### §2 现状与约束盘点

四类 fixture、parser、六类纯规则和 pipeline E2E 的位置均亲核成立；69 个相关旧测试当前全绿。

- **Q2**：“四类 fixture 已覆盖 track 闭环”是否表示评论与样例 target 的内容关联？亲核发现 comments 只属于 `666c...`，而 sample target `601d...` 的 creator contents 是 `6a466...`，两者无交集，见阻塞 3。

### §3 目标 / 非目标

本期目标与蓝图 v5 第 1 期对齐；模块数虽多，但全部可用固定 fixture/时钟/内存仓库验收，没有外部状态协调。

- **Q3**：9 步是否大到无法差量评审？每步均有独立测试面，且 live/persistence/consumer/cutover 均在后期；**规模偏大但仍可评审，不阻塞**。

### §4 接口与数据设计

`PlatformId/EntityId`、带时区时间、不可变集合、跨实体同平台校验、细粒度能力和外层 Pydantic 边界均合理；没有通用 metadata 逃逸口。

- **Q4a**：XHS 评论没有 `comment_id` 时能否构造非空 `EntityId`？旧 parser 与现有测试明确允许空 `comment_id`，旧 MySQL 以 `note_id + 正文哈希` 降级去重；新设计只规定 external ID 非空，没有稳定合成 ID 契约，见阻塞 1。
- **Q4b**：一级评论的 raw `parent_comment_id="0"` 是否会构造一个伪父 `EntityId("0")`？方案未明确 `""/"0"→None`，与 fixture 实际值存在歧义，与阻塞 1 同源。
- **Q4c**：`RunResearch` 调用公开 `SearchResearch.execute/TrackCreators.execute` 时如何避免子用例已经 `save/write`？§4.4 给两个子用例定义了写副作用，又要求组合用例调公开服务且整体只写一次，当前不可同时满足，见阻塞 2。
- **Q4d**：新旧排名投影是否能捕获 `shares` 误入旧平均互动口径？旧 `_interaction` 只含 likes+collects+comments，方案要求直接比较分数，所以测试会可证伪地捕获；**不阻塞，实施期须按旧口径**。

### §5 实现步骤

步骤从架构禁线、模型、端口、规则、fixture 递进到用例/入口，顺序合理。

- **Q5**：第 7 步“覆盖只写一次”是否有对应的可实现 seam？目前没有无副作用的 search/track 运算接口或由组合根统一 commit 的设计，仅有 mock 调用次数无法修复接口矛盾。

### §6 测试策略

模型、registry、policies、translator、contract、use-case、output、CLI、架构和旧回归的测试面完整，旧新算法比较也限定稳定投影而非类型/序列化相等。

- **Q6a**：现有 sample 能否让 `track` 断言非空评论？不能；按 CommentRequest 的 content IDs 正确过滤后为空，若不过滤则会把其他创作者的评论污染本次 track，见阻塞 3。
- **Q6b**：旧新表征是否明确到可以失败？方案已列出 ID/排序/分数/计数与现有专项向量，相关 69 个旧测试亲跑全绿；**基本足够**。对 float 容差和同分 tie-break 可在实施时用现有输出钉死，不改方案目标。

### §7 验收标准 + checklist

YAML 可被 `yaml.safe_load`正常解析；`preflight.py` 使用 shell 执行必过命令、字面 `grep` 检查必含/禁留，因此 `hasattr(`/`print(`/`TODO(临时)` 本身不会被当成 rg 正则而报错。三个 demo 的命令形式与方案一致。

- **Q7a**：全量 pytest 通过是否能证明旧实现一行未改？不能；等价修改仍可全绿，checklist 缺少以起点为基准的旧路径 no-touch 检查，见阻塞 4。
- **Q7b**：CLI 真实输出缺少某个平台 CSV 时，必存产物会报红吗？不会；它只检查 `manifest.json`，没有检查方案承诺的 `xhs/creators.csv|contents.csv|comments.csv|creator_ranks.csv|typical_contents.csv`，见阻塞 4。
- **Q7c**：`! rg ...` 在目录不存在时会假绿吗？不会；虽然 `!` 把 rg 退出码取反，`expect_empty` 仍会看到 stderr 并判红，**该命令可执行**。

### §8 风险与禁止事项

禁止 live/DB/旧目录输出/旧 import/临时残留的范围清楚；表征测试可在 tests 边界 import 旧包，与新运行时禁线不冲突。

- **Q8**：静态 grep “不写旧数据目录”是否足以阻止 `Path("data") / "exports"` 这类拼接？不足；需由 CLI/output 测试的 `tmp_path`/产物根言和差量检查共同守护，应与阻塞 4 一并落 checklist。

### §9 回滚口径

不动旧入口/数据的前提下，删除新包/测试/配置和 `data/recon` 即可回退，与蓝图一致。

- **Q9**：如果实施曾修改旧 src 但测试仍绿，“删新目录即回滚”是否成立？不成立；因此 no-touch 不是建议，而是回滚口径成立的必要机械条件。

## 阻塞项

1. **Comment 身份与父引用缺少真实降级契约**：`EntityId.external_id` 非空，但旧 parser 和测试明确接受缺 `comment_id` 的数据，旧库用 content ID+正文哈希降级。方案应定义平台翻译层的稳定合成 ID（须含 content 维度）、冲突口径，并明确 `parent_comment_id in {"", "0"}` 映射 `None`；测试覆盖两平台相同 raw comment ID 不冲突、缺 ID 重复读取稳定去重。
2. **组合用例的副作用语义自相矛盾**：`SearchResearch.execute` 与 `TrackCreators.execute` 均定义为自行 save/write，`RunResearch` 又要求调用这两个公开服务返回内存 run 且整体只写一次。应拆出无副作用的 search/track 计算服务，再由三个入口用例各自统一 commit；或定义另一个不会导致重复输出的公开组合契约。
3. **sample track 的评论验收不可达**：sample target `601d...` 的 creator content IDs 与 `comments.jsonl` 的 `666c...` 无交集。CommentProvider 若正确按 request content IDs 过滤就不会产出评论；不过滤则污染 track 结果。方案应明确 creator-feed→detail→comments 的 fixture 组合流程，并新增/转换一份与 creator content 关联的 contract fixture，或修正“track 产出评论”验收口径。
4. **checklist 不足以守住零改与文件契约**：缺少以起点为基准的旧 `src/models.py|core|adapters|pipelines|web|run.sh|旧 tests` no-touch 检查，也没有对已存用户配置差量/摘要的保持检查；必存产物只检 manifest，没有检查五个 `latest/xhs/*.csv`。应增加不误伤既有用户改动的 no-touch 基线方式，并把 `creators.csv/contents.csv/comments.csv/creator_ranks.csv/typical_contents.csv` 列为必存产物；output/CLI 测试还须断言不产生旧目录写入。

## 建议栏（不立案）

1. 实施时将排名“平均互动”明确写成 likes+collects+comments，shares 仅保留在 `Engagement`，避免非意图改算法。
2. `CreatorProfile` 本期保持通用字段是合理的；但须在第 3～5 期方案中说明 `red_id/follows/interaction/tags/ip_location` 等历史数据进平台 read model/raw store，不应因本期 domain 精简而丢失。
